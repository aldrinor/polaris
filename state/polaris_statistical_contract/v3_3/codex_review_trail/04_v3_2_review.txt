### 3.4 Scenario-family relations + pairwise DEFF (REVISED v3.2 per Codex round-2 §2)

**The conceptual fix**: v3.1's "pairwise-only family assignment" + cluster-DEFF formula `1 + ((Σm_i²/Σm_i) − 1)ρ` was internally inconsistent. Pairwise relations are not transitive: claim A related to B, B related to C does not imply A related to C. The cluster DEFF assumes non-overlapping partitions, which pairwise relations don't form.

**v3.2 fix**: keep pairwise definition; switch DEFF to **pairwise-correlation form**:

```
DEFF = 1 + 2 × P × ρ / N

derivation: Var(Σ X_i / N) for binary X_i with mean shared variance σ² and
            ρ correlation per related pair:
  Var(Σ X_i / N) = (1/N²) × [N×σ² + 2 × P × ρ × σ²]
                 = (σ²/N) × (1 + 2Pρ/N)
  
  so DEFF = 1 + 2Pρ/N
```

This avoids the transitive-blow-up problem and matches the pairwise data structure.

**Pairwise relation rules (LOCKED v3.2)**:

Two claims are scenario-family-related iff ANY of:

1. **Same source report**: same POLARIS run / report ID
2. **Substantially overlapping evidence packet**: Jaccard on cited evidence-document set ≥ 0.5
3. **Same SME construction template within 24h**: same template ID, same constructing SME, within 24h
4. **Same prompt family**: generator OR verifier prompt-family ID matches; prompt-families enumerated in `state/prompt_families.json` (hash-pinned)
5. **Microtopic + stratum + additional correlation**: shared microtopic tag AND shared severity stratum AND at least ONE of:
   - Shared SME construction template (any time window)
   - Shared source-packet class
   - Evidence overlap Jaccard ∈ [0.2, 0.5)

**Audit prerequisite (LOCKED v3.2)**: After Phase 0a pilot, compute and audit:
- `P_pilot`, `N_pilot`, `P_pilot/N_pilot`
- 95th-percentile claim degree (= max # of relations per single claim)
- If 95th-percentile degree > 25, criterion-5 implementation must tighten OR microtopic ontology refined OR §P4 amendment filed

**Pairwise computation is auditable**: lookup table maps (claim_id_i, claim_id_j) → related? (boolean). Hash-pinned alongside the gold set. Replaces the implicit "cluster membership" assumption.

### 4.4 ICC justification protocol (REVISED v3.2 per Codex round-2 §3)

**Pre-designated surrogates** (LOCKED pre-Phase-0b, in `state/icc_surrogates.json`):

| Tier | Surrogate outcome | Estimator |
|---|---|---|
| **Primary** | Source-disagreement frequency per report | One-way random-effects ANOVA on customer-grouped logit-transformed proportions; ICC = σ²_between / (σ²_between + σ²_within) |
| **Secondary 1** | Evidence-citation completeness rate per report | Same estimator |
| **Secondary 2** | Retrieval-tier prevalence (T1+T2 proportion) | Same estimator |
| **Verifier-native (NEW v3.2 per Codex round-2)** | Verifier escalation rate per report (count of claims with verifier-flagged uncertainty per report) | Same estimator |

**Why a verifier-native surrogate**: per Codex round-2, customer clustering may enter through verifier behavior, not just retrieval. Adding a verifier-side surrogate prevents the retrieval-only triad from missing customer correlation through the verifier pipeline.

**Proportion-surrogate methodology (NEW v3.2)**: One-way random-effects ANOVA on raw proportions is heteroskedastic. v3.2 locks: **logit-transform proportions before ANOVA**, then ICC = σ²_between / (σ²_between + σ²_within) on logit scale, transformed back for reporting via delta method.

**ICC CI method (NEW v3.2)**: One-sided upper 95% CI via parametric bootstrap (B=10000) on logit-ICC, transformed back. Hash-pinned random seed.

**Escalation rule (FIXED v3.2 per Codex round-2 §3)**:

v3.1 said `ceil(max_observed × 1.5)` — but `ceil(0.11 × 1.5) = ceil(0.165) = 1.0` (because `ceil` on a real number rounds up to next integer). v3.2 fixes:

```
If any of the four surrogate ICC point estimates > 0.10:
    OR any one-sided upper 95% CI > 0.15:
        Escalation triggered.
        
        Escalated ICC ceiling = max(
            round_up_to_nearest(max_observed_point_estimate, 0.05),
            round_up_to_nearest(max_observed_upper_95_CI, 0.05),
            0.15
        )
        
        where round_up_to_nearest(x, 0.05) = ceil(x / 0.05) × 0.05
        
        Examples:
          max_observed = 0.11 → escalated = max(0.15, 0.15, 0.15) = 0.15
          max_observed = 0.13 → escalated = max(0.15, 0.15, 0.15) = 0.15
          max_observed = 0.18 → escalated = max(0.20, upper_95_ceil, 0.15) = ≥0.20
          max_observed = 0.07 but upper_95 = 0.18 → escalated = max(0.10, 0.20, 0.15) = 0.20

A §P4 Category-3 amendment is also filed (pre-outcome design change).
```


The binding precision gate uses production-canary prevalence ONLY. Other prevalences are reporting-only with explicit non-safety labels.

**1. §3.2 Final DEFF Exposure**

Yes. Computing final-gold-set DEFF before outcome unblinding is structural exposure, and §P4.2 now explicitly lists “Final gold-set DEFF,” which is sufficient.

One small tightening: say “per-stratum `P_final`, `N_final`, degree statistics, relation-table hash, and `DEFF_binding` are structural exposure.” Because Gate A DEFF is stratum-specific, the artifact necessarily exposes stratum denominators. That is fine if severity labels are already locked and no verifier miss outcomes are visible.

**2. §3.4 Pairwise DEFF**

The formula is mathematically correct under the locked model: equal marginal variance, correlation `ρ` for each related unordered pair, zero covariance for unrelated pairs. It follows directly from `Var(sum X_i)`.

Edge cases are acceptable. If `P=0`, DEFF=1. If every pair is related, `P=N(N-1)/2`, giving `DEFF=1+(N-1)ρ`, matching a single exchangeable cluster. Saturation does not break the formula; it just collapses effective sample size hard.

Main risks: missed positive correlations among “unrelated” pairs, duplicate-counting pairs satisfying multiple criteria, or true pair correlations above the ICC ceiling. Require pair de-duplication and keep the ρ=0.20 sensitivity.

**3. §4.4 Escalation Rounding**

The minimum 0.15 floor is fine, but only inside the escalation branch. Your current pseudocode already reads that way.

There is an intentional discontinuity: `point=0.100` does not trigger, `point=0.101` triggers to at least 0.15. That is acceptable for a pre-registered safety ceiling. For precision, define strict comparison and numeric tolerance, e.g. “trigger if estimate exceeds threshold by more than `1e-12` after deterministic rounding.”

Also cap escalated ICC at 1.0.

**4. §4.4 Logit-ANOVA**

Logit-ANOVA is defensible as a transparent pre-outcome surrogate audit, but beta-binomial or logistic-normal random-intercept GLMM is more canonical for count-over-denominator proportions, especially with unequal denominators, 0/1 proportions, and overdispersion. Beta-binomial models are explicitly used for intra-correlated binomial data and overdispersion. ([pmc.ncbi.nlm.nih.gov](https://pmc.ncbi.nlm.nih.gov/articles/PMC5736152/?utm_source=openai))

I would revise §4.4 to make beta-binomial/logistic-normal GLMM primary where denominators are available, with empirical-logit weighted ANOVA as sensitivity or convergence fallback. If you keep logit-ANOVA primary, lock continuity correction, denominator weighting, zero-denominator handling, and boundary ICC handling.

**5. §5.4 BH Paths**

Mostly closed. BY default is right: Benjamini-Yekutieli covers arbitrary dependence, while BH is justified under independence or PRDS-type positive dependence. ([hero.epa.gov](https://hero.epa.gov/reference/6791919/))

The only wording risk is Path 3. A locked resampling FDR procedure does not “allow BH”; it is its own alternative binding procedure. Change the heading from “BH allowed iff” to “A less-conservative-than-BY binding procedure is allowed iff.” Then Path 1 can justify BH by simulation, Path 2 by proof, and Path 3 by using the locked resampling procedure directly.

No loophole if all artifacts are hash-pinned before monitoring.

**6. §6.4 PPV CI**

Parametric bootstrap is the right frequentist tool here. PPV depends on sensitivity, specificity, and prevalence, and predictive values are prevalence-dependent. ([ncbi.nlm.nih.gov](https://www.ncbi.nlm.nih.gov/sites/books/NBK557491/))

Bayesian intervals are not necessary unless you want prior-sensitive reporting. They would add governance burden: priors must be justified, locked, and sensitivity-checked. For the binding gate, keep bootstrap.

Clarify dependence: if sens/spec and canary prevalence come from separate samples, bootstrap them independently except for shared customer structure. If they share units, preserve the joint cluster resampling. Use logit-scale draws or constrained beta/binomial draws to avoid impossible values.

**7. §7.2 Missingness/Prevalence Audit**

The audit plus hash-pin is statistically sufficient if the coefficient choice is deterministic from pre-outcome missingness, prevalence, and rater-completeness data.

For governance, add a lightweight blinded sign-off: named audit custodian, timestamp, artifact hashes, and statement that miss outcomes/agreement coefficients were not viewed. A full “audit committee” is not statistically required. Codex review is useful but should not be described as independent third-party audit.

**8. §P3.4 LLM-as-Judge Leakage**

The list is strong. Add one explicit item for mechanistic and hidden-state leakage:

“Judge hidden states, attention maps, logits/logprobs, embeddings, activation patches, mechanistic-interpretability outputs, or gradient-like traces from gold-set or Phase-0a queries used to revise judge/verifier prompts, judge selection, rubrics, exemplars, or model choice before gate completion.”

Your current #39 gestures at this, but making it explicit closes the path. The concern is consistent with LLM-as-judge preference leakage and contamination-limited benchmark literature. ([arxiv.org](https://arxiv.org/abs/2502.01534)) ([arxiv.org](https://arxiv.org/abs/2406.19314))

**9. §P4.2 Structural vs Outcome**

The split is correct. Borderline artifacts should be gate-specific.

Per-domain claim counts, per-stratum denominators, per-cell eligibility counts, and customer/report concentration are structural if computed before verifier outcomes. Per-domain miss counts, miss rates, severity among misses, or customer identities of affected reports are outcome exposure.

Add an explicit row: “Domain/stratum/cell denominator distributions and eligibility distributions: structural exposure.” Note that SME agreement coefficients remain outcome exposure for Gate E even if severity labels themselves are needed structurally for Gate A.

**10. §10.0 Anti-Overclaim**

Yes, add the autonomy-implying phrases:

“AI-verified,” “machine-checked,” “auto-validated,” “self-auditing,” “certified,” “guaranteed safe,” “hallucination-free,” “formally verified,” “validated for production,” and “human-equivalent review.”

Some can be allowed only with precise qualifying language. For example, “machine-checked” should be forbidden unless it refers to an actual formal checker or deterministic test artifact, not an LLM judgment.
