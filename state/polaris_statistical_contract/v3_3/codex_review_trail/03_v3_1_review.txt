### 3.2 Binding pass/fail rule — formula-driven, v3.1 KEY CHANGE

v3.0 locked raw n at 540 (S1) and 138 (S2). **Codex round-1 caught the arithmetic break**: at n=540 zero-misses for UCB ≤0.5%, n_eff must be ≥539, so DEFF ≤1.003 — any cluster correlation fails the gate. v3.1 fixes this by separating the **formula** (locked now) from the **raw n** (locked Phase-0b after Phase-0a observes cluster sizes).

**Locked formula** (binding):

```
For stratum S ∈ {S1, S2}, at zero observed misses:
  n_eff_required(S) = z²/UCB_target(S) − z²    (Wilson zero-event minimum)
  
  Required raw n(S) = ceil( n_eff_required(S) × DEFF_obs )
  
  where DEFF_obs is computed from Phase-0a observed scenario-family sizes:
    DEFF_obs = 1 + ((Σ m_i² / Σ m_i) − 1) × ICC_ceiling
    m_i = number of claims in scenario-family i
    ICC_ceiling = 0.10 (per Codex round-1: "default conservative; can only
                       escalate via §4.4 surrogate-ICC analysis")
```

**Locked UCB targets**:

| Stratum | UCB target | z | n_eff_required |
|---|---|---|---|
| S1 | ≤ 0.5% (one-sided 95%) | 1.6449 | 539 |
| S2 | ≤ 2.0% (one-sided 95%) | 1.6449 | 132 |
| S0 | n/a — policy sentinel | n/a | n/a |

**Planning estimate** (NOT locked — for budget visibility only):

At an anticipated scenario-family mean cluster size `m̄ ≈ 10` and ICC=0.10:
- `DEFF ≈ 1 + 9×0.10 = 1.9`
- Anticipated raw n(S1) ≈ ceil(539 × 1.9) = **1024**
- Anticipated raw n(S2) ≈ ceil(132 × 1.9) = **251**
- Fab-eligible total ≈ 1024 + 251 = 1275 + 50 (S0 stress) = **~1325 fab-eligible**
- Plus SUPPORTED filler → total gold set **~3000-4000 claims**

**Phase 0a output** (hash-pinned before Phase 0b begins):

After Phase 0a constructs the first 500-claim pilot, compute:
- `m̄_obs` = observed scenario-family mean cluster size
- `CV_cluster_obs` = coefficient of variation of cluster sizes
- `DEFF_obs` = 1 + ((Σ m_i² / Σ m_i) − 1) × 0.10

If `m̄_obs > 15` OR `CV_cluster_obs > 0.5`, escalation triggers a §P4 amendment (per Cochrane Ch 23: large or highly variable clusters materially inflate variance).

**S0 framing (audit-locked, unchanged from v3.0)**: S0 is a release-policy sentinel, NOT a statistical population bound. Contract sentence: "Zero S0 fabrications were missed by the verifier on a 50-item operator-adversarial stress-test set. No statistical UCB on production S0 risk is asserted from this evidence."

### 3.3 Primary vs sensitivity reporting

Per Codex round-1: "claim-level Wilson result should be described as unclustered descriptive; the confirmatory Gate A result is the ICC-adjusted one."

**v3.1 locked**:

- **PRIMARY (confirmatory)**: Cluster-adjusted Wilson UCB at `n_eff = n_raw / DEFF_obs`, ICC=0.10. This is the binding gate.
- **DESCRIPTIVE (not binding)**: Naive claim-level Wilson UCB at raw n. Reported alongside; labeled "unclustered descriptive — not the confirmatory gate."
- **SENSITIVITY**: ICC=0.20 stress, also reported, also not binding.

### 3.4 Scenario-family definition (LOCKED v3.1, revised per Codex round-1)

A scenario-family groups claims that share material correlation. Family assignment is **pairwise-only** (NOT graph-connected-component). Two claims belong to the same family iff ANY of:

1. **Same source report**: one POLARIS run → one report → all extracted claims share family.
2. **Substantially overlapping evidence packet**: Jaccard on cited evidence-document set ≥ 0.5.
3. **Same SME construction template within 24h**: same template ID, same constructing SME, generated within 24h.
4. **Same prompt family** (NEW per Codex round-1 §2): generator OR verifier prompt-family ID matches; prompt-families are enumerated in `state/prompt_families.json` (hash-pinned pre-Phase-0b).
5. **Microtopic + stratum + additional correlation source** (RESTRICTED per Codex round-1 §2): shared microtopic tag AND shared severity stratum AND at least one of: shared SME construction template (any time window), shared source-packet class, evidence overlap Jaccard ∈ [0.2, 0.5). Microtopic alone is NOT enough.

**Audit prerequisite (LOCKED v3.1)**: After Phase 0a pilot, audit maximum and 95th-percentile family sizes BEFORE Phase 0b begins. If max family > 50 claims OR 95th-percentile > 25, the criterion-5 implementation must be tightened (or microtopic ontology refined) before proceeding.

**Pairwise-only ruling rationale**: graph-connected-component aggregation can merge otherwise-independent clusters via transitive chains, producing pathologically large families (Codex round-1: "one broad microtopic can merge otherwise separate evidence/report/template clusters into a giant connected component"). Pairwise-only keeps the cluster mechanic interpretable.

### 3.5 What this gate does NOT claim

- Does NOT bound production fabrication rate at the unweighted aggregate level.
- Does NOT claim independence of claims within stratum (per §3.3 the binding analysis is cluster-adjusted).
- Does NOT certify performance outside the locked gold-set construction scope.

### 4.4 ICC justification protocol — formal v3.1 (per Codex round-1 §3)

v3.0 said "surrogate-outcome ICC estimation." v3.1 makes this computational:

**Pre-designated surrogates** (locked pre-Phase-0b in `state/icc_surrogates.json`):

| Tier | Surrogate outcome | Estimator |
|---|---|---|
| Primary | Source-disagreement frequency per report (count of source-citation conflicts) | One-way random-effects ANOVA on customer-grouped proportions; ICC = σ²_between / (σ²_between + σ²_within) |
| Secondary 1 | Evidence-citation completeness rate (count of claims with ≥1 cited evidence per report) | Same estimator |
| Secondary 2 | Retrieval-tier prevalence (proportion of T1+T2 sources retrieved) | Same estimator |

The primary surrogate is designated BEFORE seeing the surrogate values, preventing implicit privileging of the convenient one (Codex round-1).

**Computational escalation rule** (locked pre-Phase-0b):

```
Compute surrogate ICCs (primary + 2 secondaries) on canary BEFORE outcome unblinding.

If any of the three point estimates > 0.10:
    Escalation triggered → ICC_ceiling rises to ceil(max_observed × 1.5)
    AND §P4 amendment is filed.

If any one-sided upper 95% confidence bound exceeds 0.10 by ≥50%
(i.e., upper > 0.15) for the primary surrogate:
    Escalation triggered same as above.

Otherwise: ICC_ceiling remains 0.10 (no escalation).
```

**DEFF audit** (locked pre-Phase-0b):

Before outcome unblinding, compute DEFF from actual canary customer cluster sizes (m_i = reports per customer i). Show:
- Concentration distribution (top-3 customer share, Gini coefficient on m_i)
- Variable-cluster DEFF at ICC=0.10
- Comparison to simple `1 + (m̄ − 1)ICC` (showing how much variable-cluster correction matters)

If concentration violates §4.2 structural prerequisites, halt unblinding and re-sample.

**External literature anchor** (rationale, not evidence): clinical/research RAG studies report ICCs typically in [0.005, 0.10]. The ICC=0.10 ceiling is conservative for the median; can rise per the escalation rule.

## §5. Gate C — Continuous drift monitoring

### 5.1 Two-layer structure (unchanged from v3.0)

Layer 1 = formal labeled-outcome drift (audit gate). Layer 2 = operational leading-indicator drift (process-control, NOT a multiple-testing contract).

### 5.2 Layer 1 — Formal drift contract

| Question | Locked answer (v3.1 updated estimand per Codex v2-dialogue feedback) |
|---|---|
| **Estimand** | Change Δμ in named monitoring metric vs locked baseline, where Δμ exceeds pre-registered minimum material effect size (MMES). For miss rate: MMES = max(2× baseline UCB, +1.0 percentage point). The null is "Δμ ≤ MMES." |
| **Sampling frame** | Per-cell rolling monthly window of canary/audit-sampled claims with SME-confirmed labels |
| **Unit of independence** | Window-cell pair |
| **Error-control** | **Benjamini-Yekutieli by default** (per Codex round-1 §4); BH allowed only by pre-registered baseline-simulation justification (see §5.4). Cadence: monthly. NOT lifetime FWER. NOT LORD/SAFFRON. |
| **Binding rule** | See §5.3 |
| **Discretion** | MMES, BY-vs-BH decision (per §5.4), persistence rule, cell-pooling — all hash-pinned |

### 5.3 Layer 1 binding pass/fail rule

```
Monthly cycle:
  For each cell c with N_window ≥ 30 SME-labeled outcomes:
    Compute test statistic for "Δμ ≤ MMES vs Δμ > MMES"
  
  Multiplicity correction across eligible cells:
    PRIMARY: Benjamini-Yekutieli (BY) at q=0.05
    OPTIONAL ALTERNATIVE: BH if §5.4 justification holds
  
  Cell-level ALERT iff:
    BY-adjusted (or pre-justified BH-adjusted) p < 0.05 AND
    cell breaches in 2 of 3 consecutive monthly windows (persistence rule)
  
  Sparse cells (N_window < 30): no formal test in current cycle;
    roll up per pre-registered hierarchy:
      Level 1: same domain × adjacent complexity
      Level 2: same domain × any complexity
      Level 3: global (with explicit audit annotation)

Interpretation: each monthly report controls FDR at q=0.05 within
its month's eligible cells under the chosen procedure. NOT lifetime.
```

### 5.4 BY vs BH choice (REVISED v3.1 per Codex round-1 §4)

v3.0 used Pearson correlation of cell-level p-values — Codex flagged this as a weak proxy. v3.1 inverts the default:

```
Default: Benjamini-Yekutieli (BY)
  - Conservative under arbitrary dependence
  - Defensible without dependence proof

To use BH instead, ALL of the following must hold (pre-registered, hash-pinned):
  1. Baseline-period null simulation preserving report/customer/cell structure
  2. Empirical estimation of cell-level test-statistic correlation matrix
     under the locked testing procedure
  3. Demonstration that dependence is negligible OR PRDS-like
     (positive regression dependence on subsets)
  4. Simulation outputs hash-pinned BEFORE first monitoring cycle

If any of the four fails, BY is binding.
```

**Why BY default**: BH is proved under independent test statistics (Benjamini-Hochberg 1995); under arbitrary dependence, BY is the correct generalization (Benjamini-Yekutieli 2001). For an audit-grade contract on a 72-cell × 13-window × 4-quarter structure where dependence is plausible (shared retrieval, shared verifier, shared evaluator), BY is the safer default.

### 5.5 Layer 2 — Operational leading-indicator monitors (unchanged from v3.0)

EWMA / CUSUM on: retrieval depth, evidence source mix, unsupported-claim score distribution, claim complexity, domain mix, latency / fallback rates.

- 3σ control limits, CUSUM k=0.5/h=4
- Alerts are operational ("investigate within 24h") not confirmatory
- NOT part of the audit gate

### 5.6 What this gate does NOT claim

- Does NOT claim lifetime FWER bound.
- Does NOT claim "standard practice at Google/Anthropic/OpenAI" (no public material).
- Claims: monthly FDR-within-cadence at q=0.05 under BY (or pre-justified BH).

### 6.1 Six-question contract

| Question | Locked answer |
|---|---|
| **Estimand (primary, safety)** | `P(classifier predicts ESCALATE \| claim is truly safety-critical S0/S1)` = **recall** |
| **Estimand (secondary, workload)** | `P(claim is S0/S1 \| classifier predicts ESCALATE at locked τ, at production prevalence)` = production-prevalence-conditional precision |
| **Sampling frame** | Per-cell labeled examples (72 cells); precision computed at production-canary prevalence AND at pre-specified prevalence levels (see §6.4) |
| **Unit of independence** | Example, with scenario-family cluster sensitivity per §3.4 |
| **Error-control** | Per-cell Wilson lower 95% CI on recall (primary). Precision reported per-cell at multiple prevalence levels. |
| **Binding rule** | See §6.2 |
| **Discretion** | Threshold τ, calibration, feature set, classifier version frozen pre-validation (hash-pinned); cell pooling pre-registered |

### 6.2 Binding pass/fail rule

```
Per cell c with eligible denominators:
  Recall gate (safety-primary):
    Lower 95% Wilson CI of recall(c) ≥ 95%
    Eligibility: N_true_positive(c) ≥ 30
  
  Precision gate (workload, NOT safety):
    Lower 95% Wilson CI of production-prevalence-conditional precision(c) ≥ 90%
    Eligibility: N_pred_positive_at_τ(c) ≥ 30
  
Sparse cells (failing either eligibility): classifier does NOT gate that cell;
  SME-review fallback applies to all claims in that cell.

Cell pooling hierarchy (pre-registered, invoked only with explicit annotation):
  Level 1: same domain × adjacent complexity
  Level 2: same domain × any complexity
  Level 3: global
```

### 6.3 Threshold-lock protocol (unchanged from v3.0)

Pre-validation freeze (hash-signed): classifier model weights, feature pipeline, calibration method + calibration set hash, threshold τ (single or per-cell), threshold-selection rule rationale. Post-validation operating-point movement requires §P4 amendment.

### 6.4 Prevalence-standardized precision reporting (NEW v3.1 per Codex round-1 §5)

For each cell c, report precision at multiple prevalence levels:

| Reported quantity | Audit label |
|---|---|
| Precision @ gold-set/validation prevalence | "enriched-prevalence — NOT production PPV" |
| Precision @ production-canary observed prevalence | "production-canary PPV estimate" |
| Precision @ fixed prevalence levels {1%, 5%, 10%, 30%} | "prevalence-standardized precision curve" |

The binding precision gate uses **production-canary prevalence only**. Other prevalences are reported for capacity planning + comparison across model versions, with explicit non-safety labels.

### 6.5 Why precision is workload, not safety (audit rationale)

Per Altman & Bland (BMJ 1994): positive predictive value depends on prevalence. A classifier at fixed τ has different PPV at 1% positive prevalence vs 30%. Validating on an enriched-positive gold set produces precision that is not production PPV. Recall is conditional on truly-positive — independent of prevalence — so it is the stable safety-primary metric.

## §7. Gate E — SME label quality (validity prerequisite for A, B, D)

### 7.1 Six-question contract

| Question | Locked answer |
|---|---|
| **Estimand (primary)** | Agreement on binary safety-critical "S0/S1 vs not" |
| **Estimand (secondary)** | Ordinal agreement on S0/S1/S2/S3 |
| **Estimand (tertiary)** | Safety-loss table on under-rating rates |
| **Sampling frame** | Calibration claims independent of gold-set holdout; raters from pre-qualified SME pool |
| **Unit of independence** | Claim (rater-structure preserved in bootstrap CI) |
| **Error-control** | Lower 95% CI on primary agreement coefficient |
| **Binding rule** | See §7.2 |
| **Discretion** | Coefficient choice per §7.2 rater-design tie-breaker; initial-blinded-labels protocol; 90-day re-calibration cadence |

### 7.2 Binding pass/fail rule (REVISED v3.1 per Codex round-1 §6)

**Coefficient choice — rater-design-conditioned tie-breaker** (locked pre-Phase-0a):

```
Primary (binary safety-critical, S0/S1 vs not-S0/S1):
  IF rater design is complete (every calibration item has ≥2 blinded ratings from
                                a fixed-size panel):
      → Gwet's AC1
  IF rater panels are incomplete/varying:
      → Krippendorff's nominal α
  
  Binding gate: lower 95% CI of chosen primary ≥ 0.70
  Sensitivity (always reported alongside): the other of {AC1, Krippendorff nominal α}
                                            (because they can diverge materially)

Secondary (ordinal S0/S1/S2/S3):
  IF rater design is complete:
      → Linearly-weighted κ (NOT quadratic — too forgiving on adjacent confusion)
  IF rater design varies:
      → Krippendorff's ordinal α
  Reported (not gating): lower 95% CI

Tertiary (safety-loss):
  Under-rating rate: P(rater label < SME-consensus label by ≥1 grade) per severity
  Wilson 95% CI per class
  Reported with elevation: S0→S1/S2/S3, S1→S2/S3 mis-rating rates flagged
```

**Asymmetric weights NOT used** (per Codex round-1: "raises the obvious audit question: did you tune the weights to make agreement look acceptable?"). Loss analysis is captured via the safety-loss table instead.

**Bootstrap CI method**: B=10000 iterations preserving rater structure. Resampling unit = claim. Hash-pinned random seed.

### 7.3 Initial-blinded-labels protocol (REINFORCED v3.1 per Codex round-1 §6)

IRR is measured on **first-pass blinded labels**, BEFORE any consensus discussion or adjudication. Consensus labels serve as gold-set truth — they are circular as the IRR object.

### 7.4 Sample-size proof via Monte Carlo (locked pre-Phase-0a)

Pre-registered Monte-Carlo simulation must show: at expected severity prevalence and assumed pairwise agreement of κ̂ ≥ 0.75, sample size n_per_class achieves lower 95% CI bound ≥ 0.70 with power ≥ 0.80. Simulation code hash-pinned.

Tentative anchor: 50 paired observations per severity class × 4 classes × 6 domains = **1200 paired calibration labels** total. Final number lockable only after MC sim runs.

## §P4. Prerequisite — Amendment governance — EXPANDED v3.1

Per Codex round-1 §8.

### P4.1 Amendment categories

**Category 1 — Administrative correction** (no statistical consequence):
- Typo fixes, cross-reference fixes, section renumbering.
- Allowed any time, versioned, no fresh holdout.

**Category 2 — Analysis-preserving implementation correction** (NEW v3.1):
- Bugs in pre-registered analysis code, dependency pin fixes, schema mismatches, deterministic calculation errors.
- Does NOT alter estimands, thresholds, denominators, decision rules.
- Allowed after data is seen, BUT requires:
  - Dual-run report: old-code result, corrected-code result, and rationale.
  - Hash-pin of both versions in `state/data_lineage/amendments/`.
  - Operator + Codex sign-off on the dual-run interpretation.

**Category 3 — Design-changing amendment BEFORE relevant outcomes seen**:
- Threshold change, stratum redefinition, coefficient choice, sample-size change.
- Allowed with pre-amendment versioned hash-pin.
- No fresh holdout required IFF amendment precedes outcome unblinding for the affected gate.
- "Outcome seen" defined gate-specifically (§P4.2).

**Category 4 — Design-changing amendment AFTER outcome data seen**:
- Pre-amendment analysis becomes EXPLORATORY (not citable as confirmatory).
- Confirmatory gate requires ONE of:
  - Fresh sealed holdout (not yet labeled).
  - Prospectively-defined alpha-spending plan covering both designs.
  - Bridging validation with conservative methodology + explicit audit label.
- Silent post-hoc relaxation is FORBIDDEN.

**Category 5 — Post-outcome tightening (NEW v3.1)**:
- Strictly raising thresholds or narrowing claims after outcomes seen.
- Less audit-risky than relaxation but still post-hoc.
- Versioned and labeled; does NOT require fresh holdout UNLESS it changes the confirmatory claim cited.

### P4.2 Gate-specific exposure log (NEW v3.1 per Codex round-1 §8)

"Outcome seen" is not a single event. Different exposures for different gates:

| Gate | Exposure events (any one triggers "outcome seen" for that gate) |

1. **§3.2 formula-driven raw n**  
Yes, separating the locked formula from the Phase-0a numerical lock is conceptually sound and removes the v3.0 denominator inconsistency, provided Phase 0a observes only design structure, not outcomes. But v3.1 still has two Gate A fixes before lock. First, S2’s integer effective-n minimum should be **133**, not 132: for zero events, the one-sided Wilson upper bound is `z² / (n + z²)`, so `z=1.6449`, `UCB=0.02` gives `n ≥ 132.58`. NIST’s Wilson formula supports this derivation. Second, the binding analysis should use **final analyzed gold-set DEFF**, or `max(Phase0a_DEFF, final_DEFF)`, computed before outcome unblinding. A pilot DEFF alone is not enough if final cluster sizes drift. Cochrane’s effective-sample-size framing is consistent with `n / DEFF`. ([itl.nist.gov](https://itl.nist.gov/div898/handbook/prc/section2/prc241.htm)) ([training.cochrane.org](https://training.cochrane.org/handbook/current/chapter-23))

2. **§3.4 scenario-family revision**  
It reduces the giant connected-component risk, but it does not fully close it because “pairwise-only family assignment” is not a cluster partition. The DEFF formula using `Σm_i² / Σm_i` assumes non-overlapping clusters. Pairwise relations can be non-transitive: A related to B, B related to C, A not related to C. That cannot be summarized as simple `m_i` without an additional deterministic rule. Either define a partitioning precedence rule, or switch the DEFF calculation to a pairwise-correlation form: `DEFF = 1 + 2ρP/N`, where `P` is the number of related claim pairs. That preserves the pairwise intent and avoids transitive blow-up. Also fix the criterion numbering mismatch between §3.4 and §9. Variable cluster-size effects are real, so this cannot stay ambiguous. ([pmc.ncbi.nlm.nih.gov](https://pmc.ncbi.nlm.nih.gov/articles/PMC8320970/))

3. **§4.4 surrogate ICC protocol**  
The three surrogates are defensible, with source-disagreement frequency the best primary candidate because it is closest to missed-fabrication risk. Evidence completeness and retrieval-tier prevalence are useful but more process-proxy than outcome-proxy. I would add one verifier-native pre-outcome surrogate, such as unsupported-claim score, verifier escalation rate, fallback rate, or verifier confidence dispersion, because customer clustering may enter through verifier behavior rather than retrieval alone. The escalation rule is conservative enough, but two implementation details need tightening: define the one-sided ICC CI method, and replace `ceil(max_observed × 1.5)` with “round up to nearest 0.01 or 0.05.” Mathematical `ceil(0.11 × 1.5)` equals 1.0, which is almost certainly not intended. For proportion surrogates, predefine weighting or a mixed/binomial method because one-way ANOVA on raw proportions can be heteroskedastic.

4. **§5.4 BY default with conditional BH**  
BY default is the right audit posture. BH is valid under independence and certain positive-dependence conditions, while BY is the conservative generalization under arbitrary dependence; that matches the monitoring setting, where shared retrieval, customer mix, evaluator, and prompts make dependence plausible. The four-condition BH gate is sufficient as a conservative permission rule, but it excludes some valid BH cases: a formal analytic proof of independence/PRDS, a locked resampling FDR procedure, or a known positive Gaussian dependence structure. I would allow “analytic proof of BH validity” as an alternative to baseline simulation. Also, the simulation should estimate actual FDR under global and partial-null configurations, not merely the correlation matrix. ([stat.berkeley.edu](https://www.stat.berkeley.edu/~wfithian/fdr-dependence.pdf))

5. **§6.4 prevalence-standardized precision**  
The `{1%, 5%, 10%, 30%}` grid is useful for version comparison, but it should not be the only standardized grid unless production S0/S1 prevalence is plausibly in that range. For safety-critical events, production prevalence may be below 1%, where PPV can collapse even with strong recall/specificity. Add a production-centered log grid, for example `{0.1%, 0.5%, 1%, 2%, 5%, 10%, 30%}`, plus the observed canary prevalence and its CI. Binding on production-canary prevalence is correct, but the PPV interval should propagate uncertainty in both classifier performance and prevalence. Predictive values are prevalence-dependent, so the explicit labeling is methodologically necessary. ([bmj.com](https://www.bmj.com/content/329/7458/168?utm_source=openai))

6. **§7.2 rater-design tie-breaker**  
Complete versus incomplete panel is a reasonable primary axis, but it is not the only relevant one. Incomplete or varying panels strongly favor Krippendorff’s α because it handles missingness and variable rater counts directly; this is supported by simulation literature recommending α when missing data or non-nominal scales are present. With complete panels, AC1 is reasonable for the binary safety-critical gate, especially under rare positive prevalence where kappa-like coefficients can behave poorly. But Krippendorff can still be preferable with complete panels if the design prioritizes a single coefficient family across binary and ordinal analyses. I would lock the current rule, report both, and add a missingness/prevalence audit so the coefficient choice cannot be challenged as opportunistic. ([bmcmedresmethodol.biomedcentral.com](https://bmcmedresmethodol.biomedcentral.com/articles/10.1186/s12874-016-0200-9))

7. **§P3 leakage expansion**  
The expansion is strong, but evaluation-time leakage should be explicit. Also require provider-side data-retention/training opt-out for any external judge API, frozen judge model snapshot, and no use of judge outputs to revise prompts before the gate is complete. This is partly covered by items 18 and 19-31, but not tightly enough for LLM evaluation. Recent LLM benchmark work treats test contamination and judge-related bias as first-order evaluation risks. 8. **§P4 5-category structure + exposure log**  
The structure is workable, and the exposure log does address the “I peeked but didn’t really see it” argument if logging is automatic, append-only, and attached to artifact access, dashboards, notebooks, stdout, CI logs, and analysis exports. Manual self-report is not enough. Split the log into `structural_exposure` and `outcome_exposure`. Amendments touching clustering, eligibility, ICC, or sample size after structural exposure should be restricted, but they should not be mislabeled as post-outcome amendments.
