**§-1.1 Verdict**

Do not lock 0a.-1.D yet. The draft has the right high-level shape, and `content_sha256` is correctly treated as a consistency check rather than source identity. But there are several build-blocking ambiguities that would affect clinical safety and the §3.4 relation builder.

The biggest issue is identity over-collapse. “Registry ID > DOI > PMID > URL” is only safe if the registry ID is the source document’s own identifier. An NCT ID mentioned inside a journal article must not outrank that article’s DOI, or all papers about the same trial can collapse into `nct:*`. Likewise, FDA application numbers and Health Canada DINs are often product/application identifiers, not document identifiers. They belong in `source_subject_ids[]` or `entity_ids[]` unless they identify the actual source document. Add a source-class-conditioned rule: registry/document IDs win only for registry/regulatory source records themselves; literature keeps DOI/PMID.

The `<scheme>:<value>` format is fine, but not sufficient as written. It needs locked value normalization and alias resolution: DOI lowercasing and prefix stripping, PMID-to-DOI crosswalk, URL canonicalization, and `aliases[]` so one citation with PMID and another with DOI still resolve to the DOI canonical ID when recoverable. Preprints later published should be different `canonical_source_id`s, with an optional shared `work_id`/`study_id`, not the same source document. For regulatory/legal/standards sources, add a `source_version_id` or equivalent. Re-snapshot SHA drift should not create a second source, but distinct official document versions must be auditable and not silently collapsed.

The clinical admissibility row currently contradicts repo semantics. [clinical_source_registry.py](C:/POLARIS/src/polaris_graph/clinical_retrieval/clinical_source_registry.py:7) defines clinical T1 as regulatory plus Cochrane/systematic-review paths, T2 as peer-reviewed primary literature, and T3 as registries/guidelines/government health agencies. The global [tier_classifier.py](C:/POLARIS/src/polaris_graph/retrieval/tier_classifier.py:31) uses T1 primary, T2 SR/MA, T3 government/regulatory, which matches the clinical scope template more closely. The draft creates a third mixed meaning. Fix by naming admissible source classes independent of overloaded T labels, or explicitly choose which classifier taxonomy the validator consumes.

The non-clinical admissibility table is also under-specified relative to the scope templates. Due diligence omits patents, named analyst reports, attributed business news, and company press releases allowed by [due_diligence.yaml](C:/POLARIS/config/scope_templates/due_diligence.yaml:11). Policy omits HTA reports, NGO reports, and named policy commentary allowed by [policy.yaml](C:/POLARIS/config/scope_templates/policy.yaml:11). Tech omits preprints, open-source docs, and technical blogs allowed by [tech.yaml](C:/POLARIS/config/scope_templates/tech.yaml:11). AI sovereignty and Canada-US similarly omit policy institutes, vendor sovereign-cloud docs, statistics, and peer-reviewed bilateral literature. If 0a.-1.D intentionally narrows validation admissibility, it must say so and override the templates explicitly; otherwise this is a contradiction.

The validator and exception path need tightening. `canonical_source_id` alone does not “resolve to an admissible class”; class must be independently recomputed from URL, source metadata, pinned classifier version, scope template, and deny rules. `_DENY_DOMAINS` should be hard-deny and non-exceptionable unless governed as a formal amendment. Exception approval should be role-disjoint at the `blinding_unit_id` level from 0a.-1.A, not only from the claim constructor, and should not allow the approver to adjudicate the same unit.

The Jaccard granularity is right: source-document level, not span level. Span-level would overweight heavily quoted documents. But the draft needs to define claim-level versus packet-level sets for 0a.-1.C, plus the empty-set behavior. Also, `landing_vs_pdf` as a flag does not itself prevent double-counting; the canonicalizer must actively alias landing page and PDF to the same `canonical_source_id` when they are the same source, while keeping different PDFs distinct.

The COI linkage is necessary but incomplete. Source publisher/sponsor/regulator `entity_ids[]` feed 0a.-1.A, but a claim’s COI entities cannot be derived only from its sources. Claims also need subject entities mentioned in the claim: companies, drugs/devices, regulators, governments, vendors, comparators, and beneficiaries. This should be a union of source entities plus claim-subject entities, with fail-closed unresolved IDs per [0a.-1.A](C:/POLARIS/state/polaris_statistical_contract/v3_4_phase_0a_0/0a-1A_sme_panel_role_governance.md:100).

So 0a.-1.D does not yet unblock 0a.-1.C. It needs the canonicalizer rules, alias map, version rule, admissibility-class computation, exception schema, and manifest fields before the relation builder can deterministically consume `canonical_source_id`. `source_registry.py` is not contradicted if treated as run-local SRC citation machinery, but the draft should explicitly state that `SRC-NNN` is never a canonical source identity.

Fix clinical admissibility/tier reconciliation; document-vs-study identity; DOI/PMID/URL aliasing and landing/PDF de-dup; versioned regulatory-doc rule; hard-deny/exception governance; and claim-subject COI entity union then lock.
