**§-1.1 Audit Verdict**

Do not lock 0a.-1.A yet. The draft has the right architecture, but redline #3 is not fully closed mechanically.

The biggest gap is in [§2.3](C:/POLARIS/state/polaris_statistical_contract/v3_4_phase_0a_0/0a-1A_sme_panel_role_governance.md:50): the tool only says it blocks constructors and first-pass adjudicators from some forbidden actions. It must also positively authorize actions. First-pass view/write must require `logged_in_sme_id ∈ adjudicator_ids`; tiebreak view/write must require `logged_in_sme_id == tiebreaker_id`; panel actions must require authorized panel role. Otherwise any non-constructor SME can still act, which is an honor-system gap.

The missing “4th enforcement surface” is the controlled artifact/access layer tied to the exposure log. The UI/CLI can hide constructor intent, but if adjudicators can access packet internals, constructor notes, prior labels, or consensus files outside the tool, blinding is not mechanical. 0a.-1.E can implement this, but 0a.-1.A should name it as binding: direct storage/object access must be role-scoped, and the validator/tool must exclude any SME whose exposure log shows a disqualifying read.

The [§2.2 validator](C:/POLARIS/state/polaris_statistical_contract/v3_4_phase_0a_0/0a-1A_sme_panel_role_governance.md:39) also needs more reject conditions. Add rejects for duplicate adjudicator IDs, duplicate live assignment rows for a claim, inactive/unverified SMEs, wrong domain qualification, unresolved COI/entity IDs, COI hits, and exposure-log contamination. For the current two-adjudicator workflow, use `len(set(adjudicator_ids)) == 2`, not merely `len(adjudicator_ids) >= 2`, unless you define a multi-adjudicator consensus rule.

The per-claim invariant in [§1](C:/POLARIS/state/polaris_statistical_contract/v3_4_phase_0a_0/0a-1A_sme_panel_role_governance.md:22) closes same-claim role overlap, but it does not close same-family leakage. If a “scenario-family” shares construction intent, fabricated pattern, evidence packet, or batch notes, then a constructor on sibling claim A should not adjudicate/tiebreak sibling claim B. The clean fix is to introduce `blinding_unit_id`; role-disjointness applies across that unit. If the unit is truly one claim, fine, but the manifest must say so.

Credential floors are broadly defensible, but need roster-verifiable evidence tags. Clinical is acceptable for literature/evidence adjudication, consistent with the FDA/ICH GCP principle that clinical responsibility rests on education, training, and experience, but clinical claims should require relevant specialty tags and, for treatment/pharmacotherapy/device claims, at least one MD/PharmD-qualified reviewer in the adjudication path. `ai_sovereignty` is under-specified: replace “demonstrated expertise” with enumerated qualifying evidence such as AI governance program leadership, standards work, regulatory/procurement work, peer-reviewed/public policy output, cloud/data-residency expertise, or AI risk evaluation experience. NIST AI RMF’s governance guidance supports documented roles, training, and multidisciplinary expertise, not vague assertion. Sources: [FDA E6(R3) GCP](https://www.fda.gov/media/169090/download), [NIST AI RMF Core](https://airc.nist.gov/airmf-resources/airmf/5-sec-core/).

The [§4 COI rule](C:/POLARIS/state/polaris_statistical_contract/v3_4_phase_0a_0/0a-1A_sme_panel_role_governance.md:72) is not yet mechanically enumerable because `declared_interests` are free text. This does not need to wait entirely for D4, but it does need canonical IDs. Use separate `entity_ids` for companies, sponsors, drugs/devices, governments, regulators, vendors, etc.; use D4/microtopic IDs for topic-level advocacy conflicts. The validator should fail closed if claim entities/topics are unresolved.

Blinding in [§5](C:/POLARIS/state/polaris_statistical_contract/v3_4_phase_0a_0/0a-1A_sme_panel_role_governance.md:78) hides labels, but packet curation can still leak intent. Require label-symmetric packet construction: same builder, same source-selection rules, same facet generation, no constructor-authored spotlighting, and no packet metadata that reveals fabrication rationale. The tiebreaker must also receive the same blinded packet and not see prior labels until their own label is sealed.

The [§6 tiebreak path](C:/POLARIS/state/polaris_statistical_contract/v3_4_phase_0a_0/0a-1A_sme_panel_role_governance.md:84) is directionally right. A 3-way ordinal split such as S1/S2/S3 should go to panel; do not median/average severity unless the rubric later explicitly defines that. But define whether majority is per-field or over the whole label tuple. If any required field lacks majority, panel. Also, panel escalation should include domain-qualified SME adjudication, not only operator + Codex, especially for clinical gold labels.

Panel size is the clearest operational miss. [§3](C:/POLARIS/state/polaris_statistical_contract/v3_4_phase_0a_0/0a-1A_sme_panel_role_governance.md:70) counts 2 IRR + 1 tiebreaker but omits the constructor, who is also an SME and must be disjoint. Minimum per claim is 3 unique eligible SMEs for constructor + two first-pass adjudicators, and 4 if tiebreak is a live path. With COI and scheduling, target should be at least 5-6 per domain, not 3. “Minimum 2 to operate” is false under the current role model.

Fix §2 positive authorization + access-layer/exposure-log enforcement, §2.2 validator completeness, family-level `blinding_unit_id`, enumerable COI IDs, §5 packet-leakage controls, §6 tiebreak/panel semantics, and §3/§7 panel-size minima; then lock.
