I would not lock D8 v0 yet. The core design is right, but the current text leaves too much operator discretion and has one build-order inconsistency.

The major blocker is the P/N validation timing. D8 says to run the relation-builder on the planned allocation, but also says the manifest with P/N/DEFF is hash-pinned “before construction begins” ([D8 §4](C:/POLARIS/state/polaris_statistical_contract/v3_4_phase_0a_0/D8_pilot_allocation.md:45)). That is not buildable as written: the relation-builder needs `construction_manifest` and `source_packet_manifest` fields such as cited source IDs, packet class, prompt families, microtopics, constructor SME, and construction window. Those do not exist before structural construction. Fix by making this a two-stage custody path: first, hash-pin the allocation plan before construction; second, after structural manifests are populated and validator-passed, but before adjudicator/verifier outcome exposure, run the relation-builder and hash-pin P/N/DEFF.

The second blocker is quota underspecification. “Pre-registered quota per cell” is directionally correct, and sizing by severity stratum while spreading domain/complexity/evidence/fab-type by quota is the right approach. Sizing every full factorial cell would be impossible and statistically wrong. But D8 must define the quota generator or quota table requirements now: hard vs soft marginals, rounding rule, minimum coverage rule, how SUPPORTED cells are keyed with no fabrication type, infeasible-cell handling, and how D6 template targets aggregate into D8 quotas. “Spreads coverage” is not enough for a final clinical-safety-critical design lock.

The third blocker is the resample-vs-expand rule. The target-band concept is right if it means “the DEFF/raw-N assumption used for §3.2 sizing,” but the current “RESAMPLE OR expand raw N” leaves an unregistered choice. Fix with a deterministic rule, for example: accept the first seed draw whose per-stratum DEFF is within the pre-registered upper bound; resample up to K attempts using sequential draw indexes; if still out-of-band, expand raw N for the failing stratum to `ceil(n_eff_required × observed_DEFF)` or trigger the defined §P4 Category-3 amendment. Also define whether the lower bound matters for representativeness or whether only the upper bound is binding for safety.

The intended-stratum P/N dry-run is sound only as a design-stage structural check. It should not be called the normal `severity_stratum_manifest`, because 0a.-1.C defines that as consensus-gold-derived. Add a distinct `allocation_stratum_manifest` or equivalent field with `stratum_source = planned_intent`, custody-controlled and disjoint from adjudicators. Then require the final pre-unblinding Gate A DEFF run to use the true consensus-derived `severity_stratum_manifest`, preserving contract §3.2’s `DEFF_binding = max(DEFF_Phase0a_pilot, DEFF_final_gold_set)`.

Severity allocation is mostly consistent: S1/S2 sized for n_eff × DEFF, S0 sentinel ≥50, SUPPORTED non-fab controls, and S3 excluded all align with §3.2, 0a.-1.B, and D6. The gap is matched-control specificity: 0a.-1.B says the matched-control ratio is pre-registered in D8, but D8 only says `SUPPORTED ≥ fabricated`. Pre-register the exact ratio and pairing/grouping rule.

SME/template balancing is feasible only if it is implemented as hard constraints plus soft balance penalties. Role-disjointness and COI are non-negotiable; load/template balance must yield when needed. At ~18 SMEs total, feasibility is brittle and may fail under COI/tiebreak/clinical MD-or-PharmD constraints; D8 should require a roster feasibility precheck and fail closed if no assignment exists.

Custody is close but should explicitly include input manifest hashes, code pins, solver/randomizer version, objective weights, constraint set, seed/draw index, infeasible attempts, relation-table hash, planned-stratum manifest hash, roster/template/prompt/ontology hashes, and matched-control IDs. 0a.-1.E covers much of this globally, but D8 should point to it explicitly.

The run gate is also missing dependencies. D8 correctly gates on D4-seed, D6-content, D5-inventory, and roster, but it should also name D7' / source-packet manifest readiness and the post-E accepted relation-builder code + edge-fixture run. The header/§7 dependency list also omits D5 and D7', even though D8 consumes prompt families and packet class.

Fix: two-stage allocation/P/N custody; exact quota generator/table rules; deterministic target-band and resample/expand rule; separate planned-intent stratum manifest plus final consensus-stratum rerun; exact SUPPORTED matched-control ratio; roster feasibility precheck; expanded manifest hashes/code pins; add D5/D7'/relation-builder-fixture gates then lock.
