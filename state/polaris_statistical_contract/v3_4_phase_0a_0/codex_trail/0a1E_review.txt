**§-1.1 Audit Result**

Do not lock yet. The draft is close in shape, and it correctly carries forward the main custody requirements from the locked prerequisites, but I found several build-blocking custody gaps.

The largest issue is **§3 seed handling**. A single pinned root seed is fine; a single reused seed/stream for allocation plus all bootstrap procedures is not. Allocation randomness and CI/bootstrap randomness should be reproducible but domain-separated. Use one master seed, then derive named child streams, e.g. `HMAC(master_seed, "D8-allocation")`, `HMAC(master_seed, "GateB-bootstrap")`, `HMAC(master_seed, "GateD-bootstrap")`, etc. That preserves reproducibility without accidental stream coupling or repeated identical pseudo-random sequences across independent procedures.

**§1-§2 hashes/code pins are not complete.** The draft pins the obvious custody code, but several result-affecting surfaces are still missing: JSON Schema files and schema validator code; D8 allocation/min-cost matching code and allocation manifests; Gate A/B/D/E analysis runners; bootstrap/CI implementation; ICC/GLMM/surrogate-analysis code; consensus/tiebreak/panel label-resolution code; IRR/agreement computation code; source fetch/snapshot code; admissibility classifier/tier code and deny-list config; URL param-strip/canonicalization config; source exception records; roster and assignment manifests; and the hash/exposure/replacement registry schemas themselves. 0a.-1.D also requires a pinned classifier version and deny rules, but 0a.-1.E only names the admissibility validator.

**§4 ACLs need one important blinding fix.** The table correctly keeps constructor intent sealed, hides other first-pass labels, and gates consensus labels. But `severity_stratum_manifest` is derived from consensus and exposes claim-level `SUPPORTED/S0/S1/S2`. It is structural for design, but it is still answer-bearing for an adjudicator. Any actor who reads claim-level severity strata for a blinding unit must be barred from first-pass/tiebreak/panel adjudication for that unit, or the `builder/design team` roles must be mechanically disjoint from adjudicators. As written, 0a.-1.A’s validator disqualifies reads of constructor intent, consensus, or another adjudicator’s first-pass label, but not severity-stratum reads.

**§4 also has a small buildability gap.** Packet access should explicitly include tiebreakers and authorized panel SMEs for their assigned claims. “Assigned adjudicator” may be intended to cover this, but 0a.-1.A treats first-pass adjudicator and tiebreaker as distinct roles.

**§5 packet snapshots are directionally correct.** Storing the rendered bytes plus hash closes the “renderer hash is not what the SME saw” gap, provided the snapshot is self-contained: final HTML/text, cited spans, embedded assets/CSS/fonts or their hashes, render parameters, and any screenshot/PDF if the visual rendering matters. If the HTML still depends on external assets or live browser/client rendering, the gap remains.

**§6 exposure logging is correct in principle but incomplete as an implementable map.** Reads must be logged, not just writes, and the listed channels cover the core paths. However the structural/outcome table should import the full contract §P4.2 event map, not just examples. Missing explicit outcome examples include under-rating confusion matrices and retrieval/extraction recall numerators. Missing channels to name: temp files, browser/download/cache artifacts, backups/replicas, data warehouse/BI copies, crash/trace/observability logs, ticket/chat/email attachments, LLM/agent prompts/transcripts, and object-store/CDN signed-URL access logs.

**§8 replacement/drop rules are mostly safe but need tighter preregistration.** Reallocation is safe only if it happens pre-outcome, from the same D8 cell, using the next deterministic allocation draw/child seed, and without looking at verifier outcomes or agreement coefficients. Source 404 handling should distinguish pre-SME from post-SME: after a rendered packet snapshot has been shown, reconstructing the packet changes what was adjudicated and should not silently replace the evidence. Builder-run rejection is safe if failed runs emit only validation errors; if P/N/DEFF are visible before fixes, reruns can become structural tuning.

**§9 is not fully airtight yet.** The draft says the exposure log enforces the boundary, but logs record; they do not deny access. The spec needs an explicit deny-by-default access broker/storage policy: outcome-class reads are blocked until the gate state reaches the registered outcome-exposure point, and every direct storage/export/notebook/CI path must go through that broker or equivalent IAM policy. Otherwise there is still honor-system residue.

**Upcoming 0a.-1.C code half:** the sequencing is right: C-code and edge-fixture runs must occur only after E. But because E currently omits several code pins, child seed derivation, severity-stratum disqualification, and explicit deny enforcement, it does not yet fully govern that run.

Fix seed child-stream derivation; add the missing result-affecting artifact/code pins; disqualify severity-stratum readers from adjudication or make builder/design roles mechanically disjoint; expand packet snapshot self-containment; make §P4.2 exposure classification a complete machine map; add missing read channels; tighten replacement rules; and specify deny-by-default outcome access enforcement, then lock.
