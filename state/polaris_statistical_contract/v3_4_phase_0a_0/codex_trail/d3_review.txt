**§-1.1 Audit**

D3 is directionally correct, but I would not lock it as written. The bin design itself is sound; the remaining problems are buildability/reproducibility edges.

The E1-E4 boundaries in [D3_evidence_pool_bins.md](C:/POLARIS/state/polaris_statistical_contract/v3_4_phase_0a_0/D3_evidence_pool_bins.md:24) are inclusive, non-overlapping, and exhaustive for positive packet sizes 1-80: E1 1-5, E2 6-20, E3 21-40, E4 41-80. There is no 5/6, 20/21, or 40/41 off-by-one. The missing case is 0. Since `construction_manifest.evidence_pool_bin` only permits E1-E4 and `source_packet_manifest.canonical_source_ids` currently has no `minItems`, D3 must explicitly say: 0 admissible distinct sources is not E1 and not E0; it is a fail-closed construction error requiring reconstruction/replacement before the claim can enter the gold set. A fabricated claim with no admissible cited evidence should not reach a packet, because that breaks label-symmetric packet construction and gives the adjudicator an obvious no-evidence tell.

The packet cap is correctly not lifted. Repo grounding matches the draft: `top_k_per_query=50` in [thresholds.yaml](C:/POLARIS/config/settings/thresholds.yaml:106) and `max_sources_per_vector=300` in [sota_parameters.yaml](C:/POLARIS/config/settings/sota_parameters.yaml:61). Eighty is a packet/adjudication ceiling, not a retrieval ceiling. Keeping it below 300 avoids the context-handling confound from making large packets partly a model/SME context-load test instead of an evidence-pool-size stratum.

The downselect concept is acceptable, but §3 is not yet reproducible enough. “Top-N” needs a source for `N`: either always top 80 when retrieval exceeds 80, or a pre-registered/recorded `target_packet_source_count` chosen before downselect. The example `retrieved_source_count=250` downselected to a 60-source packet implies `N` can be less than 80, so it cannot be left implicit. Also, hash-pinning the rule and seed is not enough unless the pre-downselect input universe is pinned: canonical IDs, admissibility class/tier, authority score/rank inputs, selected/dropped flag, and algorithm/code hash. A count alone tells that sources were dropped; it does not let an auditor reproduce which ones.

Authority-ranked downselect is biased by design: it will preferentially drop lower-authority admissible sources. That bias is acceptable for this clinical-safety context if it is label-symmetric, pre-outcome, applies after canonical dedup/admissibility, respects scope-template caps/context rules, and is auditable through the full downselect manifest. It should not be replaced with random selection, but the rank tuple must be exact: e.g. scope-template authority/tier priority, then descending pinned `authority_score`, then lexical `canonical_source_id`, with missing/NaN score behavior fail-closed or explicitly ordered.

Binning from the constructed packet is the right basis. D3 is a packet property; D2 is a pre-retrieval question/claim-complexity property. So a retrieved universe of 250 downselected to a locked packet of 60 is E4, not “250-sized,” and `retrieved_source_count` must never drive `evidence_pool_bin`. This is orthogonal to D2 as locked in [D2_complexity_rubric.md](C:/POLARIS/state/polaris_statistical_contract/v3_4_phase_0a_0/D2_complexity_rubric.md:9): source count belongs only to D3.

`retrieved_source_count` is structural, not outcome, if it is computed pre-outcome from production retrieval before verifier/adjudication labels. It can correlate with domain or task shape, but it is not a miss count, recall numerator, agreement value, or label-derived field. It should be added to 0a.-1.E’s structural exposure handling and to the construction metadata schema. As written, [construction_manifest.schema.json](C:/POLARIS/state/data_lineage/schemas/construction_manifest.schema.json:7) has `additionalProperties:false` and no `retrieved_source_count`, so D3’s forward note requires an actual 0a.-1.C schema/custody addendum before execution.

Fix zero-source fail-closed handling, downselect N/rank/input-manifest pinning, and the retrieved_source_count/schema-custody addendum then lock.
