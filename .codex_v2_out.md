# Deep Cove Research — Code Review Readiness Plan (v2)

**Goal:** make the code clean and professional enough to pass an independent Telus code review — **without ever changing the pipeline's score or its faithfulness behavior.**

**Date:** 2026-07-18 · **Status:** for owner approval

**Why "v2":** v1 was written by a research+audit workflow. Then codex (GPT-5.6) reviewed it independently and corrected it in **four** ways:
1. **Measure a baseline FIRST** — before touching any code.
2. **"Byte-identical" needs a real test**, not just keeping the old default values.
3. **Some items were rated too scary** (messy names and docs are tidiness, not danger).
4. **Reorder** so nothing risky happens before the baseline exists.

---

## The one rule (never break this)

The running pipeline's **RACE score** and its **faithfulness behavior** must stay **exactly the same** through every change. No change may move a threshold, a model name, a temperature, or a verdict. If we cannot *prove* a change is safe, we do not ship it.

---

## The good news: the code is better than it feels

Before the problem list — an honest look shows we already have a lot:

- **96%** of functions have type annotations.
- **98.5%** of modules have a top docstring.
- **1,082 tests** already exist.
- **Secrets are clean** — no keys in git; `.env` is ignored.
- The "borrowed" parts (STORM, FS-Researcher) are **our own rewrite from the papers, with citations — not copied code**. This is a *naming* choice, not a legal problem.

So the real gap is **not substance**. It is **discipline a reviewer can see**: tests that don't run automatically, settings scattered everywhere, messy names, and a few loose ends.

---

## The real problems (honest severity)

1. **HIGH — Tests are not enforced.** 1,082 tests exist but **never run in CI**. No lint, no type-check, **no `pyproject.toml`**. To a reviewer, "tests that don't run" looks the same as "no tests." Also all 28 CI actions use moving tags (a security weakness).
2. **HIGH — Settings are scattered.** **1,707 `os.getenv` calls, ~923 different settings, only 24 written down.** Model names and numbers are hardcoded in 9+ files. There is no single control panel.
3. **HIGH — One real architecture mess:** a live **3-way fork** (`graph.py` / `graph_v2.py` / `graph_v3.py`) chosen by an env var. **This is the one genuinely risky thing in the whole codebase.**
4. **MEDIUM — Messy names (tidiness, not danger):** `honest_*`, `_v2` files with no `v1`, two dead files. Cosmetic, but a reviewer notices. *(Codex lowered this from HIGH.)*
5. **MEDIUM — A checkpoint built but never turned on:** the code to save a generation checkpoint exists, but **nothing calls it**, so every restart re-runs the 30–40 min generation.
6. **NARROW — Docs gaps in specific places only:** ~530 missing docstrings, mostly in the contract/API files. Overall docs are fine; this is targeted, not broad. *(Codex lowered this from broadly HIGH.)*

## The plan — step by step

Each phase says: **what we do**, **why**, and **how we stay safe**. The referee for "did the pipeline change?" is the acceptance harness (`acceptance_outline_agent.py`), plus a full score replay for anything touching the graph fork.

### Phase 0 — First, MEASURE. Then do only the safe tidy-ups.

This is the change codex insisted on: **before touching any code, prove exactly where we stand.**

**0A. Measure the baseline (read-only — cannot hurt anything):**
- Run the full test suite in a fixed environment. Record: does it pass? how long? any flaky tests?
- Run the acceptance harness and save the result as the **"before" baseline**.
- Save a snapshot of how every setting resolves today (what value each env knob gives for *unset / empty / normal / bad* input).
- Record how all **3 graph versions** (v1/v2/v3) behave and which one actually runs.

**0B. Do only the changes that touch NO running code:**
- **SHA-pin all 28 CI actions** (lock to exact versions) — safe, and closes a security finding.
- **Fix third-party attribution** (license files, notices). The `third_party/` folder is never imported → zero risk.
- **Add `pyproject.toml` + a CI workflow** (lint, type-check, tests) in **report-only mode** — it shows results but never blocks anything.
- **Tidy the README / architecture doc.**

**Codex correction (important):** we do **NOT** delete the "dead" files yet, and we do **NOT** change secret types yet. Both *looked* zero-risk but aren't — dead files can still be loaded by dynamic imports or outside code, and changing a secret to a `SecretStr` type changes runtime behavior. They wait until after the baseline.

**Why Phase 0 first:** highest reputation for zero risk, and every later "we didn't move the score" claim depends on the baseline existing.

---

### Phase 1 — Build one central settings file (careful, NOT "mechanical")

Build one typed settings file (`settings.py`) that controls model choice, numbers, and secrets — a real **master control panel**, which is exactly what you asked for.

**Codex correction:** this is **not** a simple copy-paste. Moving ~923 settings into one object can *quietly* change behavior — how empty values, upper/lowercase, type conversion, and *when* a value is read are handled. So:

- **Build a truth table first.** For every setting, record what today's code returns for *unset / empty / normal / bad* input, and lock that behavior with a test. (Not just "the default value matches" — the *whole behavior* matches.)
- **Migrate one file at a time.** Keep every env name exactly the same. Each default equals today's exact value.
- **Run the acceptance harness after every file.** If "after" differs from the "before" baseline → stop and fix.
- Make the **model list the single source of truth** (one place names all the models).
- Auto-generate a documented `.env.example` so all ~923 knobs are finally written down.

**Highest-risk spots to go slow on:** `state.py` (114 env reads) and the model resolver (a wrong model name trips a self-check guard). **Never change a value in the same commit that moves it.**

---

### Phase 2 — Clean names + turn on the checkpoint (both safe, reversible)

**2A. Rename messy names.** `honest_*` → plain names, drop `_v2` where there's no `v1`, delete the confirmed-dead files (now that the baseline exists).
- **Only rename code symbols and file names — NEVER the env-var text or the stage names.** Those text strings control the pipeline; renaming one silently disables a feature and moves the score.
- One rename per commit; acceptance harness before/after each.

**2B. Turn on the generation checkpoint.** Save only **pre-check data** (drafts, outline) — **never a verdict** — so faithfulness still runs fully on restart. **Gate the save behind the flag**, so a normal run stays exactly the same (this closes codex's point that "an extra file on every run" isn't truly identical).

---

### Phase 3 — Docs, then enforce CI, then the one dangerous change (LAST)

- **Docs:** fill the ~530 missing docstrings (contract/API files first). Add decision records (ADRs) for the locked design rules. Pure text, zero score risk.
- **Enforce CI:** flip CI from report-only to **required** — but only **after one fully green run**. Add dependency hash-pinning, license/secret scanning.
- **The graph fork — last and most careful.** Codex's warning: **one replay is not enough proof.** Before deleting any graph file: build a version-by-version comparison, confirm which version is really used, test any saved-state migration, and keep a rollback window. This is its own separate, reviewed change.

## The safety rules (apply to every change)

1. **Baseline first** — no runtime change before the "before" measurement exists.
2. **Every setting** defaults to today's exact value and keeps its exact env name; a **truth-table test** proves the whole behavior, not just the default.
3. **No behavior change** in the same commit as a rename or a settings move.
4. **Renames change symbols/filenames only** — never env text or stage names.
5. **Checkpoint saves pre-check data only, never a verdict; off by default.**
6. **Acceptance harness before and after** every code-touching change; the graph fork also needs a full score replay.
7. **CI starts report-only,** becomes required only after one green run.
8. **Do not rename real domain terms** (SmartArt, deep_crawl, OpenAlex, PubMed, Semantic Scholar). Keep paper citations.

## Questions the reviewer will ALSO ask (new — from codex)

The code may be fine, but we should have answers ready:

- **Security:** API login/permissions, crawler safety (SSRF), prompt-injection handling, PII, log redaction, checkpoint retention.
- **Operations:** timeouts, retries, rate/cost limits, monitoring, runbooks, failure recovery.
- **Compatibility:** which Python versions, how to install, how saved state migrates, rollback.
- **Test quality (not just count):** does the suite pass today? runtime? flakiness? real coverage?
- **Settings governance:** the ~923 knobs need labels (supported / internal / secret / deprecated) and owners — a list is not a contract.
- **Reproducible builds:** can the exact same build be rebuilt? (An SBOM alone doesn't prove this.)
