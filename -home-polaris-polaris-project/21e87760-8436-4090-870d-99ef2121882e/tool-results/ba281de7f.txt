## Area 1 — source/scope gate

### End-to-end finding

The actual path was:

```text
Task prompt
  → constraint-extraction capability recognizes journal/English requirements
  → retrieval projection adds scholarly-looking queries/routes
  ╳ hard typed eligibility policy is not propagated through every caller
  → Serper + Semantic Scholar + additive OpenAlex candidates
  → body fetch occurs
  → topic/content relevance demotes but keeps
  → source-type classification occurs after fetch
  ╳ source-quality/RQ eligibility not called by audited compose
  → prebuilt corpus loaded directly
  → all retained section prose + all bibliography rows assembled
  → non-journal sources reach report
```

But that does **not** prove the typed intake contract was bound to this acquisition. ### 7. Did journal filtering ever work?

**Yes, but not the intended smart fetch-time design.**

Git history shows:

- `2b42bc5f` on 2026-07-07 introduced the old journal-only hard filter;
- `b67506a3` on 2026-07-17 generalized/retired it;
- `3d891790` on 2026-07-21 added the newer evidence-substrate eligibility levers.

The current `journal_only_filter.py` is deliberately retired and returns inactive/identity behavior ([retirement](/home/polaris/wt/faithoff/src/polaris_graph/nodes/journal_only_filter.py:76), [identity filter](/home/polaris/wt/faithoff/src/polaris_graph/nodes/journal_only_filter.py:293)). Historical logs show it reducing 59 items to 11, excluding 48, then aborting because only five journal sources remained against a twelve-source adequacy floor ([session_log.md:5144](/home/polaris/wt/faithoff/logs/session_log.md:5144)). The ensuing review rejected that tunnel-vision/starvation behavior ([session_log.md:5155](/home/polaris/wt/faithoff/logs/session_log.md:5155)). A prior audit nevertheless confirms that the old mask really removed 15 of 23 non-journal sources ([journal audit](/home/polaris/wt/faithoff/outputs/audits/I-ready-017/journal_only_s11_audit.md:14)).

Therefore:

- A narrow **post-fetch journal-only mask** once worked and caused evidence collapse.
- The intended **prompt-derived, general, fetch/search-stage eligibility system** has never been fully wired on this path.
- The current behavior is not a regression from a working smart gate; it is an incomplete replacement for the retired narrow filter.

### Correct general fix

The fix must be an acquisition contract, not another report-time screen:

1. **Compile one typed `AcquisitionContract` from the task.** It should represent source type, language, date interval, geographic/entity scope, publication status, and requested quality semantics as `{operator, strength, normalized value, exact prompt span}`. Deterministic parsing handles explicit clauses; an LLM fallback interprets genuinely opaque constraints. No task numbers, industry lists, journal lists, or fixed descriptive-word lists.

2. **Make the contract mandatory on every acquisition route.** The same contract/hash must reach primary retrieval, outline-gap retrieval, FS-researcher, IterResearch, CRAG/deepeners, recovery/refetch, and resume. A caller without it must fail acquisition for a constrained task rather than silently issue unscoped searches.

3. **Select backends by required metadata capability.** For a hard journal-article constraint, use scholarly indexes and metadata resolution as the admission path. Generic web search may discover a DOI or canonical record, but the web page itself is not corpus-eligible.

4. **Apply metadata eligibility before body fetch.** Definite source-type, language, or date mismatches are rejected before download. Unknowns enter metadata resolution—not the corpus—until publication type, canonical version, language, and date are resolved.

5. **Validate at corpus admission.** If body-language inspection or canonical metadata contradicts the pre-fetch record, reject that candidate before corpus commit and immediately backfill it. This is acquisition-stage eligibility, not post-generation content removal.

6. **Define quality from prompt semantics plus observable source-independent features.** Use publication status, document type, peer-review evidence where available, retraction/version state, stable bibliographic identity, and task relevance. Do not encode named venues or word lists as proxies.

7. **Backfill by prompt-derived coverage facets.** Continue retrieval until qualified evidence covers the task’s requested dimensions. Filtering without replacement repeats the historical 59→11 failure and damages Comprehensiveness.

8. **Emit an acquisition receipt.** It must contain the prompt-contract hash, backend filters, candidate decisions and reasons, unresolved count, qualified count per facet, and canonical-version substitutions. Prebuilt compose must require a matching receipt or rebuild the corpus; `not_evaluated_prebuilt_corpus` cannot be accepted for a constrained run.

9. **Test generality metamorphically.** Paraphrase source, language, and date constraints; test mixed hard/soft requirements and prompts with no source constraint. The invariant is that the task’s own contract controls acquisition, while unconstrained tasks remain broad.

### Expected RACE effect

The gate directly targets the 0.25 Instruction-Following dimension. Task 72 assigns 0.15 of that dimension to journal-only compliance and 0.10 to English-only compliance ([criteria](/home/polaris/wt/outline_agent/third_party/deep_research_bench/data/criteria_data/criteria.jsonl:72)). It also affects the Comprehensiveness criterion for depth and representativeness of qualified literature.

But there are two important limits:

- The RACE cleaner removes citation markers and the entire references section, so the judge cannot directly inspect most source identities. Direct IF gains from cleaner bibliography compliance may therefore be muted.
- Filtering alone can reduce Comprehensiveness. Only replacement with broader, stronger qualified evidence can improve both dimensions.

My estimate is **0 to +0.005 for the gate alone**, likely sub-noise; **+0.007 to +0.015 when it also replaces weak sources with broad, higher-value journal evidence**. A +0.02 move requires that improved evidence materially changes the synthesis, not merely the bibliography.

## Area 2 — K3 reasoning and completion budgets

### Critical correction to the seed finding

The configured 2K/16K K3 reasoning caps are mostly **not effective K3 caps**.

`OpenRouterClient.generate()` calls the internal request path with `reasoning_enabled=False` ([client generate](/home/polaris/wt/faithoff/src/polaris_graph/llm/openrouter_client.py:3357)). The client sends a reasoning configuration only for `_ALWAYS_REASON_MODELS`, explicitly enabled calls, or `_REASONING_FIRST_MODELS` ([request branches](/home/polaris/wt/faithoff/src/polaris_graph/llm/openrouter_client.py:2010)). That registry contains GLM and DeepSeek models but **not** `moonshotai/kimi-k3` ([registry](/home/polaris/wt/faithoff/src/polaris_graph/llm/openrouter_client.py:821)).

Consequently:

- `reasoning_max_tokens=2048` or `16384` supplied by these callers is ignored for K3.
- K3 uses the provider’s default always-thinking behavior.
- The enclosing `max_tokens` is the effective combined reasoning-plus-visible-output ceiling.
- That explains matrix responses with 5,997 reasoning tokens inside a 6,000-token total ceiling.

The existing globals `PG_REASONING_FIRST_MIN_MAX_TOKENS=32768` and `PG_REASONING_FIRST_HARD_CAP=384000` do not protect K3 because it is absent from that registry ([config](/home/polaris/wt/faithoff/src/polaris_graph/config_defaults.py:744)). The token-limit resolver’s static model table also omits K3 ([resolver](/home/polaris/wt/faithoff/src/polaris_graph/llm/token_limit_resolver.py:129)).

### Correct K3 policy

Current official metadata identifies K3 as a reasoning model with a 1,048,576-token context window, reasoning enabled by default, supported reasoning effort up to `max`, and pricing of $3/M input and $15/M output; OpenRouter currently reports no published top-provider completion cap rather than an unlimited cap ([OpenRouter model API](https://openrouter.ai/api/v1/models), [K3 model page](https://openrouter.ai/moonshotai/kimi-k3-20260715)). Moonshot’s own configuration identifies K3 as `always_thinking` and supports `max` reasoning effort ([Moonshot configuration](https://moonshotai.github.io/kimi-code/en/configuration/config-files.html)).

I recommend:

- **Client capability:** recognize K3 through provider/model capability metadata, not only a hardcoded slug set. Send `reasoning.effort="max"` explicitly and register all K3 call types in tracing.
- **Preflight:** query model and endpoint metadata, then run a completion-cap smoke. Treat a null published cap as unknown, not unlimited.
- **Main sections:** request a **64K combined completion ceiling**, effort `max`, with a context-aware bound of `validated_provider_cap` and `context − input − safety margin`. This is insurance; the audited sections did not hit 24,576.
- **Synthesis matrix:** use at least **32K**, preferably **64K**, because a prior direct K3 audit used 28,862 completion tokens—23,739 reasoning plus roughly 5,123 visible content ([K3 audit](/home/polaris/wt/faithoff/research/audits/deepaudit_kimi.md:1)). Retry from scratch on `finish_reason=length`, increasing to the validated cap. Never silently omit all matrices.
- **Compact K3 side calls:** 32K combined with effort `max` if K3 remains the selected model. Better still, make deterministic extraction and rendering handle tasks that do not benefit from long-form reasoning.
- **No blanket global-max rewrite:** use per-call-class K3 policy. A universal 384K request would worsen latency/capacity without improving terse tasks.

There is no pricing tier cliff. The real cliff is architectural: when K3’s reasoning fills the shared completion ceiling, visible content drops to zero. Reasoning is billed as output at $15/M. Fully consuming 64K costs approximately $0.96 per call, or $9.60 for ten matrix calls; actual usage, not the cap, is billed. Latency and the current single-provider/capacity situation are the practical downsides. The audited sections already took roughly 276–440 seconds each ([first](/home/polaris/wt/faithoff/outputs/all_levers_run.gen.log:4672), [last](/home/polaris/wt/faithoff/outputs/all_levers_run.gen.log:4748)).

**Magnitude:** fixing matrix truncation is necessary. Raising the main cap alone is likely **0 to +0.003**, sub-noise. Combining correct K3 handling with focused high-value evidence and an explicit synthesis prompt is plausibly **+0.010 to +0.020**.
