## FS-Researcher winning mechanisms (why it beats frontier)
1. Dual agent: Context Builder (librarian → hierarchical KB bigger than context) + Report Writer (composes section-by-section, KB = sole fact source). Merging them: -10.35 RACE, Insight -16.89.
2. KB = two-level topic tree (index.md) accounting for EVERY note; + comparative_analysis/ cross-source notes.
3. Checklist loop: "Exhaustive Coverage — can I raise a question the KB cannot answer?"; "Information Density — specific data or vague?". Failed item → next round fixes exactly that gap.
4. Section-by-section writing (one section per session). One-shot: -5.13 RACE, Insight -8.80, Readability -4.80.
5. UNBOUNDED writing stage — "no budget limit"; most wall-clock is here.
6. Test-time scaling: more builder rounds → bigger KB → longer denser report (notes 47→75→98, chars 21,306→29,245, citations 18.5→37.1, RACE 51.18→53.05).
7. Harness > backbone: same Gemini-2.5-Pro = 31.9 bare, 49.7 official harness, 52.5 in FS-R. The FRAMEWORK is worth +2.8 RACE over a frontier product → license to believe our loop can beat them.

## ACCEPTANCE (SIGN_OFF) — two separated classes per §-1.1.1
(A) STRUCTURAL FLOORS (accounting facts, disclosed in cp5 payload; NOT quality verdicts; gate refuses if any fail):
- cap_primary==0, no --sections subset.
- Basket coverage ≥95% of credible baskets have ≥1 citation in body; each uncovered basket listed by id+reason (fail loud).
- Distinct works cited ≥90% of distinct credible works.
- Words: body 9,000-18,000; no section <~600w unless its basket supply is honestly small (disclosed undersupplied).
- Structure: every section with ≥8 baskets renders ≥2 named H3 sub-themes; residual ≤5% of baskets.
- Faithfulness unchanged & only hard gate; zero unlabeled unverified sentences; every deletion disclosed in Methods.
(B) CONTEXT-LEVEL READ (the only quality verdict; a FRESH judge, never the builder): read whole report line-by-line vs cited spans AND vs ChatGPT-Scoped + Gemini-Scoped on the same question; confirm (1) per-basket paragraphs carry Gemini-grade depth (method+N+numbers+interpretation, judged by MEANING not keyword count); (2) RACE verdict ours ≥ BOTH competitors on Comprehensiveness & Insight, meets required-aspect coverage, readable; (3) no chrome/quote-dump/off-topic. SIGN_OFF = all (A) AND (B). Faithful-but-thin FAILS (usable is the line).

## NORTH STAR — KEEP IT SIMPLE (operator 2026-07-11, binding on all design)
Do NOT overcomplicate. The whole design is one sentence: **the OUTLINER is an AGENT with mighty TOOLKITS and a mighty CORPUS.**
- ADOPT a proven open-source agentic toolkit (from the toolkit research) as the outliner's brain — do NOT build a bespoke orchestration from scratch.
- Give that agent our TOOLS (search, fetch, weight/select, code-execute/compute) and our CORPUS (the baskets = its knowledge base).
- It plans + builds the report; when it needs more, it uses its tools to search/fetch/compute more; it folds results back.
- Anything more elaborate than 'agent + tools + corpus' is over-engineering — cut it. General across domains, faithfulness the only hard gate.

## ADOPTED AGENTIC TOOLKIT (research 2026-07-11) — smolagents
The outliner's agent brain = HuggingFace **smolagents** CodeAgent (github.com/huggingface/smolagents, Apache-2.0, v1.26.0 2026-05-29, ~28k stars). ONE toolkit does BOTH: agentic search AND traceable compute.
- CodeAgent writes its actions AS Python and runs them; the emitted code IS the provenance trace for any derived number (audit = re-run the code on the cited data → same number). Fits faithfulness.
- Model plugs via LiteLLM → OpenRouter → our locked open-weight models. NO closed lock-in. Sovereignty OK.
- REUSE our modules: wrap query-gen/search/fetch/weight-select as smolagents Tool subclasses (~15 LOC each); SKIP smolagents' own web tools.
- Sandbox MUST be Docker (self-hosted) for web-derived compute — the built-in LocalPythonExecutor is NOT a security boundary. Context-manager teardown each call (§8.4).
- A computed number re-enters the pipeline as a normal claim (inputs carry evidence_ids + the code) → strict_verify. Never bypass the faithfulness engine.
- Proven: Open Deep Research recipe = 55% pass@1 GAIA, #1 open submission at release.
- Optional stronger brain for hardest search steps: Tongyi-DeepResearch-30B-A3B (Apache-2.0, GAIA 70.9), a MODEL swap into the SAME smolagents harness — no architecture change.
Re-verify on our own DeepTRACE/DRB-II before locking (prove-our-scorer rule).
