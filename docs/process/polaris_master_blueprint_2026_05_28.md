# POLARIS MASTER BLUEPRINT — One Integrated Deep-Research Machine

## 1. PLAIN SUMMARY

POLARIS is one clinical-grade deep-research machine. A research question goes in, and a signed report comes out where every single claim has been checked against the exact source text. Inside, four AI models work in a line: a Generator writes the answer, a Mirror checks each claim is really supported, a Sentinel hunts for made-up statements, and a Judge gives every claim one of five honest grades. Around that line we wrap two more things: the power to pause and resume at any stage without paying twice, and a memory that snowballs, so the machine remembers what it already proved and can catch a new claim that contradicts something we verified weeks ago. The one rule above all others: we do a full dry run and confirm the whole machine is correct before we spend a single dollar on rented GPUs. Correct means a deep, line-by-line review of every output, every reasoning step, and every log, done independently by both Claude and Codex.

## 2. THE INTEGRATED MACHINE

Here is the whole flow in plain words, from question to signed report.

A question and a research protocol arrive. First come the front-door gates. The machine checks the question is in scope, checks there are enough good sources, and checks the source set was genuinely approved and not rubber-stamped. If any of those fail, the run stops right there, before the Generator spends anything. The moment those gates pass, we save a checkpoint. The checkpoint remembers the evidence pool and the approval decision, so if we resume later we never re-fetch and never re-pay for retrieval.

Next comes retrieval, where the machine gathers the live evidence. Right after that is the memory's first touch point. The machine asks its long-term knowledge graph: do we already hold any rock-solid, fully-confirmed prior claims about the things in this question? If yes, those priors are handed to the Generator as extra evidence rows. There is a hard safety rule here. A memory entry is fed in carrying its original source and its original span of text, never as "the memory says so." The memory re-injects the source, never the claim as if it were its own citation. And only the most trusted tier of memory is allowed to do this. I will come back to what that tier means. Important honest note: the memory is completely empty today, and the most-trusted tier is filled only by a later audit step, so for the first stretch of runs this touch point does nothing at all and the machine behaves exactly like the plain four-model pipeline. The memory ships switched off and lights up slowly as trusted claims accumulate. That graceful start is a design requirement, not an accident.

Then the Generator drafts the answer. It is DeepSeek V4 Pro, and today it is the only one of the four models that actually has a live endpoint. It writes the prose with a little provenance tag on every sentence pointing at the exact evidence span it used. We save another checkpoint here, holding the draft and its tags.

Now the first deterministic layer runs. This is plain Python, no AI. It checks each sentence: does the evidence id really exist in the pool, are the span boundaries valid, does every number in the sentence actually appear in the cited span, and do the sentence and the span share at least two real content words. Any sentence that fails is dropped. What survives is the set of claims the rest of the machine will scrutinize. This Python check runs before the three checker models, not after them. That ordering matters and I will flag why later.

Then comes the per-claim verification loop, the heart of the four-model design. For each surviving claim, in order: the memory's second touch point fires first. It runs a deterministic cross-time contradiction check. It builds a clinical-safe identity key from the claim's subject, predicate, dose, arm, timepoint, population, and unit, and it looks for a matching prior in the long-term graph. If it finds a match whose numbers diverge past a set threshold, it raises a flag. If the new claim and the old one are the same tier, it flags a genuine contradiction. If the new claim is from a stronger or newer source, it marks it as a supersession, meaning knowledge moved on, which is not a contradiction. This detection stays deterministic on purpose, because an AI treats fourteen percent and seventeen percent as roughly the same, and in clinical work they are not. The memory only flags. It never decides.

With that flag in hand, the claim goes to the Mirror, which is Cohere Command A Plus. The Mirror runs in two passes. One pass gives a grounded answer with citations. A separate pass, not in grounding mode, returns a clean calibration verdict. Two passes are needed because the citation mode and the structured-verdict mode cannot be combined on this model. Then the claim goes to the Sentinel, which is IBM Granite Guardian. The Sentinel answers a yes-or-no question about whether the claim is grounded. There is a lethal detail here: for this model, yes means ungrounded, a hallucination was detected, and no means grounded. Getting that polarity backwards would let fabrications walk straight through, so it is tested explicitly. Finally the claim reaches the Judge, which is Qwen. The Judge is the terminal decision-maker. It takes the Mirror's calibration, the Sentinel's flag, and the memory's cross-time flag, and it issues exactly one of five verdicts: verified, partial, unsupported, fabricated, or unreachable. The Judge is forced to pick one of those five and cannot wander off. After each claim, we append one line to a verdict ledger and bump a cursor. That is a checkpoint too.

When the Judge is done, one verdict drives three things at once, and this is the payoff of building it as one machine instead of three stapled-together parts. First, report release. A fabricated claim gets refused in place immediately and gates the run, because a fabrication happening at all is a red flag regardless of cleanup. An unsupported claim gets one rewrite-and-recheck, and if it still fails it is refused in place and the run is gated only if what remains is too thin. We never count or rate fabrications; a single one in a material claim is enough. Even a gated run still emits a report as a reviewable verdict artifact, never silence. Second, memory write eligibility. Verified and partial claims are eligible to be remembered; fabricated and unreachable claims never enter the memory. Third, the claim is queued for the final audit.

Then the single memory write happens. For each verified claim, the machine builds the identity key. A brand-new claim becomes a new node. A claim that matches an existing one with agreeing numbers is a re-confirmation; we do not duplicate it, we increment its confirmation count and link the runs. Crucially, the claim is written at the lower trust tier, called provisional, because the Judge alone is still just an automated check. If the memory store is unavailable, the write fails loudly. It does not quietly return nothing, which is the exact silent-degradation bug that left today's store empty.

Last comes the Codex line-by-line audit, the final human-equivalent gate and the single hinge of the whole machine. Both Claude and Codex independently audit every claim against the exact fetched source spans, reasoning step by reasoning step, citation by citation, using the proper clinical frameworks. This audit does triple duty. It signs off the report that exits. Its reconcile pass promotes matching verified claims from provisional up to the trusted durable tier, and only durable claims are ever fed back into the Generator. And, as you will hear in section five, this same audit is what authorizes the very first dollar of GPU spend.

Where does pause and resume sit? It is wrapped around the whole flow, written in POLARIS's own plain Python. It is deliberately not the LangGraph engine in the unused web pipeline, which is switched off and has never saved a thing, and it is deliberately not the Claude Code Workflow tool, which is a command-line feature we cannot run inside the product. There are two files per run. One holds the resume cursor: which stage we reached, the protocol fingerprint, the config fingerprint, the retrieval state, and two integer cursors, one for how many verdicts are written and one for how many claims reached the memory. The other is the verdict ledger. That single ledger file serves three masters at once: it is the resume log, it is the memory's write queue, and it is the audit's input. The two cursors are the crash-safe seam. If the machine dies between writing a verdict and writing it to memory, we lose neither and double-write neither. On resume, the machine re-checks the protocol and config fingerprints and refuses to continue if either drifted. It skips finished stages and replays claims from where the cursor left off.

Where does the memory plug in? It is a cross-cutting layer wrapped around the locked four models, never a new model inside the line. It reads twice and writes once: it reads trusted priors at Generator time, it reads for cross-time contradictions just before the Judge, and it writes after the Judge. Both reads are signals only. The Judge always decides. The lock that pins the four models pins models and roles, not the words we feed into a prompt, so handing the Generator a memory-derived evidence row is provably not a change to the locked architecture. In the lock file, the memory and the pause layer are declared under a brand-new top-level key that sits beside the four required roles, not inside them, so the freeze gate that counts the four roles never even sees it.

## 3. THE NO-SPEND DRY RUN — THE HEART OF IT

This is where the operator's one rule lives, so I will be careful and honest.

First, the honest truth about what zero-spend can and cannot mean. A literal full dry run of the whole machine for zero dollars on rented GPUs is impossible. Two of the four models, the Mirror and the Sentinel, are not available the way the Generator is; to get real output from them we must rent a GPU. So you cannot produce a real Mirror answer or a real Sentinel answer to audit without spending at least a little. We honor the rule, not by pretending a free full run exists, but by making the first spend a single, small, bounded run that is itself the dry run, fully audited by both Claude and Codex before any sustained spend follows.

So there is one no-money stage and then one tiny first rental. I will call the no-money stage Gate A, the dual audit of a green Gate A the money-authorization gate, and the first bounded rental plus its live probes Gate B.

Here is what Gate A runs, all for zero GPU dollars. The whole orchestration runs for real. The only thing faked is the network call to each model. We do this through one clean seam: an injectable per-role client. In the real product, that client talks to the real model endpoints. In the dry run, that same client replays recorded responses from fixture files, keyed to each claim. The dispatch code never knows it is being mocked; the mock is injected from outside. This is the single hard build rule, because if the fake lived any higher up, say faking "run the whole Mirror pass," the dry run would prove nothing about how the machine dispatches and flows. The fake must sit at the model-call boundary and nowhere higher.

Around those faked model calls, everything else is real and exercised end to end. The role dispatch and ordering are real. The per-claim Mirror-then-Sentinel-then-Judge flow is real. The deterministic Python sentence check is real. The release routing is real. The memory write of verified claims is real, on fixtures. The pause and resume is real, on fixtures. Retrieval runs as a couple of single cheap real probes, not a full sweep. And the Generator runs for real and cheap, because it is the one model with a live endpoint. No full-power sweep fires in the dry run; that is strictly after the money gate.

Now, what does green Gate A actually mean. It means every one of these checks passed. The unit and fixture test suites all pass. The lock-consistency checker exits clean, confirming the model slugs and roles in code match the lock. The offline preflight returns clean. The role-dispatch returns exactly four pins, generator, mirror, sentinel, and judge, where today it returns only two. The coverage gate accepts all four pins, tested against a fixture copy of the lock flipped to the locked state, because the real lock is not signed yet. The Sentinel polarity test confirms yes maps to ungrounded and no maps to grounded; an inverted mapping fails the gate. The Judge parser confirms it returns exactly one of the five verdicts. The Mirror two-pass parses, with the citation pass and the separate verdict pass both working. The release routing is correct: a fabricated claim gates the run on occurrence, an unsupported claim gets one rewrite then gates only on thin residual coverage. The completeness check confirms all four roles actually appear in the captured calls. The pause-and-resume integrity holds under crash injection: we kill the run at every checkpoint boundary and confirm the Generator never re-bills, verdicts replay from the cursor, nothing double-writes, and a drifted protocol fingerprint fails the resume closed. The memory write discipline holds: only verified and partial claims enter, fabricated and unreachable never do, the write is idempotent so re-running never double-counts, and the write fails loudly when the store is missing. The cross-time contradiction check fires correctly against a pre-loaded prior. And the three cheap real probes succeed: one search call, one Semantic Scholar call, one tiny Generator generation, each returning real output from a model whose served identity matches the pin.

What does a correct dry run produce for review? It produces one review dossier per question, in two synchronized forms so both auditors can work line by line: a machine-readable trace with one line per claim, and the same rows rendered as readable markdown. Each claim row carries the Generator sentence, its provenance tag, the Mirror's two passes, the Sentinel score and what its polarity means, the deterministic Python result, the Judge's five-way verdict and its reasoning trace, the release action and whether release was allowed, the memory decision and the reason, and one more field that matters for honesty: whether each fixture was a recorded real response or hand-built from the model's contract. Alongside that go the ordered log lines and the two audit ledgers, one from Claude and one from Codex, written into the existing ledger artifact, not a parallel one. Because the models are mocked and free, the dossier can include things too expensive to run live, like killing the run at every single checkpoint and like adversarial memory fixtures that try to poison the graph. That is why Gate A is genuinely strong, not a consolation prize.

And here is the honest boundary, the line the operator must hear travel with any green verdict. Gate A proves the machine is correctly wired and that its control flow and contracts are right. It does not prove the models are good. A canned Sentinel "yes" proves the pipeline routes an ungrounded flag to the right place. It does not prove the real Granite Guardian would actually say "yes" when shown a real fabrication. Gate A validates wiring and logic. Only the first real rental, Gate B, validates model behavior. That boundary is not negotiable, and it is the honest answer to "is the machine ready to spend on."

## 4. THE DEEP DUAL REVIEW

Before the dry run is ever called a success, both Claude and Codex run an independent line-by-line review, and then they reconcile. Because the three checker models are mocked, the fixtures are planted traps with known-correct answers. So a green dry run means the harness caught every trap, and the dual review is the two auditors confirming the harness's verdicts and routing match the known labels exactly.

The planted traps cover the cases that matter. A genuine fabrication with no supporting span: the harness must flag it ungrounded, judge it fabricated, refuse it in place, gate the run, and keep it out of memory. A claim whose numbers do not match its span: dropped by the Python check or judged unsupported, with a recorded gap. A clean verified claim: kept in the prose and written to memory only at the provisional tier, never straight to durable. A partial claim on a high-severity topic: one clarify attempt, then a visible advisory marker. A missing must-cover item, like an absent contraindication: the run is gated even though overall coverage looks fine. A citation that points at a source that does not exist: escalated to the fabricated gate. A claim that contradicts a seeded prior in memory: the contradiction is recorded. And a provisional, not-yet-audited memory claim: confirmed it can never feed back into grounding, which is the firewall.

For each trap, each auditor independently checks the same things. Is every cited span the exact fetched text, not a title or an abstract, and does the citation resolve in the evidence pool. Does each reasoning step actually follow from its cited evidence, with no leap and no qualitative-negation that slips past a number check. Is the Judge's verdict the right one given the Mirror and Sentinel signals, and does it match the trap's known label, with the Sentinel polarity asserted right there in the trace. Did release routing behave exactly as the trap expects, and was every drop a refuse-in-place with a recorded gap, never a silent deletion. Did only the right claims enter memory, at the provisional tier, with fabrications excluded, the write idempotent, and a simulated store failure raising loudly. Did pause and resume survive a mid-loop crash with the protocol and config fingerprints re-verified and failing closed on injected drift, with the two cursors proving no claim lost and none double-counted, and the final report byte-identical to an uninterrupted run. And are all the logs present and consistent across the resume cursor, the verdict ledger, and the run logs.

When the two audits disagree on a claim, we do not invent a new tiebreaker. We reuse the machine's existing conservative rule: take the worse verdict. A reconciled approval means zero unresolved trap mismatches across every trap, under both auditors. That reconciled, durable approval artifact is the only thing that authorizes asking the operator to load GPU credit. Not a green test run alone. Not Claude's word alone. The dual sign-off.

If the audit lags, the memory holds only provisional claims, the feedback read stays a no-op, and the snowball never grows. Is a visible quarantine of stale provisional claims enough, or do we need an alarm on a growing provisional backlog?

Third, the identity-key completeness. The key uses subject, predicate, dose, arm, timepoint, population, and unit. Are we missing clinical qualifiers that matter, like fasting versus fed, background therapy, or intention-to-treat versus per-protocol? A missing qualifier is a clinical-safety bug, a false merge or a false contradiction, not a tuning knob. Does this need a clinical expert's sign-off before the memory is built?

If the machine fails to recognize that tirzepatide, Mounjaro, and the lab code are the same drug, it silently splits one entity into two and the cross-time contradiction is never caught.
