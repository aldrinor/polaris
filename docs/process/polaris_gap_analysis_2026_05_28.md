# POLARIS Gap Analysis — Mighty Machine Vision vs. What's in the Code

## 1. Plain Summary (read aloud)

Here is the honest picture. The big four-role machine you want is real on paper and partly real in code, but it does not run yet. Right now POLARIS still runs the old two-model setup — one writer model and one checker model — not the four roles. The plans, the locks, and the guardrails for the four roles are all built and committed; the three new roles themselves have no working call sites yet. So we are mid-build, not done.

One honest line per dimension:

- The four-role pipeline (Generator, Mirror, Sentinel, Judge): **partly exists.** The Generator works. The other three are declared everywhere but called nowhere.
- Pausable and resumable from any stage: **partly exists, but the real pipeline cannot do it.** We can cancel a run. We cannot truly pause-and-resume the production pipeline.
- Snowball knowledge-graph memory across runs: **does not exist yet.** There is a pretty per-run picture in the web UI, but nothing accumulates verified claims across runs, and the one cross-run store that exists is empty and not a graph.
- The dev-process loop (your BOOT to BRIEF to gate to BUILD to SMOKE to gate to CLOSE to NEXT machine): **mostly exists as guardrails, but is not yet an executable machine.** The gates and the lock are committed and working. The loop itself is a document plus a nudge, not yet running code.

## 2. The Four Gaps

**Gap one — the four-role pipeline.** The Generator role is fully wired and real: `src/polaris_graph/generator/multi_section_generator.py` reads the Generator model setting and builds a live client. The other three roles — Mirror, Sentinel, and Judge — are declared as settings in one file, `src/polaris_graph/llm/openrouter_client.py`, and referenced nowhere else. No code ever builds a client for them. There is no calibration pass, no hallucination-detector pass, no final-arbiter pass. The architecture lock at `config/architecture/polaris_runtime_lock.yaml` names all four roles correctly, and a freeze guard in `scripts/dr_benchmark/pathB_run_gate.py` deliberately blocks the gated smoke test until the lock is signed — so today neither the old path nor the new path actually runs four roles. The gap is large. The one main thing to do: build a role-dispatch layer that constructs a client for each of Mirror, Sentinel, and Judge and adds their three call sites after generation. Important wrinkle the operator already decided: two of these models — Cohere Command A+ (Mirror) and Granite Guardian (Sentinel) — are not available on OpenRouter, so they must be self-hosted on rented Vast.ai GPUs, while Generator and Judge stay on OpenRouter. That means the dispatch layer needs two serving modes, and the whole four-role base is blocked on one operator action: the Vast.ai account balance is currently zero and needs credit loaded before Mirror and Sentinel can run anywhere. Also note: the Judge model name in the lock still carries a small typo, but that fix is already approved and in flight under issue I-meta-002 — it is not a fresh discovery.

**Gap two — pausable and resumable.** What works today is cancel, not pause. The production run store at `src/polaris_v6/queue/run_store.py` can move a run to cancelled, and the main sweep script honors a cancel request at stage boundaries. There is no paused state and no real resume in that path. Separately, a capable pause-resume state machine does exist in `src/polaris_graph/audit_ir/job_queue.py`, with pause, resume, and checkpoint methods, plus working pause and resume web endpoints. But here is the catch: the real production runner explicitly refuses to pause — it throws an error saying pause is not supported and tells you to cancel and re-run from scratch. And resume, even where it exists, just re-runs the job from step zero, because no runner ever reads back the saved checkpoint state. There is also a third, genuinely capable any-stage rewind engine in `src/polaris_graph/checkpoint_manager.py`, but it is switched off by default and its checkpoint database is an empty 3-byte file — it has never saved a single checkpoint, and it sits on a pipeline that the production sweep does not use. The gap is large. The one main thing to do: make the deployed pipeline save its full intermediate state at each stage boundary, then teach the runner to read that state and skip already-done stages on resume. This is exactly the open, unstarted GitHub issue #629.

**Gap three — the snowball knowledge graph.** This one is the furthest from real. What exists is a per-run picture in the web UI at `web/app/runs/[runId]/graph/page.tsx`, with an "expand snowball" button that walks the connections within a single run only — it is a viewer, not a memory. There is one cross-run store in `src/polaris_graph/memory/cross_vector.py`, but it is a flat list of text embeddings with no nodes, edges, entities, or relations, so it is not a knowledge graph at all — and it is empty, with zero claims accumulated, and it is wired only into a dormant pipeline the production sweep never touches. There is no store that saves verified claims across runs, and no code anywhere that checks a new run's claims against what was verified in past runs — so the cross-time contradiction catch you want simply does not exist. The gap is large. The one main thing to do: this is design-only for now under issue I-meta-003, and it is deliberately blocked until the four-role base passes a real dry run. So the honest move is to leave it as design and build it after the four roles work.

**Gap four — the dev-process loop.** This is the closest to done. The guardrails are committed and working: the architecture lock verifier `scripts/architecture/verify_lock.py`, the weekly drift report `scripts/architecture/weekly_drift_report.py`, the conformance CI gate, the Codex-required CI gate, the runtime lock file, and the freeze guard that ties smokes to the lock. A per-prompt reminder hook also runs and nudges the loop on every prompt. What is missing: the loop is not yet an executable machine. The workflow script the design points to — `.claude/workflows/polaris_task_cycle.js` — does not exist; the whole `.claude/workflows/` folder is absent. The binding rule it claims to live under, a CLAUDE.md section 3.0.1, does not exist either, so the loop is a proposal, not law yet. The reminder hook itself is untracked by git, so it would vanish on a clean checkout. And one more real hole the operator just found: the Codex-required gate is written but is not actually set as a required check on the main branch — branch protection lists no required checks — so the gate exists but is not enforced at merge. The gap is medium. The one main thing to do: write the executable workflow script and add the CLAUDE.md section 3.0.1 rule so the loop becomes binding, and fix the branch-protection setting so the gate is truly enforced.

## 3. The Bottom Line

Of the four, the dev-process loop is the closest to done — most of its parts are committed and working, and its gap is only medium. The snowball knowledge graph is the furthest — its core piece, a persistent claim graph, exists in no form at all, the stores are empty, and it is still design-only and blocked.

The single highest-value next build step is to finish the four-role base: build the role-dispatch layer and add the three missing call sites for Mirror, Sentinel, and Judge — but this is gated on one operator action first, loading Vast.ai credit, because two of the three new models must be self-hosted there and the balance is currently zero.

The honest sequence holds and is confirmed in the code's own issue tracking: four-role base first — we are mid-build on it now under I-meta-002 — then pausable and snowball-KG layered on top. The snowball-KG issue I-meta-003 is explicitly marked build-blocked until the four-role base passes a dry run, so this ordering is not a preference, it is already locked in.
