# Reproducible build & supply-chain provenance

Status: **S6 (review-readiness) — supply-chain hardening.**
Scope of this document: **build / dependency / documentation artifacts only.** No `src/`
runtime logic is involved in anything described here.

This document is the single source of truth for how to build POLARIS from a pinned,
hash-verified dependency set, how to (re)generate the supply-chain artifacts, and how to
prove two independent builds produce byte-identical results.

---

## 1. Pinned Python version

The runtime is pinned to a single interpreter patch version:

| Pin | File | Value |
|---|---|---|
| Interpreter floor/exact | `.python-version` | `3.11.10` |

`.python-version` is honoured by `pyenv` and by `uv` when it selects/provisions an
interpreter, so a fresh checkout **that uses one of those tools** resolves to the same
interpreter used to compile the lock and the SBOM.

**Important limitation:** `.python-version` is *not* self-enforcing. Plain
`python -m venv .venv` uses whichever `python` is on `PATH` — it does **not** read
`.python-version`. To actually pin 3.11.10 you must either drive the interpreter through
`pyenv`/`uv` (which read this file) or verify explicitly after creating the venv:

```bash
pyenv install -s 3.11.10 && pyenv local 3.11.10   # or: uv python install 3.11.10
python -m venv .venv && source .venv/bin/activate
python -c "import sys; assert sys.version_info[:3]==(3,11,10), sys.version"  # hard gate
```

The `assert` line is the enforcement; the file is only the declaration.

**Why `.python-version` and not `requires-python` in `pyproject.toml`:** `pyproject.toml`
in this repo is **tooling-only** (it carries `[tool.ruff]`, `[tool.mypy]`,
`[tool.interrogate]` and intentionally has **no `[project]` and no `[build-system]`
table**). The project is run from source; it is **not** a pip-installable/buildable
package. Adding a `[project]` table purely to hold `requires-python` would falsely signal
that the repo is a distributable wheel and would change tooling assumptions. The
interpreter contract is therefore expressed with `.python-version`, which matches how the
lock was originally compiled (`uv pip compile --python-version 3.11 ...`).

## 2. Dependency locking & cryptographic hashes

### 2.1 Files

| File | Pinned (`==`) | Cryptographic hashes | Role |
|---|---|---|---|
| `requirements.txt` | partial | no | Hand-maintained convenience set. **Not** for reproducible installs (self-described as non-minimal). |
| `requirements.lock` | yes (287 pkgs) | **no** | Original uv-compiled version lock. Kept for provenance/diffing. |
| `requirements.hashed.txt` | yes (287 pkgs) | **yes (5125 `--hash=sha256:` lines)** | **The reproducible-install source of truth.** |

`requirements.hashed.txt` was **generated, not hand-written.** It was produced from the
existing pinned `requirements.lock` with `--generate-hashes`, so every one of the 287
pinned versions is carried through unchanged and each is now bound to its full set of
PyPI artifact SHA-256 hashes. Version drift between `requirements.lock` and
`requirements.hashed.txt` is **zero** (verified by diffing the `name==version` pin sets).

### 2.2 Regenerating the hashed lock

The lock is regenerated with [`uv`](https://docs.astral.sh/uv/). `uv` is the tool that
produced the original lock (see the header comment in `requirements.lock`).

```bash
# From the repo root, with uv >= 0.11 available.
uv pip compile \
    --python-version 3.11 \
    --generate-hashes \
    requirements.lock \
    -o requirements.hashed.txt
```

Compiling **from `requirements.lock`** (rather than re-resolving from the original
loose inputs `req-stripped.txt` + `requirements-v6.txt`, one of which no longer exists in
the tree) guarantees the pinned versions cannot move — the step only *adds hashes* to an
already-frozen version set.

> Tooling note for this environment: `uv`, `pip-tools`, `cyclonedx-py`, and `pip-audit`
> are **not installed** in `/home/polaris/pipeline-env` (that env has pip 24.2 + Python
> 3.11.10). The artifacts committed here were generated by installing `uv` and
> `cyclonedx-bom` into a throwaway, isolated location (leaving `pipeline-env` untouched)
> and running the commands above/below. To reproduce on a clean machine:
>
> ```bash
> pip install 'uv>=0.11' 'cyclonedx-bom>=7'
> ```
>
> If `uv` is genuinely unavailable and only `pip-tools` is present, the equivalent is:
>
> ```bash
> pip-compile --generate-hashes --output-file requirements.hashed.txt requirements.lock
> ```
>
> Do **not** hand-edit hashes — they must always come from a resolver.

### 2.3 Reproducible install (hash-enforced)

```bash
# 1. Provision the pinned interpreter THROUGH pyenv/uv (they read .python-version;
#    plain `python -m venv` does not — see §1).
pyenv install -s 3.11.10 && pyenv local 3.11.10     # or: uv python install 3.11.10
python -m venv .venv && source .venv/bin/activate
python -c "import sys; assert sys.version_info[:3]==(3,11,10), sys.version"  # enforce

# 2. Install ONLY from the hashed lock, with hash verification MANDATORY.
pip install --require-hashes --no-deps -r requirements.hashed.txt
```

`--require-hashes` makes pip **refuse** any package whose downloaded artifact does not
match a hash in the file, and refuse the whole file if any requirement lacks a hash. This
is the property that turns a version pin into a tamper-evident supply-chain control.
`--no-deps` is safe because the hashed lock is already a fully-resolved, transitively
complete set.

## 3. Software Bill of Materials (SBOM)

| File | Format | Components | Reproducible |
|---|---|---|---|
| `sbom.cdx.json` | CycloneDX 1.5 (JSON) | 287 | yes — byte-identical across runs |

`sbom.cdx.json` was **generated** from `requirements.hashed.txt` (the pinned+hashed
source of truth), not from the possibly-drifted live environment. Every component carries
a `pkg:pypi/...` PackageURL and its artifact hashes.

### 3.1 Regenerating the SBOM

```bash
# From the repo root, with cyclonedx-bom (cyclonedx-py) installed.
python -m cyclonedx_py requirements \
    --output-reproducible \
    --sv 1.5 \
    --of JSON \
    -o sbom.cdx.json \
    requirements.hashed.txt
```

`--output-reproducible` strips the non-deterministic fields (random `serialNumber`,
wall-clock timestamps), which is what lets two runs be byte-identical — verified here:
regenerating `sbom.cdx.json` a second time produced a `diff`-clean, byte-identical file.

> Note: `uv export --format cyclonedx1.5` is **not applicable to this repo** — `uv export`
> operates on a project's `uv.lock`, which this repository does not carry. The
> `cyclonedx_py requirements` form above is the correct generator here because it binds the
> SBOM to exactly the committed `requirements.hashed.txt`.

### 3.2 Vulnerability audit (documented step)

`pip-audit` is not installed in `pipeline-env`. To audit the pinned set against advisory
databases:

```bash
pip install pip-audit
# Audit the pinned lock directly. --require-hashes keeps the hash gate on during audit.
pip-audit --require-hashes -r requirements.hashed.txt
```

`pip-audit`'s `--format cyclonedx-json` is an **output** format, not an input mode:
`pip-audit` cannot *consume* an SBOM. To scan `sbom.cdx.json` itself, feed it to an
SBOM-aware scanner (e.g. `grype sbom:sbom.cdx.json` or `osv-scanner --sbom sbom.cdx.json`)
rather than `pip-audit`.

This is a **documented manual step**, not an artifact committed by S6.

## 4. Two-build hash-match verification

The goal: prove the build is reproducible by building it **twice in independent clean
environments**, hashing the produced artifacts, and diffing. Any difference is a
non-determinism bug to be chased down.

This mirrors the **deterministic-oracle byte-diff** precedent already in the repo
(`tests/oracle/demo_artifact_diff.py`), which freezes the LLM I/O with a cassette so the
only thing that can change an output artifact is the code, then asserts a **byte-identical**
replay on a no-op and a **diff** on a real change. Here we apply the same
"byte-compare final observable outputs" discipline to *build* artifacts instead of
pipeline outputs.

### 4.1 Method

Pin the SBOM generator so the tool itself is not a hidden variable — a `>=` floor (as in
§2.2's convenience install) is **not** reproducible. The exact generator version is a
concrete, committed value in **`build_tools.pin`**: `cyclonedx-bom==7.3.0`. That version is
**not hand-guessed** — the committed `sbom.cdx.json` records its own generator under
`metadata.tools` as `cyclonedx-py 7.3.0` (with `cyclonedx-python-lib 11.11.0`), and the
pin file is derived from that. Read it back in both builds (fail-closed if missing):

```bash
# Concrete generator pin, committed alongside the artifacts it produced.
CYCLONEDX_PIN="$(grep -m1 '^cyclonedx-bom==' build_tools.pin)"; test -n "$CYCLONEDX_PIN"
# To regenerate build_tools.pin from a fresh install (must reproduce 7.3.0 for byte-match):
#   pip show cyclonedx-bom | awk '/^Version:/{print "cyclonedx-bom=="$2}' > build_tools.pin

# --- Build A: clean env #1 ---
git clean -xdf
pyenv local 3.11.10                                   # or: uv python install 3.11.10
python -m venv .venvA && source .venvA/bin/activate
python -c "import sys; assert sys.version_info[:3]==(3,11,10), sys.version"
# Capture the exact selected archive per package — URL + SHA-256 of the wheel/sdist pip
# actually downloaded. `pip install --report` writes this (pip freeze gives versions only,
# NOT which archive was chosen, so it cannot detect a wheel swap — do not rely on it here).
# `pip install --report FILE` installs AND writes the resolution report in one shot.
pip install --require-hashes --no-deps --report /tmp/reportA.json \
    -r requirements.hashed.txt
pip install "$CYCLONEDX_PIN"
# Normalize the report to (name, version, resolved-archive-url, archive-hash) per package
# for a stable diff:
python - <<'PY' > /tmp/selA.txt
import json
r=json.load(open('/tmp/reportA.json'))
for i in sorted(r['install'], key=lambda x:x['metadata']['name']):
    m=i['metadata']; d=i.get('download_info',{})
    h=d.get('archive_info',{}).get('hashes',{})
    print(m['name'], m['version'], d.get('url'), sorted(h.items()))
PY
python -m cyclonedx_py requirements --output-reproducible --sv 1.5 --of JSON \
    -o /tmp/sbomA.cdx.json requirements.hashed.txt
sha256sum requirements.hashed.txt /tmp/sbomA.cdx.json > /tmp/hashesA.txt
deactivate

# --- Build B: clean env #2 (fresh checkout / fresh venv, SAME OS/arch/Python) ---
git clean -xdf
pyenv local 3.11.10
python -m venv .venvB && source .venvB/bin/activate
python -c "import sys; assert sys.version_info[:3]==(3,11,10), sys.version"
pip install --require-hashes --no-deps --report /tmp/reportB.json \
    -r requirements.hashed.txt
pip install --require-hashes --no-deps -r requirements.hashed.txt
pip install "$CYCLONEDX_PIN"
python - <<'PY' > /tmp/selB.txt
import json
r=json.load(open('/tmp/reportB.json'))
for i in sorted(r['install'], key=lambda x:x['metadata']['name']):
    m=i['metadata']; d=i.get('download_info',{})
    h=d.get('archive_info',{}).get('hashes',{})
    print(m['name'], m['version'], d.get('url'), sorted(h.items()))
PY
python -m cyclonedx_py requirements --output-reproducible --sv 1.5 --of JSON \
    -o /tmp/sbomB.cdx.json requirements.hashed.txt
sha256sum requirements.hashed.txt /tmp/sbomB.cdx.json > /tmp/hashesB.txt
deactivate

# --- Compare: must be byte-identical ---
diff /tmp/selA.txt /tmp/selB.txt   && echo "SELECTED ARCHIVES: identical (same url+hash per pkg)"
diff /tmp/sbomA.cdx.json /tmp/sbomB.cdx.json && echo "SBOM: byte-identical"
diff <(cut -d' ' -f1 /tmp/hashesA.txt) <(cut -d' ' -f1 /tmp/hashesB.txt) \
    && echo "HASH-MATCH: two builds agree"
```

Precise claims (what each leg does and does **not** prove):

- **Hash gate, not full artifact identity.** `--require-hashes` guarantees every archive
  pip downloads matches one of the allow-listed SHA-256s, or the install fails. It does
  **not**, on its own, prove two builds *selected the same* archive: a pinned package can
  carry hashes for several platform wheels (e.g. `manylinux` vs `macosx` vs `win`), and
  pip legitimately picks the one matching the host. So two builds on **different OS/arch**
  can both pass `--require-hashes` while installing *different* (but each-allow-listed)
  wheels. The `pip install --report` diff above (`selA.txt` vs `selB.txt`) is what actually
  proves identical selection — it records the resolved **archive URL and SHA-256 pip chose
  per package**, which `pip freeze` (versions only) cannot. It only holds when Build A and
  Build B share **OS + CPU arch + Python version**; that environment triple is a required
  precondition of this proof, not a nice-to-have.
- **SBOM byte-identity is real but narrower.** `sbom.cdx.json` is generated *from the text
  of `requirements.hashed.txt`*, not from the installed environment, so its byte-identity
  across runs depends only on the input file + the pinned generator + `--output-reproducible`
  (which strips the random `serialNumber` and wall-clock timestamp). This was confirmed
  in-place for S6: regenerating the SBOM twice from the committed lock yielded a
  `diff`-clean, byte-identical file. It does **not** by itself attest the installed wheels.

## 5. What is automated vs a documented manual step (honesty ledger)

| Item | S6 status |
|---|---|
| `.python-version` pin (`3.11.10`) | **Committed artifact.** |
| `requirements.hashed.txt` (287 pkgs, 5125 real SHA-256 hashes, zero version drift) | **Generated & committed** (real, from `uv`). |
| `sbom.cdx.json` (CycloneDX 1.5, 287 components, reproducible) | **Generated & committed** (real, from `cyclonedx-py 7.3.0` / `cyclonedx-python-lib 11.11.0`, per its own `metadata.tools`). |
| `build_tools.pin` (concrete generator pin `cyclonedx-bom==7.3.0`) | **Committed artifact** — the exact generator version, verified against the SBOM's self-recorded `metadata.tools`. |
| Hashed-lock regeneration command | **Documented** (§2.2) — re-runnable. |
| SBOM regeneration command | **Documented** (§3.1) — re-runnable, SBOM byte-identity verified in-place. |
| `--require-hashes` install command | **Documented** (§2.3). |
| Two-build hash-match procedure | **Documented** (§4) — SBOM byte-identity verified in-place; the *selected-archive* identity leg uses `pip install --report` (exact wheel URL+hash per package, not `pip freeze`) under an explicit OS/arch/Python precondition, and is **not** executed across two machines in S6. |
| `pip-audit` vulnerability scan | **Documented manual step** (§3.2, `pip-audit --require-hashes -r ...`) — tool not installed in `pipeline-env`. |

`uv`, `cyclonedx-bom`, and `pip-audit` are **not** part of `pipeline-env`; the two
committed artifacts were produced by installing `uv` + `cyclonedx-bom` into an isolated
throwaway location so the runtime environment was never mutated. No hashes or SBOM
components were ever hand-written — everything committed came out of a resolver/generator.

## 6. Build-doc pointer fix

The README "Quick start" installs the hand-maintained, un-hashed `requirements.txt`. For
a **reproducible, supply-chain-verified** install, use the hashed lock instead:

```bash
pyenv local 3.11.10                                    # or: uv python install 3.11.10 (reads .python-version)
python -m venv .venv && source .venv/bin/activate
python -c "import sys; assert sys.version_info[:3]==(3,11,10), sys.version"
pip install --require-hashes --no-deps -r requirements.hashed.txt
```

`requirements.txt` remains as a developer convenience; `requirements.hashed.txt` is the
build-of-record.
