## 3. Coverage severity — FAIL

What is fixed correctly:

- `_required_entity_ledger_failed_under_strict` is lineage-independent and retains the exact HEAD
  boolean implementation, `bool(strict and ledger_forced_on and ledger_failed)`:
  `scripts/run_honest_sweep_r3.py:1934-1956`.
- Its F27 caller still converts a ledger implementation exception into a loud hold:
  `scripts/run_honest_sweep_r3.py:20891-20912`.
- The new change targets the real native coverage result generated by
  `release_policy.py:235-254`, at the outer four-role disposition:
  `scripts/run_honest_sweep_r3.py:19583-19628`.
- Coverage telemetry and gaps remain serialized at
  `scripts/run_honest_sweep_r3.py:19635-19654`; the separate required-entity fraction and evidence
  gaps remain at `:20512-20529`.
- Set equality protects fabrication, S0-must-cover, and pending-rewrite reasons from a
  coverage-only downgrade: `scripts/run_honest_sweep_r3.py:1912-1931`.

Blocking regression:

- The production outer condition at `scripts/run_honest_sweep_r3.py:19604-19623` checks
  `held_reasons`, the fabrication latch, lineage, and status, but never checks the already-computed
  `_release_outcome.hard_block` / `hard_block_reasons`, available from `:19543-19573`.
- Zero-grounding is represented outside `held_reasons` by `ReleaseOutcome.hard_block` and
  `hard_block_reasons=["zero_grounding"]`:
  `src/polaris_graph/roles/release_policy.py:539-568,583-608`.
- A decision with only
  `d8_unsupported_residual_below_coverage`, `fabricated_occurrence_latched=False`, and true
  zero-grounding therefore satisfies `_legacy_coverage_shortfall_report_only`. The new caller sets
  `manifest["release_allowed"]=True` and `summary_status="released_with_disclosed_gaps"` despite the
  hard block.
- The later A18 invariant does not repair this: an adjudicated release is accepted without checking
  `hard_block` (`src/polaris_graph/roles/release_policy.py:811-868`).

The read-only behavioral reproduction against the actual `compute_release_outcome` produced:

```text
ordinary_coverage_only: hard_block=False, predicate=True, outer would downgrade=True
zero_grounding: hard_block=True, hard_block_reasons=['zero_grounding'],
                predicate=True, outer would downgrade=True
```

There is also a scope leak: the disposition is gated by the global environment selector at
`scripts/run_honest_sweep_r3.py:19596-19606`, not the per-query legacy marker. A non-benchmark query
run with the selector set can receive the legacy coverage downgrade despite never being rebound or
stamped as a legacy task.

The advertised behavioral test is not behavioral. It calls only the pure reason-set predicate at
`tests/dr_benchmark/test_stage0_lineage_seam.py:377-405`; it does not exercise the outer disposition,
`ReleaseOutcome.hard_block`, manifest status, default-versus-legacy lineage, or preserved telemetry.
It therefore passes the zero-grounding release bug.

Required fix:

1. At `scripts/run_honest_sweep_r3.py:19604-19623`, gate the downgrade on the current query's actual
   legacy marker and on the absence of any `ReleaseOutcome` hard block/zero-grounding condition.
   Do not overwrite an insufficient-safety or other specialized terminal.
2. Add a real disposition-level behavioral test using an actual native below-threshold decision:
   default blocks, marked legacy downgrades coverage-only, telemetry/gaps remain exact, and
   fabrication/S0/pending-rewrite/zero-grounding/safety terminals never downgrade.

  `src/polaris_graph/generator/corpus_snapshot.py:134-137`.

Read-only harnesses confirmed both mismatch directions reject and both matching directions load.

The required caller-level tests are still absent. The tests define a local mirror
`_seam_expected_lineage = g.resolve_lineage(q_marker)` and then call `load_corpus_snapshot`
directly: `tests/dr_benchmark/test_stage0_lineage_seam.py:310-353`. They never execute
`run_one_query` or a production helper used by that caller. The prior false-confidence failure mode
therefore remains: the production caller could regress back to bare `q.get(...)` while these tests
stay green.

Required fix: exercise the production caller for legacy-stored→default-run and
default-stored→legacy-run, or extract the caller's effective-lineage resolution into one production
helper and have both `run_one_query` and the tests call that exact helper.
