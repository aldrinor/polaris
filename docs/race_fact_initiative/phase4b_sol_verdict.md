# Phase 4B Sol verdict — clean V30-targeted master plan

## Verdict

The discarded Phase-4 apparatus is not repairable by renaming or narrowing it. Runtime licensed-inference admission, an operator-license entailment predicate, premise-relation records, exact emitted-to-admitted audits over rewrite routes, and report suppression on mismatch are all members of the prohibited entailment/content-gating family. `MASTER_ACTION_PLAN.md` section 12 expressly voids that design and says to restart from the Phase-3 gaps and the valid V30 retarget map (`MASTER_ACTION_PLAN.md:660-698`). C2 is likewise VOID; its runtime canary must not be retained (`MASTER_ACTION_PLAN.md:184-209`).

The clean replacement is intentionally smaller:

1. establish the real V30-on, legacy-task-72 baseline through the Stage-0 lineage seam;
2. change only question decomposition, outline/focus text, and producer prompts before generation;
3. leave `provenance_generator` and `strict_verify` untouched;
4. use deterministic output inspections and paired RACE/FACT probes only as offline ship decisions;
5. drop a lever when the active verifier removes its effect, the score is flat, or the effect would require runtime content admission, editing, deletion, or suppression.

This plan retains twelve narrowed levers and drops U11 and U13. U1 is the first lever to build and ship, alone, after Stage-0 and re-baselining.

## A. EXECUTION_CHARTER — clean, binding

### A1. Architecture and lineage lock

The active producer is the V30 hybrid writer: contract-slot narrative plans are prepended, unmatched legacy-enrichment plans remain, contract plans dispatch through the narrative route, and other plans dispatch through the legacy route (`phase4_v30_retarget_verdict.md:45-64`; `multi_section_generator.py:11467-11489`). The narrative prompt is built by `slot_fill.build_slot_narrative_prompt`, receives `PG_NARRATIVE_PROSE_SYSTEM_MESSAGE`, and each generated sentence goes through the existing contract runner verification path (`phase4_v30_retarget_verdict.md:89-126`; `slot_fill.py:703-786`; `multi_section_generator.py:796-807`; `contract_section_runner.py:1909-1929,2121-2143`).

No lever experiment may begin until Stage-0 proves that the launched question, writer question, H1/title question, benchmark-packed question, and scorer question are the same canonical legacy task-72 prompt. Gate-B currently forces index 56, whereas the selected canonical question is the legacy task-72 registration; the forced override must be bypassed at the lineage seam (`phase4_v30_retarget_verdict.md:474-507`; `run_gate_b.py:5659-5687`; `run_honest_sweep_r3.py:7919-7934`; `score_report_race.py:44-68`).

The baseline is V30 enabled, task 72, full Gate-B slate, the same frozen corpus, generation model, judge, scorer, and configuration used by candidates. The prior `mf_baseline` score of 0.5009 is invalid because it was V30-off (`phase4_v30_retarget_verdict.md:443-468`). Baseline artifacts and their hashes are frozen before the first candidate.

### A2. No-ghost faithfulness rule

Every candidate must satisfy all of the following:

- No entailment, NLI, licensed-inference, premise-license, operator-license, or other faithfulness-ghost machinery.
- No generated-content edit, rewrite, deletion, withholding, report suppression, or “non-scoreable” result. The only post-generation exception is whitespace and heading-layout normalization that proves byte-for-byte identity of the ordered words, numbers, citation markers, and sentences.
- No runtime gate deciding whether a sentence, claim, relation, section, or report may pass. No before/after producer canary. No exact emitted-versus-admitted comparison.
- All substantive changes occur before generation in a question-derived scope query, outline/focus instruction, or active producer prompt.
- Existing `provenance_generator` and `strict_verify` behavior and code remain unchanged and are relied on as-is.
- Deterministic inspections run only in tests or evaluation tooling and decide whether the candidate commit ships. They are never imported or called from production generation.
- If a desired effect needs any prohibited machinery, or if the existing verifier removes it, the lever is dropped.
- A shared pre-generation note is allowed only when it contains ordinary question/evidence metadata such as comparison dimension, context, measure, period, or writer focus. It may not contain a candidate sentence, admissibility state, permission state, entailment relation, premise ID, claim license, binding identifier, or scoreability status.

These rules implement the rollback charter (`MASTER_ACTION_PLAN.md:660-698`) and preserve the Phase-3 requirement that faithfulness stay with the existing engine rather than a duplicate verifier (`PIPELINE_GAP_AUDIT.md:43-84`).

### A3. Mandatory reviewer ghost audit

Both independent reviewers run this on every candidate diff before considering scores:

```bash
git diff --unified=0 <approved-base>...<candidate> |
  rg '^\+.*\b(admission|canary|entailment|binding|permitted-operator|non-scoreable)\b'
```

Any added-line match is a blocking rejection, not a request to rename the construct. Reviewers also search added symbols and strings for `LicensedInference`, `OPERATOR_LICENSE`, `PremiseRelation`, `emitted.*admitted`, `admitted.*emitted`, and any production call into the new deterministic tests. Finally, `git diff --exit-code <approved-base>...<candidate> -- '**/provenance_generator.py' '**/strict_verify.py'` must be clean. A post-generation code change must be rejected unless it is U12’s layout-only normalizer and its preservation test proves that no word, number, marker, or sentence changed.

Sol and Fable must independently sign all four facts: the change reaches the active V30 or retained legacy producer, it is pre-generation-only, its test is offline-only, and no prohibited construct exists. Disagreement means no ship until Opus/operator adjudication.

### A4. Generalization gate

Production changes may contain no task/domain literals, magic counts, benchmark-specific entity lists, or adjective/version flag names. Fixed role/type/status vocabularies may be general, but benchmark answers may not be encoded. Test fixtures may hold exact expected data. Held-out tasks 91, 100, and 73 are mandatory; task 91 must prove no-fire for analytical-only behaviors where its question does not request them (`MASTER_ACTION_PLAN.md:136-182`; `PIPELINE_GAP_AUDIT.md:800-820`).

Activation must follow ordinary question structure and evidence metadata, not a new runtime experiment flag. Candidate-versus-baseline measurement uses separate commits/config snapshots rather than a hidden production branch.

### A5. Measurement gate

For every lever:

1. Run its deterministic test through the real producer seam and inspect the final emitted report. A helper/packer/unit result without visible report behavior does not pass (`PIPELINE_GAP_AUDIT.md:19-41`).
2. Run at least three baseline and three candidate replicates against the frozen V30-on/task-72 setup. Pair replicate seeds/order and retain all report, RACE, FACT, log, and configuration artifacts (`MASTER_ACTION_PLAN.md:227-257`; `phase4_v30_retarget_verdict.md:587-602`).
3. Predeclare the targeted dimension. Its paired-mean gain must exceed +0.027. A mean change within ±0.027 is inconclusive; a negative mean inside the band is not a pass. One prespecified confirmatory batch is allowed.
4. Any RACE dimension at or below -0.027 is a no-go. FACT precision/citation accuracy is non-inferior for every wave, and U14 additionally targets a positive effective supported-pair change.
5. Measure each accepted wave cumulatively against the last approved state. A flat lever may remain only as an explicitly necessary dependency whose downstream behavioral test has already passed; otherwise drop it.

The ±0.027 boundary comes from the benchmark replication limit (`SCORING_SPEC.md:231-245`). Insight is the main U1/U2/U3/U4/U6/U7/U10 target; Comprehensiveness is the main U5/U8/U9 target; Readability is the main U12 target; FACT remains an independent supported statement–URL-pair surface (`phase1_sol_verdict.md:436-462`; `phase1_fable_verdict.md:431-472`).

### A6. No-regression and stop rule

A lever ships only when both reviewers approve the clean architecture, its deterministic test passes at the emitted-report seam, its targeted paired mean clears +0.027, no RACE dimension regresses by 0.027 or more, and FACT is non-inferior. Otherwise revert the candidate, record the failure, and either make one clean prompt/scope correction or drop it. No apparatus may be added to rescue a failing lever.
