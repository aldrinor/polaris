# Codex V30 sweep integration audit

**Verdict**: CONDITIONAL-blockers

## Answers
1. `PG_V30_ENABLED=0` is functionally safe once `run_v30_post_generation()` is reached: it returns immediately with `enabled=False` before the inner chain does any V30 work, and the runner only mutates the manifest inside `if v30_result.enabled` (`src/polaris_graph/v30_sweep_integration.py:117`, `scripts/run_honest_sweep_r3.py:1661`). Caveat: the runner still imports and calls V30 unconditionally, so an import-time failure would still stamp `manifest["v30_error"]` even when disabled (`scripts/run_honest_sweep_r3.py:1650`, `scripts/run_honest_sweep_r3.py:1701`). Good enough for this commit, but not perfectly hermetic gating.
2. Exception containment is strong enough to keep the sweep alive. Inner-chain failures are converted into `V30SweepResult.error` (`src/polaris_graph/v30_sweep_integration.py:127`), and the runner has a second broad guard around the integration block (`scripts/run_honest_sweep_r3.py:1650`, `scripts/run_honest_sweep_r3.py:1701`). I do not see a path where V30 overwrites existing manifest keys other than adding `v30_*` and optional `frame_coverage_report`; the bigger issue is semantic overclaiming, not structural corruption.
3. I do not see a race with the existing post-processing path. `report.md` is written first (`scripts/run_honest_sweep_r3.py:1407`), V30 then runs synchronously before the final manifest write (`scripts/run_honest_sweep_r3.py:1637`, `scripts/run_honest_sweep_r3.py:1710`), and the sweep loop itself is serial across queries (`scripts/run_honest_sweep_r3.py:1815`). Within this file, V30 is ordered cleanly after generation and before manifest persistence.
4. The current non-migrated-slug shape is safe but under-specified. `compile_frame() -> None` returns `enabled=True`, `frame_coverage_report=None`, `error=None` (`src/polaris_graph/v30_sweep_integration.py:199`), so the manifest only gets `v30_enabled: true` (`scripts/run_honest_sweep_r3.py:1661`). That will not break the sweep, but `v30_skipped_reason: "no_contract"` would make manifests easier to interpret in live runs.
5. Yes, the phase-1 validation synth overstates coverage. For every non-gap row it fabricates `PASS`, `observed_completion_count=1`, and `bound_ev_id_present_in_prose=True` without checking the actual verified report output (`src/polaris_graph/v30_sweep_integration.py:448`, `src/polaris_graph/v30_sweep_integration.py:458`, `src/polaris_graph/v30_sweep_integration.py:459`). The tests lock this behavior in by expecting all stubbed non-gap rows to count as passes (`tests/polaris_graph/test_v30_sweep_integration.py:271`, `tests/polaris_graph/test_v30_sweep_integration.py:523`). If Phase 1 wants to claim `PASS`, it should cross-check the legacy output at least at the citation/prose-presence level; otherwise the field names or verdict semantics should be downgraded to retrieval-only/provisional.
6. The completions workflow is implemented as current-contract validation, not as a strict replay of `run1` task artifacts. `tasks_equiv` is rebuilt from the current run’s contracted entities on every rerun (`src/polaris_graph/v30_sweep_integration.py:354`), so this works for “rerun same slug, same contract,” but it is looser than “only the entities listed in the previous `human_gap_tasks.json`.” That means stale completions can continue to be accepted for any still-contracted entity, not just current gaps.
7. Using the contract DOI is the right choice. If M-56 retrieval fails and the row DOI is `None`, validating against the row DOI would reject the exact human completion that is supposed to repair the gap; the contract DOI is the stable identity anchor (`src/polaris_graph/v30_sweep_integration.py:357`). The acceptance/rejection tests are aligned with that contract-DOI behavior (`tests/polaris_graph/test_v30_sweep_integration.py:351`, `tests/polaris_graph/test_v30_sweep_integration.py:416`).
8. I do not see a new JSON-serialization risk in these three files. The integration only adds plain dict/list/string values to the manifest (`src/polaris_graph/v30_sweep_integration.py:288`, `scripts/run_honest_sweep_r3.py:1663`, `scripts/run_honest_sweep_r3.py:1667`), and the final manifest write is a plain `json.dumps(manifest)` (`scripts/run_honest_sweep_r3.py:1710`). Within this scope, serialization looks clean.
9. The described `report.md` boundary is not what the code does. The runner checks `report_path.exists()` only for the read, then unconditionally writes `existing + disclosure_block`, so a missing `report.md` would produce a disclosure-only file (`scripts/run_honest_sweep_r3.py:1677`, `scripts/run_honest_sweep_r3.py:1688`). In the current success-path ordering that file should already exist (`scripts/run_honest_sweep_r3.py:1407`), so this is a low-probability path, but the implementation is looser than intended.
10. In the audited files, the only explicit new runtime dependency is `PG_UNPAYWALL_EMAIL` for Unpaywall (`src/polaris_graph/v30_sweep_integration.py:52`). I do not see any additional V30-specific API key or env requirement here. The remaining dependency risk is whatever M-56 already requires from CrossRef/PubMed/Unpaywall availability and latency.
11. The isolation tests are the right boundary for most of `run_v30_post_generation()`, but they do not cover the runner edit itself. The test file explicitly scopes itself to the integration module (`tests/polaris_graph/test_v30_sweep_integration.py:3`), so there is no smoke test asserting manifest merge behavior, `v30_error` fallback, or the `report.md` append path. A small runner-level unit test with `run_v30_post_generation()` monkeypatched to a canned result would be worthwhile and network-free.
12. For a one-off sequential live exercise, the incremental cost profile looks acceptable. This layer adds HTTP fetches and file writes, not LLM spend, and I do not see any obvious rate-limit blocker in the serial sweep loop (`scripts/run_honest_sweep_r3.py:1815`). The launch risk I would fix before a live sweep is the truthfulness of `frame_coverage_report`, not the raw runtime overhead.

## Findings
- **Blocker** `src/polaris_graph/v30_sweep_integration.py:448`, `src/polaris_graph/v30_sweep_integration.py:458`, `src/polaris_graph/v30_sweep_integration.py:459`, `tests/polaris_graph/test_v30_sweep_integration.py:271` — Phase-1 validation marks every non-gap row as `PASS` and as present in prose without checking the actual legacy generator output. Live manifests can therefore overstate coverage and hide entities that were omitted, filtered as off-topic, or dropped by verification.
- **Medium** `scripts/run_honest_sweep_r3.py:1650`, `scripts/run_honest_sweep_r3.py:1701` — Opt-in gating is not fully hermetic because the runner imports/calls V30 even when disabled. A future import-time failure would still alter disabled-run manifests via `v30_error`.
- **Medium** `scripts/run_honest_sweep_r3.py:1677`, `scripts/run_honest_sweep_r3.py:1688` — The `report.md` append path will create a new file if the original report is missing, despite the intended “append only on success-path report” boundary. Low-probability in current flow, but the implementation does not enforce the stated boundary.
- **Medium** `tests/polaris_graph/test_v30_sweep_integration.py:3`, `scripts/run_honest_sweep_r3.py:1661` — The runner integration itself is untested. A unit smoke test should verify disabled gating, manifest merge, `v30_error` fallback, and disclosure append behavior without hitting network.
- **Nit** `src/polaris_graph/v30_sweep_integration.py:199`, `scripts/run_honest_sweep_r3.py:1661` — Non-migrated slugs produce `v30_enabled: true` with no explicit skip reason. That is safe, but `v30_skipped_reason: "no_contract"` would materially improve live-run observability.
- **Nit** `src/polaris_graph/v30_sweep_integration.py:229`, `src/polaris_graph/v30_sweep_integration.py:334` — Comment says completions are loaded from `run_dir` or the parent domain directory, but the implementation only checks `run_dir`.

## Next
Before a live sweep, fix or re-scope the phase-1 coverage semantics: either cross-check against the actual verified legacy output before claiming `PASS`, or rename/downgrade the manifest fields so they explicitly mean retrieval coverage rather than report coverage.

After that, add one runner-level smoke test for the `run_honest_sweep_r3.py` hook. With those changes, the integration is ready for live-run exercise.
