**Root Cause**

The structural bug was suspect **(a), with a nuance**: REDUCE’s full `[#ev:evidence_id:start-end]` tokens were not mangled. They survived unchanged and therefore bypassed the legacy sentence-aware span binder.

Legacy path asks the model for `[ev_XXX]` markers, then `_rewrite_draft_with_spans` computes a support span for the final generated sentence: [multi_section_generator.py](C:/POLARIS/src/polaris_graph/generator/multi_section_generator.py:1309), [live_deepseek_generator.py](C:/POLARIS/src/polaris_graph/generator/live_deepseek_generator.py:349), [live_deepseek_generator.py](C:/POLARIS/src/polaris_graph/generator/live_deepseek_generator.py:373).

Distill REDUCE was using pre-bound MAP finding spans instead. Those spans are offsets for the atomic `support_quote`, not for the reducer’s final sentence. strict_verify then checks the final sentence against exactly `direct_quote[tok.start:tok.end]`, including number match, content overlap, and entailment: [provenance_generator.py](C:/POLARIS/src/polaris_graph/generator/provenance_generator.py:1634), [provenance_generator.py](C:/POLARIS/src/polaris_graph/generator/provenance_generator.py:1646), [provenance_generator.py](C:/POLARIS/src/polaris_graph/generator/provenance_generator.py:1706), [provenance_generator.py](C:/POLARIS/src/polaris_graph/generator/provenance_generator.py:1748), [provenance_generator.py](C:/POLARIS/src/polaris_graph/generator/provenance_generator.py:1859).

So the reducer was structurally denied the legacy recovery mechanism that makes the same model/evidence pass about half the time.

**Fix Applied**

I changed REDUCE to produce legacy `[ev_XXX]` markers, not full span tokens, and made the filter normalize stale full tokens back to `[ev_XXX]` before `_rewrite_draft_with_spans`.

Key changes:

- REDUCE prompt now requires `[[finding:f...]]` plus `[ev_XXX]`, and explicitly forbids direct `[#ev:...]` span tokens: [evidence_distiller.py](C:/POLARIS/src/polaris_graph/generator/evidence_distiller.py:183).
- Ledger renders `cite=[ev_001]` instead of `span=...`: [evidence_distiller.py](C:/POLARIS/src/polaris_graph/generator/evidence_distiller.py:251).
- Filter accepts evidence markers and rewrites stale `[#ev:ev_001:10-51]` to `[ev_001]`: [evidence_distiller.py](C:/POLARIS/src/polaris_graph/generator/evidence_distiller.py:1017), [evidence_distiller.py](C:/POLARIS/src/polaris_graph/generator/evidence_distiller.py:1071), [evidence_distiller.py](C:/POLARIS/src/polaris_graph/generator/evidence_distiller.py:1081).
- Distiller cache version bumped to avoid replaying old full-token reducer cache: [evidence_distiller.py](C:/POLARIS/src/polaris_graph/generator/evidence_distiller.py:68).
- `_run_section` remains the same behavioral pipeline: filter, then `_rewrite_draft_with_spans`, then strict_verify: [multi_section_generator.py](C:/POLARIS/src/polaris_graph/generator/multi_section_generator.py:2395).

- **(b) Partial symptom**: REDUCE paraphrase becomes fatal only because the span is pre-bound to the MAP quote.

The architecture is compatible with strict_verify only if REDUCE emits evidence IDs and the deterministic postprocessor binds spans after final prose exists. It is not compatible with trusting reducer-supplied or MAP-supplied final span offsets.

5. IS MAP-REDUCE FUNDAMENTALLY INCOMPATIBLE WITH strict_verify?
================================================================================
No. strict_verify's exact-span matching is compatible with map-reduce. What is
incompatible is having the LLM emit FINAL machine tokens ([#ev:id:start-end]) and
synthetic finding-ids. The compatible architecture is map-reduce that emits SHORT
citation markers and DEFERS span computation to the deterministic finder — i.e.
the proven legacy contract, applied to the validated-findings ledger instead of
raw quote blocks.
