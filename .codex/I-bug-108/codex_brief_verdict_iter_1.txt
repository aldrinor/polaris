## Architecture

### Where the loop runs

`src/polaris_graph/generator/multi_section_generator.py` per-section drafting flow:

```
section_draft  → strict_verify  → kept[], dropped[]
                                   ↓
                                   I-bug-108 NEW: repair_loop(dropped, evidence_pool)
                                   ├─ For each dropped sentence (up to MAX_REPAIR_PER_SECTION):
                                   │   ├─ build repair prompt (sentence + cited spans + failure reason)
                                   │   ├─ single LLM call to generator (DeepSeek V3.2-Exp)
                                   │   ├─ verify_sentence_provenance on the repaired output
                                   │   └─ if verified → kept; if still dropped → record original drop
                                   └─ return augmented kept[]
                  ↓
                  Section result with the larger kept[]
```

The loop runs AFTER strict_verify, NOT inside it. This keeps the verifier pure and the repair concern isolated. Each repair attempt is exactly one LLM call.

### What the repair prompt looks like

```
ORIGINAL SENTENCE (was dropped by strict_verify):
<the dropped sentence with [#ev:...] tokens>

CITED SPAN(S) FROM THE EVIDENCE:
<the actual text the [#ev:...] tokens point to>

VERIFIER REASON FOR DROPPING:
<the failure_reasons list, e.g. "entailment_failed: NEUTRAL — sentence
 introduces specific patient characteristics not in cited span">

TASK: Rewrite the sentence so the cited span actually entails it.
- Keep the [#ev:...] markers EXACTLY as they appear (same evidence_id, same start-end)
- Remove the unsupported specifics the verifier flagged
- If you can't make the claim entailed, output exactly NULL_DROP and we'll keep the original drop.
- Do NOT introduce numbers, study names, jurisdictions, or specific
  outcomes that aren't in the cited span.

OUTPUT: just the rewritten sentence (one line) with [#ev:...] markers preserved. No preamble.
```

### Bounds (cost discipline)

- `PG_REPAIR_LOOP_MAX_RETRIES_PER_SENTENCE = 1` env-gated, default 1 (single retry — Codex can recommend higher if you want)
- `PG_REPAIR_LOOP_MAX_PER_SECTION = 10` env-gated, default 10 (cap per section to bound runaway cost)
- `PG_REPAIR_LOOP_ENABLED = "true"` env-gated, default on. Operators can disable.
- Cost per repair: ~$0.002 per call. With 30 dropped × 1 retry = ~$0.06/run incremental. Within Carney delivery budget.

### Failure modes the loop should NOT trigger on

- `no_provenance_token`: sentence has no [#ev:...] at all. The repair prompt would have nothing to anchor on. SKIP.
- `invalid_token` / `span_out_of_range`: token references unknown evidence or bad bounds. The generator likely hallucinated the token. Repair won't help. SKIP.

The loop should only fire on the "drift" failures where the sentence has a valid cited span but the prose drifted from it:
- `entailment_failed` (the prime case)
- `number_not_in_any_cited_span`
- `no_integer_overlap_any_cited_span`
- `trial_name_mismatch`
- `overlap_too_low`
- `no_content_word_overlap_any_cited_span`

### Telemetry

Add to manifest:
- `repair_attempts: int`
- `repair_successes: int` (verified after repair)
- `repair_failures: int` (still dropped after repair OR NULL_DROP returned)
- `repair_cost_usd: float`
- `recovery_rate: float` (successes / attempts)

## What I want from you

1. **Verdict** APPROVE / REQUEST_CHANGES on the design.
2. **MAX_RETRIES per sentence**: 1, 2, or 3? My read: 1 (cost discipline; if first repair didn't work, the sentence is probably structurally drifted). Cost scales linearly so 2 doubles the budget. Recommend?
3. **MAX_PER_SECTION**: 10 is my proposal. If a section drops 30, we'd repair 10 of them per section (~5 sections = 50 repair calls = ~$0.10 per run). Is 10 right or should it be the section's actual drop count without cap?
4. **Should the repair pass also re-run the entailment judge after the verifier?** verify_sentence_provenance handles the mechanical checks, but the entailment judge is a downstream call. My read: yes, repaired sentences should re-pass through the FULL strict_verify chain including entailment, otherwise we could re-introduce drift.
5. **Acceptance criterion for the metric "30% recovery rate"** — too low? too high? Should we set 50% per your iter-1 framing?
