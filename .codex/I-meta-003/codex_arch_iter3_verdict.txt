## 1. END-TO-END PIPELINE (full-power stage sequence)

**Stage 1 — Scope & decomposition (PICO faceting).** Detects domain (clinical / regulatory / policy), parses the question into Population / Intervention / Comparator / Outcome plus the safety facets clinical RAG requires — contraindications, adverse-event signals, special populations, regulatory-label divergence, comparator head-to-heads — and sets a per-domain `relevance_tau` floor. **Model/tool:** existing scope gate (`src/polaris_graph/nodes/`, §9.2) + Writer for PICO + synonym/MeSH-candidate proposal, validated against the keyless NCBI E-utilities MeSH database so no hallucinated MeSH ID reaches a query. **Fixes vs shipped:** `query_decomposer.py` is pure string-splitting — no LLM, no MeSH, no PICO, no synonyms, hard cap 6 sub-queries (`query_decomposer.py:33-36,108-148`); it cannot expand "tirzepatide"→"Mounjaro" or map "T2DM"→MeSH `Diabetes Mellitus, Type 2`. Full-power replaces flat keyword fan-out with semantic PICO faceting (~20-40 backend-native queries) feeding the existing deterministic expanders (`regulatory_expander.py`, `primary_trial_expander.py`).

**Stage 2 — Research plan (explicit, editable, enforced).** Compiles the PICO plan into an ordered set of report sections with, per section, an **evidence target** (minimum tier mix + minimum distinct findings) and a machine-checkable **definition of done** (each gold-rubric coverage axis backed by ≥1 verified finding, else the section is flagged INCOMPLETE — never silently thin). Rendered in the built editable plan-review UI (tasks #459/#754) so the operator edits scope before any generator token is billed. **Model/tool:** Writer drafts from the scope template (`nodes/scope_gate.load_scope_template`) + `section_blueprint.py`; targets written to the run manifest as a checkable `plan` block. **Fixes vs shipped:** sections today come from a static template with no per-section evidence target and no done-definition; a section that retrieves nothing still renders prose at ~30% verified rate. Full-power makes "done" an enforced per-section contract.

**Stage 3 — Multi-round agentic search (STORM-style, saturation-stopped).** Round 1 fires the full PICO query plan across all backends. Between rounds a **gap-analysis** compares retrieved findings to the per-section targets and PICO facets and emits **targeted follow-up queries for gaps only**. Stop rule is **saturation, not a fixed count**: continue until the new-unique-finding rate drops below ~1 per 20 newly-fetched sources for two consecutive batches (`corpus_relevance_driven_logic.md §4.6`), OR all section targets are met, OR the budget cap binds — and report where it saturated. **Model/tool:** Writer runs gap-analysis + follow-up query generation; backends = Serper + Semantic Scholar + Europe PMC (existing, `domain_backends.py:331`) + **new first-class PubMed E-utilities (esearch/efetch, keyless)** + **new first-class ClinicalTrials.gov v2 API**. **Fixes vs shipped:** today is effectively single-pass + one bolt-on R6 expansion capped at 4 queries (`run_honest_sweep_r3.py:1885-1891`) with no gap-analysis and no saturation rule; PubMed is treated as "already well-served" via generic Serper (`domain_backends.py:7`) and ClinicalTrials.gov is **skipped** (`:452`). Full-power adds the two clinical-channel backends and a true gap-driven loop.

**Stage 4 — Relevance + tier-driven fetch (no count cap; `corpus_funnel`).** Fetches every candidate that clears all three bars and drops the rest **with a counted reason**: (1) tier bar (T1-T4 admissible; T5/T6 only with a regulatory-content marker; hard-drop T7 stubs/predatory via `tier_classifier.py`); (2) relevance bar (`_row_relevance ≥ tau`, tau from the score-distribution knee, floored by `PG_RELEVANCE_TAU_FLOOR`, adaptive — never constant); (3) dedup-by-finding (collapse same-trial/DOI-family sources to one FINDING with N provenance pointers). Emits the **binding `corpus_funnel` invariant** to the manifest: `discovered / dropped_tier / dropped_relevance / dropped_dedup / kept=N / tau`. **Model/tool:** `live_retriever` fetch loop with the `chosen[:fetch_cap]` truncation (`live_retriever.py:1573-1591`) replaced by a relevance/tier floor; `evidence_selector.select_evidence_for_generation` (`evidence_selector.py:893,:976,:1500`) with `max_rows` truncation removed; no new model — selection logic only. **Fixes vs shipped:** this is the central loophole — the pipeline DISCOVERS ~1,000 candidates but DUMPS ~96% at two fixed-COUNT cuts (FETCH_CAP=40 then MAX_EV_TO_GEN=20) by POSITION not quality, and the dump is SILENT (drops go to a best-effort Path-B trace, not a counted manifest funnel) (`corpus_relevance_driven_logic.md:37`). Keeping ALL 1,000 would hurt (lost-in-the-middle); the goal is the **covering set** (~120-200, an unmeasured prior to MEASURE), reported by count.

**Stage 5 — Relevance corpus → per-source MAP digest.** One independent DeepSeek digest call per kept source → a structured claim record (≈2,500 tok in / 500 out), embarrassingly parallel. **Model/tool:** new `evidence_reducer` (`corpus_relevance_driven_logic.md §4.2`) + Writer. **Fixes vs shipped:** replaces the cram-then-truncate single-context approach; unbinds N from the 1M window and avoids lost-in-the-middle degradation (Liu 2307.03172, Levy 2503.04388). Each source is read in isolation, so no source is diluted by neighbours.

**Stage 6 — Dedup-to-FINDINGS + hierarchical REDUCE synthesis.** Clusters digests into unique findings-with-N-provenance, then synthesizes section-by-section **over the digests** (≈500 tok each), not over raw source text. **Model/tool:** `multi_section_generator` re-pointed at findings (`corpus_relevance_driven_logic.md §4.3-4.4`) + Writer. **Fixes vs shipped:** lifts the ~30% verified-prose ceiling — the 30% is a SUPPLY problem (only 20 evidence rows reach the writer, `:2266`), so the writer runs out of grounded material mid-report; more verified supply → higher verified fraction at the same coverage. **Stage 8 — 4-role verification (per-claim terminal verdict).** Per atomized claim: **atomize** (Writer splits a sentence with two numbers into two claim-records sharing one span — finer than shipped sentence-level strict_verify) → **Mirror** 2-pass calibration (re-read claim blind to Writer confidence; flag over-claim, emit `calibration_delta`) → **Sentinel** binary RAG-faithfulness gate (grounded y/n; `UNGROUNDED` is **fail-closed** — cannot proceed regardless of Judge) → **Sentinel** qualitative-negation pass (`(did not|no|never|absent|without)+endpoint` must be asserted in the span) → **Judge** terminal 5-enum **VERIFIED / PARTIAL / UNSUPPORTED / FABRICATED / UNREACHABLE** (may downgrade, may NOT upgrade a Sentinel-UNGROUNDED claim) → **python strict_verify** deterministic floor (evidence-id in pool, span bounds valid, every decimal appears verbatim in the span, ≥2 content-word overlap; any failure → DROP, no model override) → **guarded citation-repair** only for span-locality failures (right source, wrong offset): Writer proposes a narrower span from the SAME source, re-judged against the short span; still failing → fail-closed DROP; repair can never invent a source or widen scope. **Model/tool:** `role_pipeline.py` Mirror/Sentinel/Judge (`corpus_relevance_driven_logic.md §4.5`); `generator/strict_verify` (§9.1.3) + M-45 refetch; `openrouter_client.check_family_segregation` (§9.1.1). **Fixes vs shipped:** wires the built-but-not-default gate into the default CLI/UI/queue path (`:3940`); adds atomization, the 5-enum, the negation pass, and the guarded repair (shipped has none — a correct claim with a slightly-off offset is dropped outright, costing coverage). The guarded repair is the corrected design from `project_complete_solution_codex_corrections_2026_05_30` (whole-doc widening would smuggle ungrounded claims; candidate-generator-only + re-judge short span + fail-closed is the safe form).

## 6. PLAIN-SPOKEN SUMMARY (for a blind operator)

1. Today POLARIS searches once, keeps only 40 sources, and only about a third of its sentences are actually backed by a citation — that loses on coverage to the frontier tools.
2. Full-power searches in several rounds until it stops finding anything new, keeps every relevant source instead of the first 40, and reads each source on its own so none gets drowned out.
3. It writes evidence-first: it only writes a sentence if a real source already supports it, and it tags every sentence with the exact span it came from.
4. Then four different AI models check the work — one for over-confidence, one purpose-built hallucination catcher, one final judge, plus a strict math check that every number appears word-for-word in its source; anything that fails is dropped or refused, not smoothed over.
5. The frontier tools never refuse and never let you click a number to see its proof — Gemini's compute report had 4,887 words and zero citations; ChatGPT's footnotes point to nothing you can check. POLARIS makes every number one click from its source and shows you what it threw out.
6. We win on trustworthy, checkable answers — not on length or memory, which we deliberately don't chase; nothing is proven until one real blinded run on the five clinical questions scores it.
