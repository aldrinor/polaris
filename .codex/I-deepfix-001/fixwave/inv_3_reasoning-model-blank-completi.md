# Investigation: Reasoning-model blank-completion + retry/backoff liveness stall on the D8 4-role Judge (verdict-token starvation of an UNBOUNDED reasoning burst, compounded by account-wide 429 congestion and generous per-call liveness walls). It is NOT a faithfulness hole — the blank already fails closed to per-claim UNGROUNDED; the surviving defect is a latency/liveness stall that blocks the final manifest + DeepTRACE scoring.

## Root cause
The predicate responsible is `_judge_reasoning_block(slug, effort)` at src/polaris_graph/roles/openrouter_role_transport.py:364-375. For the benchmark Judge (moonshotai/kimi-k2.6, the unpinned 21-provider default at :211) it returns a BARE, UNBOUNDED reasoning block `{"enabled": True}` — no effort tier (kimi advertises none) and, critically, NO `reasoning.max_tokens` cap. That block is set into the body at :1343-1347 and re-set on every retry rung at :1808-1812, against a top-level `max_tokens` of only 16384 (`PG_D8_VERDICT_MAX_TOKENS`, resolved at :1443-1449). OpenRouter counts reasoning tokens as OUTPUT tokens drawn from that same top-level budget (confirmed by the OpenRouter reasoning-tokens doc + the Anthropic "max_tokens must be strictly higher than the reasoning budget" rule), so a deliberative kimi pass that reasons past ~16k emits ZERO `content` → the observed "blank verdict with bare reasoning" (#1026) at :2162-2168. WHY THE PREDICATE IS BLIND: the numeric reasoning cap that PREVENTS exactly this (`build_judge_reasoning_block` -> `reasoning_cap_for`, src/polaris_graph/llm/judge_reasoning_block.py:90-148) is applied ONLY to GLM-family slugs (`is_glm_slug`); every other reasoning model — including the kimi Judge — falls through to the un-capped `{"effort": effort}` / bare `{"enabled": True}` path, so nothing bounds kimi's chain-of-thought below the 16384 output budget. The code comment at :1428-1431 literally states the assumption ("16384 leaves ample room for kimi reasoning AND the verdict") but never enforces it — it is an unguarded assumption, not a bound. The STALL is the second-order effect: for a bare-reasoning model the ONLY recovery is provider rotation (effort-stepping is inert), so each blank appends the provider to `ignore` and retries; under the run's ~31 account-wide 429s (the shared OpenRouter TPM/RPM is consumed by the all-GLM generator+mirror+sentinel burst) each retry also rate-limits into the #1173 backoff sleep, and the per-call Judge liveness walls are generous — per-POST httpx timeout 900s (:514) which the composition watchdog explicitly does NOT interrupt mid-flight (:536-546), and a Judge total-deadline wall floored at 1800s (`PG_ROLE_TRANSPORT_TOTAL_S_JUDGE_FLOOR`, :592-604). So a single congested judge claim sits for minutes in a hung/trickling POST or backoff (the observed 8+ min, ~10% CPU, no new log line) before force-closing to a fail-closed UNGROUNDED; across the ~178-claim final scoring pass at judge concurrency <=4 (:1080-1116) this compounds into the pass-blocking stall.

## Best practice 2025/2026
Current deep-research / RAG / grounding-judge systems handle empty completions from reasoning models under rate limits with four converging techniques, all matched to the fix below. (1) BOUND THE REASONING BUDGET, DON'T JUST RAISE max_tokens: OpenRouter's reasoning-tokens guide confirms reasoning tokens are output tokens drawn from the same budget, that `reasoning.effort` and `reasoning.max_tokens` are the two (mutually exclusive) controls, and CRITICALLY that 'for models that only support reasoning.max_tokens, the effort level will be set based on percentages' — i.e. a model exposing reasoning but no effort tiers (exactly kimi-k2.6) IS controllable via `reasoning.max_tokens`. Community testing (Felix Horvat, Medium 2026; HN 47475518) documents the identical 'reasoning ate the whole budget -> empty content' failure and prescribes reserving explicit content headroom. (2) 429 HANDLING IS ARCHITECTURE, NOT JUST RETRIES: exponential backoff WITH JITTER honoring Retry-After/x-ratelimit-* headers is necessary but insufficient; sustained reliability needs provider fan-out/load-balancing and, above all, CONCURRENCY CAPPED TO THE SUSTAINABLE RATE ('if your account supports 10 concurrent, don't run 200 coroutines and hope' — getmaxim.ai, portkey.ai, requesty.ai, truefoundry.com 2026) — a steady low concurrency finishes faster than a self-inflicted 429 storm. (3) FAIL-FAST-AND-ROTATE on a slow/trickling host rather than burning the full per-call wall (truefoundry LLM failover). (4) STRUCTURED-OUTPUT CONSTRAINED DECODING (JSON-schema / response_format via FSM token masking) mathematically prevents an empty content field by masking invalid/empty tokens at generation time (tensoria.fr, dev.to/pockit, arXiv 2502.14905 'Think Inside the JSON') — used as a belt-and-suspenders guarantee that a reasoning trace is always followed by a schema-valid verdict. Sources: https://openrouter.ai/docs/guides/best-practices/reasoning-tokens ; https://medium.com/@fhorvat90/i-tested-reasoning-tokens-on-5-llms-via-openrouter-most-models-silently-drop-them-b8071b5d857d ; https://www.getmaxim.ai/articles/handle-429-errors-in-production-llm-applications/ ; https://portkey.ai/blog/rate-limiting-for-llm-applications/ ; https://www.truefoundry.com/blog/llm-failover-load-balancing-provider-outages ; https://tensoria.fr/en/blog/structured-outputs-llm-production ; https://arxiv.org/html/2502.14905v1

## Proposed fix
PRIMARY (surgical, faithfulness-neutral) — bound the kimi Judge's reasoning so it can NEVER eat the 16384 output budget, WITHOUT narrowing the 21-provider spread. File: src/polaris_graph/roles/openrouter_role_transport.py, function `_judge_reasoning_block` (:364). Change its signature to also accept the resolved verdict budget, and for a kimi slug return a NUMERIC-capped block reusing the proven helper: `return {\"enabled\": True, \"max_tokens\": reasoning_cap_for(max_tokens)}` (import `reasoning_cap_for` from src/polaris_graph/llm/judge_reasoning_block.py:90). `reasoning_cap_for(16384)` yields ~12384 (reserves the >=4000-token content floor), strictly below the top-level 16384 → the verdict JSON always has room, so the starvation blank cannot occur. Keep top-level `max_tokens` UNCHANGED at 16384 (:1449) so all 21 kimi endpoints (DeepInfra 16384 floor) stay in the OpenRouter load-balance — no provider narrowing. Thread the budget in at the two call sites that already know it: :1347 (`_judge_reasoning_block(model_slug, _REASONING_EFFORT, verdict_budget)`) and :1812 (retry rung). Note `reasoning.max_tokens` and `reasoning.effort` are mutually exclusive, which is fine because kimi already carries no effort. GUARD (operator rule 'read the API, don't guess'): before shipping, do a live `GET /models` + `/endpoints` read for moonshotai/kimi-k2.6 to confirm its endpoints accept `reasoning.max_tokens` without 400; if a subset rejects it, keep the bare block for those and lean on the throttle+liveness levers. SECONDARY LEVERS (tune existing knobs, add nothing): (a) Concurrency — the judge semaphore (default 4) + AIMD (default-on) already exist; for an account-shared-rate all-GLM run seed a lower steady judge concurrency (e.g. PG_FOUR_ROLE_JUDGE_CONCURRENCY=2) so the judge stops adding to the account-wide 429 storm. (b) Liveness — set a judge-specific tighter per-call wall (lower PG_ROLE_TRANSPORT_TOTAL_S_JUDGE_FLOOR from 1800 toward the healthy p99, e.g. 300-600, mirroring the proven Sentinel 300s / entailment 150s fail-fast) so the already-ON force-close rotation abandons a hung/trickling congested provider in minutes-not-tens-of-minutes. (c) OPTIONAL belt-and-suspenders — add JSON-schema `response_format`/`structured_outputs` to the judge body so constrained decoding masks empty-content tokens; gate behind a flag since response_format support varies by provider and can narrow the spread. FAIL-CLOSED (item 4) is ALREADY correct — do NOT touch it: on ladder exhaustion the transport re-raises BlankVerdictError/RoleTransportError, sweep_integration `future.result()` (:701-702) propagates it, and with PG_ROLE_TRANSPORT_DEGRADE ON (default) it consumes per-claim as an UNGROUNDED degrade that `_compose_final_verdict` downgrades (never upgraded/approved). The fix keeps this intact.

## Faithfulness-neutral: True
The primary change only reshapes a transport-level TOKEN BUDGET (splitting the existing 16384 into a bounded reasoning cap + a guaranteed verdict content floor); it touches NO verification threshold. strict_verify, NLI entailment, the 4-role/D8 arbitration logic, and provenance span-grounding are all untouched. The blank-still-fails-closed path (blank -> BlankVerdictError -> per-claim UNGROUNDED degrade, never an approved/upgraded verdict) is explicitly preserved. It honors §-1.3 WEIGHT-not-FILTER: the concurrency and liveness levers govern only HOW MANY judge POSTs run and HOW FAST a hung host is abandoned — never which claim/source passes; no caps/targets/thinners are added to sources or claims, and basket/consolidation faithfulness is not relaxed. Direction of error is safe: any residual blank under-credits (disclosed gap), never over-credits.
