# Investigation: Non-substantive page-furniture ("chrome") that is a FAITHFUL verbatim copy of its source span passes strict_verify and renders as a finding. This is a recurrence of the I-wire-013 "blind chrome predicate" defect on 4 NEW furniture vocabularies (service-offline notice, OpenAlex record scaffold, "Name - Honorific" byline, "Publication date:/NN:N" masthead) that the enumerated denylist never enumerated. It is a render-QUALITY gap, not a faithfulness gap.

## Root cause
The one shared render-side quality predicate is `is_render_chrome_or_unrenderable` (weighted_enrichment.py:2663). Its NEW-category leg `_is_new_chrome_category` (:2609) ends in the containment cascade `_contains_forensic_chrome` (:2513). That cascade — and every leg it fans into (`_MASTHEAD_CHROME_RE`, `_SUBMISSION_META_RE`, `_BIBLIO_CHROME_RE`, `_METADATA_RECITAL_RE`, `_is_short_nav_topic_item`, the box1/boxc/drb72/iwire016 rule packs) plus the density screen `chrome_furniture_screen.is_furniture_dominant` (:2634) — is an ENUMERATED, LABEL-ANCHORED denylist tuned precision-first (§-1.3: over-strip a real finding is worse than a leak). Each rule is keyed to specific literal tokens. Any furniture whose surface vocabulary is not on that list is passed through. Mechanism per surviving form (each proven False by running the real predicate): (1) "This journal is currently offline." — a grammatically complete, stopword-dense sentence (subject+copula+status adjective+period); it carries no URL/DOI/ORCID/email/institution/volume token and there is NO service-status/dead-fetch-interstitial rule anywhere (the DOI-Not-Found and cookie classes are the only interstitials covered), so it clears `_base_junk` and every leg. (2) "…DetailsLocations Year: 2026 Type: article Abstract: …" — the OpenAlex field-ladder; `_METADATA_RECITAL_RE` (:2207) requires the vocabulary "journal article … volume N … authored by", and `_BIBLIO_CHROME_RE` requires "doi:/issn:/volume title publisher", so the "Year:/Type: article/Abstract:" colon-label ladder and the glued UI-button token "DetailsLocations" match nothing. (3) "Varsha Parikh - Ms." — every byline rule demands a DUAL signal (email+institution, or ≥2 surname-digit superscript pairs, or a street/ZIP affiliation); a bare "Name - Honorific" directory stub carries none, and `_is_short_nav_topic_item` (:2255) requires a trailing BARE ORDINAL which "- Ms." lacks. (4) "Publication date: December 2022. 87:2 K." — `_MASTHEAD_CHROME_RE` (:1608) enumerates "date issued / pages posted / posted: / last revised / working paper NN / vol.: N" but NOT the field label "Publication date:", and the bare volume:issue "87:2" colon-locator matches no rule because `_MASTHEAD_VOL_TOKEN_RE` needs the literal "Vol/Issue/No." keyword. strict_verify cannot and must not catch any of these: chrome text SELF-ENTAILS (the claim string is byte-identical to its own evidence span, so span bounds are valid, every decimal in the sentence appears in the span, and content-word overlap is trivially ≥2). Faithfulness is structurally the wrong instrument; only a quality/chrome predicate can catch furniture, and the denylist has a hole for each new vocabulary — the exact recurrence pattern flagged in I-wire-013.

## Best practice 2025/2026
Current deep-research / RAG systems solve chrome-as-content with a TWO-STAGE, confidence-routed extractor rather than a single enumerated denylist. Stage 1 is a fast heuristic main-content extractor operating on RAW HTML — Resiliparse HTML2Text (hand-crafted tag rules + regex, precision-first) and Trafilatura 2.1 / jusText, which score blocks by STOPWORD DENSITY and LINK DENSITY (boilerplate = low stopword density + high link density) instead of matching specific phrases; this generalizes to unseen furniture vocabularies, unlike a literal denylist. The 2026 WCXB benchmark (arXiv 2605.21097) and rs-trafilatura formalize PAGE-TYPE-AWARE extraction (article/forum/product/collection/listing/documentation/SERVICE) with per-type profiles, and — key for the recurring-blindness problem — CONFIDENCE ROUTING: any page/unit scoring below ~0.80 is escalated to a neural/LLM extractor (MinerU-HTML) rather than trusted. Firecrawl / Scrape.do guidance is that raw HTML is 40-60% boilerplate and that garbled/furniture context forces the synthesis LLM to hallucinate, so boilerplate removal is a mandatory pre-synthesis stage with quality gating. POLARIS already ports these SIGNALS to a text-level predicate because the render seam has no HTML (the code comment at weighted_enrichment.py:2081 states exactly this). Mapping to POLARIS: the surgical move is to add the 4 missing high-precision structure-anchored rules to the existing denylist (matches the current architecture and the 200-LOC/precision-first constraints); the DURABLE move that stops the whole recurrence class is (a) a general stopword/link-density backstop rule in is_furniture_dominant (jusText-style, catches novel furniture without a literal token list), and (b) routing residual low-confidence units to the existing chrome-as-claim canary (already enforce-mode, floor 0.05) or an LLM chrome-classifier fallback — the standard 'heuristic on the 92%, neural on the hard cases' pattern. Sources: https://arxiv.org/html/2605.21097 (WCXB benchmark 2026); https://trafilatura.readthedocs.io/en/latest/evaluation.html and https://dl.acm.org/doi/pdf/10.1145/3539618.3591920 (Bevendorff et al., web content extraction eval); https://dev.to/murroughfoley/rs-trafilatura-page-type-aware-web-content-extraction-in-rust-2ppf (page-type-aware + confidence routing); https://www.firecrawl.dev/blog/llm-grounding and https://scrape.do/blog/llm-ready-data/ (boilerplate degrades grounding; mandatory pre-synthesis cleaning).
