# Investigation: Explicit user-stated TEMPORAL scope constraint ("based on academic research published before June 2023") not enforced as a grounding constraint — post-window 2024/2025/2026 sources ground cited findings. This is a scope / instruction-following (faithfulness-to-the-question) defect, NOT a faithfulness-engine defect. It is the §-1.3-legitimate case: honoring an EXPLICIT user-stated date bound is scope-compliance, not a breadth-forcing hard filter.

## Root cause
The blind predicate is constraint_enforcement._row_out_of_window (src/polaris_graph/retrieval/constraint_enforcement.py:136-152) fed by _row_pub_ym (90-111). It can only judge a row out-of-window when the row carries one of six structured date keys (pub_date / publication_date / published / date / publication_year / year). The ONLY code that populates those keys is live_retriever.py:6227-6234, and it sets them ONLY from the OpenAlex authority enrich (_auth_dict['publication_year']/['publication_date'] resolved at live_retriever.py:5801-5818). Every non-OpenAlex source — crawled web pages (St Louis Fed, OECD web, Microsoft Work Trend blog, CQ Researcher), non-indexed institutional PDFs, general-web/news — therefore arrives with NO date field. For those rows _row_pub_ym returns None, and _row_out_of_window returns False by explicit design (line 140-141: "Undated => False (fail-open, kept)"). So even with timeline_hard_active True, the timeline branch (constraint_enforcement.py:308-322) adds ~zero urls to grounding_excluded_ids, and the undated post-window sources sail straight into the billed grounding set evidence_for_gen at the selection seam (run_honest_sweep_r3.py:13669-13699). I verified the extraction side is NOT at fault: running the real extractor on the canonical DRB-II idx-56 question returns timeline_strictness=hard, date_end_iso=2023-06, trigger_span="needs to be based on academic research published before June 2023" — _HARD_TIMELINE_REQUIREMENT_RE (intake_constraint_extractor.py:105-112) fires correctly, and workforce.yaml has no date_range so scope_gate.py:1026-1030 fills the (None, 2023-06) ceiling into the protocol. The 180→81 mask that DID fire was the source-type/facet + blocked-reference (do-not-view Salari review) masking — masked by TOPIC. The date ceiling never bit because its predicate is starved of per-source publication dates for every non-OpenAlex source and fail-opens on the resulting Nones. In short: the pipeline extracts a publication date from structured API metadata only; it performs NO publication-date extraction from the HTML/PDF it already fetched, so the temporal-enforcement predicate is blind to the majority of the corpus.

## Best practice 2025/2026
Publication-date extraction (directly solves Part A): the canonical open-source approach is a cascade over ALREADY-fetched HTML — the htmldate Python library (JOSS paper https://www.theoj.org/joss-papers/joss.02439/10.21105.joss.02439.pdf) probes, in precedence: JSON-LD datePublished (schema.org datePublished, ISO-8601, https://schema.org/datePublished) inside <script type="application/ld+json">, then OpenGraph/meta tags article:published_time / citation_publication_date / DC.date.issued / prism.publicationDate, then microdata itemprop="datePublished", then <time datetime=...>, then URL date patterns (/YYYY/MM/), then body heuristics. Parse.ly's metadata guidance (https://docs.parse.ly/metadata-jsonld/) and Webz.io's extractor overview (https://webz.io/blog/web-data/articles-publication-date-extractor-an-overview/) confirm JSON-LD as the most reliable signal, with meta tags and <time> as fallbacks. Reference impls: htmldate, github.com/preciz/html_date. Temporal-constraint compliance in research agents (Part B rationale): the 2025 temporal-RAG literature — survey "It's High Time: A Survey of Temporal Question Answering" (arXiv 2505.20243), plus TA-RAG / DyG-RAG / T-GRAG and IA-RAG (arXiv 2606.06044) — the pattern is: (1) extract the temporal constraint from the query at inference time, (2) filter/rank candidate documents within the window at document-timestamp granularity, (3) distinguish document-level publication date from fact-level validity ("a 2023 document may discuss 2020 events"). For drb_72 the stated constraint is explicitly on the source's PUBLICATION date ("academic research published before June 2023"), so document-level publication-date enforcement is exactly the right granularity. Best practice is thus: resolve a publication date for every retrieved source (structured metadata first, HTML/PDF extraction second), then enforce the query-stated window as a retrieval/grounding gate with an explicit, disclosed policy for sources whose date cannot be resolved.

## Proposed fix
PART A (highest leverage, faithfulness-neutral) — give the enforcement predicate the dates it lacks by extracting publication date at fetch time for non-OpenAlex sources. Add src/polaris_graph/retrieval/publication_date_resolver.py exposing resolve_publication_date(*, raw_html: str, jsonld: str, url: str, pdf_meta: dict|None) -> tuple[str|None, int|None] that runs the htmldate cascade over content ALREADY fetched: (1) JSON-LD datePublished/dateCreated from the existing _extract_jsonld_blocks output; (2) meta article:published_time / citation_publication_date / citation_date / DC.date.issued / prism.publicationDate; (3) microdata itemprop=datePublished; (4) <time datetime=...>; (5) URL /YYYY/MM/ pattern; (6) PDF /CreationDate for PDF sources. Return normalized ISO YYYY-MM[-DD] + year; only accept 1900<=year<=2100. Wire it in live_retriever.py immediately after line 5818, guarded ONLY for the OpenAlex-miss case: `if _pub_date is None and _pub_year is None and _resolve_pubdate_from_html_enabled(): _pub_date, _pub_year = resolve_publication_date(raw_html=..., jsonld=jsonld, url=cand.url, pdf_meta=...)`. Because the existing 6227-6234 block already stamps pub_date/publication_year/year from _pub_date/_pub_year, no downstream change is needed — constraint_enforcement._row_pub_ym and evidence_selector._row_year both immediately start seeing dates on web/institutional rows. Env gate PG_RESOLVE_PUBDATE_FROM_HTML (LAW VI), default operator-set. Zero new network (reuses fetched HTML/JSON-LD). Pure placement/provenance metadata — never enters a verified claim.
PART B (policy for rows that remain genuinely undated after Part A — OPERATOR QUESTION, hard-vs-soft) — add env knob PG_TIMELINE_HARD_UNDATED_POLICY read inside constraint_enforcement.build_scope_enforcement with three values: `keep_disclose` (current fail-open, default) | `mask` (fail-closed: under a HARD stated bound an undated row is treated as out-of-window, added to grounding_excluded_ids + disclosed) | `resolve_then_mask` (mask only rows still undated after Part A). Concretely, in the timeline block (constraint_enforcement.py:308-322), when timeline_hard_active and policy in {mask, resolve_then_mask} and _row_pub_ym(row) is None, add url to grounding_excluded_ids with a distinct record action `timeline_hard_undated_excluded_disclosed` and reason "publication date unverifiable under hard stated timeline". Recommendation: keep default `keep_disclose` until Part A lands, then operator flips to `resolve_then_mask` (maximally scope-compliant with minimal false-masking, since Part A removes the resolvable-date false-negatives). This choice — hard-exclude an unverifiable-date source vs down-weight+disclose it — is the operator decision to sign off; it is NOT a §-1.3 breadth cap because it fires ONLY under an explicit user-stated bound and every excluded row stays in the pool + disclosure.
PART C (secondary, composition consistency) — the Methods/Limitations "horizon through June 2023" sentence is written from the protocol date_range while post-window sources are cited, creating internal inconsistency. Derive that disclosure sentence from the actual grounded-source date span and the scope_timeline_disclosure.json enforcement counts (masked N, undated-kept M), not from a parroted protocol bound. Separate composer/render change; flag but do not bundle.

## Faithfulness-neutral: True
The fix never touches strict_verify / NLI entailment / 4-role D8 / provenance span-grounding. Part A only populates fetch-time provenance metadata (pub_date/year) that the faithfulness engine already ignores. Part B reuses the ALREADY-blessed I-scope-001 HARD grounding-mask seam (run_honest_sweep_r3.py:13656-13699 / constraint_enforcement grounding_excluded_ids) which is the same class of §-1.3 exception as the operator do-not-view registry: the masked source is withheld only from the ANSWER GROUNDING surface and is KEPT in the evidence pool + disclosure (weight-don't-filter, consolidate-don't-drop preserved for the corpus; basket faithfulness untouched). Enforcing an EXPLICIT user-stated date ceiling is faithfulness-TO-THE-QUESTION / instruction-following, expressly distinguished from the §-1.3-banned "hardcoded cap/target/thinner to force a breadth number": it fires ONLY when the extractor detects an explicit requirement clause (already opt-in via _HARD_TIMELINE_REQUIREMENT_RE), only for the stated window, and it strengthens rather than relaxes any faithfulness verdict. Both parts are env-gated (LAW VI) and byte-identical when off.
