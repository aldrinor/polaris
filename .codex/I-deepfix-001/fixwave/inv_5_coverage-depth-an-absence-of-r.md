The query frontier is built by fs_researcher_query_gen._plan_expert_facet_queries (lines 284-441) from expert_facet_planner.plan_expert_facets — abstract sub-TOPIC × analytical-angle queries (mechanism/stakeholder/temporal), scope-anchored to the question's FIELD. The STORM-style named-entity/perspective widener sub_entity_query_expander is PG_SUBENTITY_QUERY_EXPANSION default OFF (line 62). (2) The one GenAI primary that WAS retrieved (Brynjolfsson [7]) is a V30 contract-frame row (paywalled, DOI-only). frame_fetcher._fetch_frame_entity_inner resolves OA via Unpaywall/best_oa_location which KEYS ON THE JOURNAL DOI (frame_fetcher.py:361). The engine is NOT blind — it does its job on a starved input; the blindness is in the FETCHER, which never tries a title/author OA-preprint discovery when DOI-Unpaywall reports closed.

(3) Cross-source synthesis is STARVED, not broken — two sub-mechanisms. (3a) depth_synthesis eligibility = _distinct_origin_supports(basket) >= _CROSS_SOURCE_MIN_ORIGINS(=2) (depth_synthesis.py:103,484,559-561,768-772). Only 3/65 baskets clear it. Clustering that keeps genuine same-finding corroborators in SEPARATE clusters (different phrasing/number-format) yields singleton baskets → 62/65 ineligible. (3b) The M6 analytical composer cross_source_synthesis.compose_cross_source_analytical_units pairs baskets ONLY when they share an EXACT normalized subject|predicate anchor (_basket_anchor lines 85-94; bucketing 521-534). Two baskets stating the same finding with different predicate wording never become a candidate pair → eligible_pairs=0 → the certified NLI/edge licensing engine is never even asked → silent zero analytical yield (the fail-loud canary at 595-603 only fires when pairs EXIST but 0 survive, so an all-singleton corpus is silent). So "0 cross-source" is the join of (1) empty core, (2) dropped landmark primary, and (3) an anchor/clustering recall too tight to assemble corroboration baskets. Every one is a WIDEN/CONSOLIDATE lever; the faithfulness engine is untouched throughout.

## Best practice 2025/2026
Current deep-research/RAG systems solve this exact class as follows:

(A) COVERAGE via gap-driven iterative retrieval + query decomposition + citation-chasing. Agents decompose the question, retrieve, self-critique for uncovered aspects, and re-retrieve to close gaps (Search-R1 policy optimization, arXiv 2602.19526; "From Web Search towards Agentic Deep Research", arXiv 2506.18959; Deep Research survey, arXiv 2508.12752; DeepResearch-Slice retrieval-utilization gap, arXiv 2601.03261). To reliably surface the empirical CORE, the systematic-review best practice is CITATION CHASING / SNOWBALLING from seed papers via the OpenAlex and Semantic Scholar citation-graph APIs (citationchaser; automated-citation-searching simulation, PMC12621532; Paperfetcher, arXiv 2110.12490) — forward+backward neighbors of a seed reliably pull in the co-cited landmark studies. OpenAlex also returns ALL host locations for a work (incl. the free preprint) even when the journal DOI is closed — the fix for DOI-keyed OA misses.

(B) DEPTH / cross-document synthesis via recall-first candidate pairing + NLI-confirmed clustering. The "synthesis gap" (the DeepResearch-Bench analysis dimension) is now a named, measured failure: "Can Deep Research Agents Retrieve and Organize? Evaluating the Synthesis Gap with Expert Taxonomies" (arXiv 2601.12369) and "Faithful by Construction: Claim-Anchored Attribution for Multi-Document Summarization" (arXiv 2606.23989). The state-of-the-art clustering method: agglomerative clustering of claim embeddings (sentence bi-encoder cosine threshold) with each merge CONFIRMED by BIDIRECTIONAL entailment from a 3-class NLI model (refuse non-mutually-entailing merges); the cross-source relation/conflict candidate pool = cross-cluster pairs that are embedding-similar yet unmerged, PLUS pairs sharing a named entity or numeric quantity, then NLI-scored (comprehensive RAG surveys arXiv 2507.18910 and arXiv 2410.12837; SIGIR-2025 LiveRAG, arXiv 2507.04942). This is exactly recall-first CANDIDATES gated by NLI — widen who the engine sees, keep NLI as the merge/relation gate — which maps 1:1 onto POLARIS's certified consolidation_nli engine and is faithfulness-neutral.
