# Investigation: Aspect-blind topicality gate: a same-entity / wrong-sub-domain source ("GenAI" entity satisfied, "labor-market" aspect absent) passes the off-topic discriminators because both judge ENTITY/subject-domain match, not the question's ASPECT/facet. Class = query-aspect mismatch / topic drift surviving a relevance gate that scores entity overlap.

## Root cause
The entire off-topic suppression chain is driven by ONE boolean sidecar, `weighted_enrichment._is_confirmed_offtopic` (src/polaris_graph/generator/weighted_enrichment.py:131-145), which is a PURE READER: it returns True only if an upstream judge already stamped `topic_offtopic_demoted is True` OR `content_relevance_label in {demoted, escalated_demoted}` (labels set at weighted_enrichment.py:120). It can only suppress what an upstream judge flagged. There are exactly TWO upstream judges that can set those flags, and BOTH decide ENTITY/subject match, never the ASPECT:

(1) Topic gate LLM prompt `_build_batch_prompt` (src/polaris_graph/retrieval/topic_relevance_gate.py:168-199, decision text 181-186): it asks the model to judge ON/OFF for "the research question's SUBJECT DOMAIN" and defines OFF as "clearly about a DIFFERENT subject (different disease, different field, different population)" plus "When in doubt, answer ON." [21] "Students' Perceptions of GenAI ... dishonestly in L2 writing" IS about the same subject (generative AI), so it is NOT a different disease/field/population, and the fail-open "when in doubt answer ON" biases keep -> verdict ON -> no `topic_offtopic_demoted` stamp. The prompt has no notion of the question's aspect ("effect on the labor market / employment"); it collapses the multi-facet question to its dominant entity.

(2) W2 content-relevance judge (src/polaris_graph/retrieval/content_relevance_judge.py): Stage-1 Qwen3-Reranker-0.6B scores a single scalar P(yes) for (research_question, passage) via the instruction "judge whether the document passage is RELEVANT evidence that helps answer it" (lines 486-489). A cross-encoder's scalar is dominated by the shared salient entity term; a passage saturated with "generative AI" tokens is pulled toward P(yes) high regardless of the labor-market facet. The band is LOW=0.05 / HIGH=0.70 (line 58-59): a high-entity-overlap passage scores >=0.70 -> LABEL_RELEVANT full weight and is NEVER escalated to the GLM. Worse, the run uses window max-pool (lines 372-411): each passage takes the BEST-scoring of up to 6 windows, so the single most-GenAI-dense window sets the score, biasing an off-aspect paper upward. Even the Stage-2 GLM escalation prompt `_CONTENT_RELEVANCE_PROMPT` (src/polaris_graph/generator/relevance_judge.py:238-256) judges "RELEVANT EVIDENCE for answering the QUESTION" — its INSUFFICIENT branch mentions "the right subject with no answering content" but only fires IF the reranker escalates, which for a high-entity-overlap passage it does not.

Net mechanism: entity-match dominates aspect-match at BOTH judges, so [21] is never stamped confirmed-off; `_is_confirmed_offtopic([21])` returns False; the DEFER-1 cite-suppression (weighted_enrichment.py:780), the section-assignment hold-out `_drop_offtopic_rows_for_assignment` (multi_section_generator.py:2028), and the F3 compose demote `_compose_relevance_floored_ev_ids` (multi_section_generator.py:4340-4368) all skip it; [21] flows to citation and grounds the Abstract + OPPORTUNITIES lead. The 24 topic-gate demotions and 40 DEFER-1 suppressions in the log were all CLEARLY cross-subject sources (cosmetics/recycling/M&A) — the guard is calibrated for cross-domain contamination, structurally blind to same-entity-wrong-aspect.

## Best practice 2025/2026
2025/2026 deep-research/RAG systems converge on ASPECT/FACET-scoped relevance rather than a single entity-overlap scalar: (1) Query/aspect decomposition — decompose the question into atomic facets (entity + aspect/relation/outcome) and judge a source against the SPECIFIC facet, not the whole topic: 'Query Decomposition for RAG' (arXiv 2510.18633), 'Multi-Head RAG: Solving Multi-Aspect Problems' (arXiv 2406.05085), UniFAR facet-aware retrieval (arXiv 2602.23766), and 'Facet-Level Tracing of Evidence Uncertainty and Hallucination in RAG' (arXiv 2604.09174) which decomposes each question into atomic reasoning facets and shows hallucination arises from retrieval-generation ASPECT misalignment even under high nominal relevance. (2) Relevance-alone is insufficient — 'Relevance Isn't All You Need: Scaling RAG Systems With Multi-Criteria Reranking' (arXiv 2504.07104) argues a single relevance score must be augmented with intent/aspect criteria. (3) Nugget/facet-level answer relevance — TREC 2025 RAG Track (arXiv 2603.09891) moves to long multi-sentence NARRATIVE queries and AutoNuggetizer (arXiv 2504.15068) scores per-nugget vital-vs-okay coverage, i.e. does the source address the specific information need, not just the subject. (4) LLM-as-judge context-relevance rubric decomposition — RAGAS/TruLens/Bedrock separate 'context relevance' from 'answer relevance' precisely to catch 'right subject, wrong evidence' (the exact [21] failure), and Recursive Rubric Decomposition (2026) / rubric-based judging break a holistic relevance call into per-facet subpoints for a same-entity/wrong-aspect verdict. The through-line: score relevance to the FULL question INCLUDING its constraint facet (the 'labor market' aspect), via a facet-conditioned judge or NLI-to-the-aspect, and treat entity match as necessary-not-sufficient.

Sources: arXiv 2510.18633; arXiv 2406.05085; arXiv 2602.23766; arXiv 2604.09174; arXiv 2504.07104; arXiv 2603.09891 (TREC 2025 RAG); arXiv 2504.15068 (AutoNuggetizer); RAGAS/TruLens context-relevance decomposition; Recursive Rubric Decomposition 2026.

## Proposed fix
PRIMARY (load-bearing) — make the topic gate ASPECT-SCOPED. File: src/polaris_graph/retrieval/topic_relevance_gate.py, function `_build_batch_prompt` (lines 168-199). Replace the entity-level instruction block (lines 181-186) with a two-part, facet-scoped rubric that is domain-agnostic (LAW VI — no hardcoded 'labor market'; the aspect is derived at runtime from `research_question`):
  - Instruct the model to FIRST name the question's SUBJECT ENTITY and its SPECIFIC ASPECT (the outcome / relation / sub-domain / population the question asks about).
  - A source is ON-TOPIC only if it plausibly bears on BOTH the entity AND that specific aspect.
  - A source about the SAME entity but a DIFFERENT aspect / use-case / population is OFF-TOPIC ("same subject, wrong question") — give a domain-neutral one-shot exemplar (entity X aspect A asked; a source on entity X aspect B => OFF).
  - Preserve fail-open explicitly: "If you genuinely cannot tell whether the specific aspect is addressed, answer ON." Keep the exact `<index>: ON|OFF` output contract (parser at lines 202-238 unchanged).
This flips [21] to OFF, which through the UNCHANGED default demote path (lines 375-383) stamps `topic_offtopic_demoted=True` (keep-all, no hard-drop). `_is_confirmed_offtopic([21])` then returns True, so the existing DEFER-1 cite-suppression (weighted_enrichment.py:780), section-assignment hold-out (multi_section_generator.py:2028), and F3 compose demote (multi_section_generator.py:4340-4368) all engage — [21] is withheld from citation, held out of the OPPORTUNITIES section, demoted from composed findings, and thus cannot ground the Abstract; it stays in evidence_pool + the off-topic-excluded disclosure. No new plumbing, no new flag, faithfulness engine untouched.

REINFORCING (defense-in-depth, cheap) — sharpen the GLM escalation. File: src/polaris_graph/generator/relevance_judge.py, constant `_CONTENT_RELEVANCE_PROMPT` (lines 247-250). Extend the INSUFFICIENT definition to explicitly cover same-entity-wrong-aspect: "INSUFFICIENT ALSO applies when the PASSAGE concerns the QUESTION's main entity but a DIFFERENT aspect / use-case / population than the QUESTION asks about." Maps onto the existing INSUFFICIENT->demote weight (no new labels). Catches any off-aspect source that DOES land in the reranker's ambiguous band.

SECONDARY (optional hardening; do NOT tighten the numeric band — that would be the §-1.3-banned knob). The reranker's >=HIGH short-circuit (content_relevance_judge.py, band line 58-59) is a residual blind spot because it never escalates a high-entity-overlap passage. The §-1.3-faithful fix is NOT to lower the threshold but to add an ASPECT-CONDITIONED second score: also score the passage against an aspect-focused reformulation of the question and ESCALATE to the GLM when the aspect-score is materially below the entity-score (an aspect-gap), surfacing same-entity-wrong-aspect into the GLM band without dropping or capping anything. Recommend deferring this to a follow-up; the topic-gate prompt is sufficient for the reported RED->GREEN.
