# HEADROOM NOTE (why these will not OOM 2xRTX3090Ti or 429-storm):
# Widening breadth here adds MORE sequential retrieval work, not bigger tensors. Peak GPU VRAM is set by
# the per-forward-pass batch sizes and PG_CONTENT_RELEVANCE_SCORE_CHUNK=2 (the reranker one-pass OOM
# guard) — ALL UNCHANGED — so more sources = more sequential embed/rerank passes = more TIME on the same
# fixed-VRAM kernels, never a larger peak. The L3 device split (embedder + W5 on cuda:0; reranker + NLI +
# consolidation-NLI on cuda:1) is untouched, and no clinical mineru-vLLM server contends on these
# workforce/coverage boxes. On the network side, the per-host politeness cap (=6) and the cross-host
# worker ceiling (=48) — the two 429 governors — are UNCHANGED, so per-host request RATE is identical to
# the safe run; only cross-host breadth grows, and the per-question retrieval wall (5400s) plus run-wall
# (14400s), both unchanged, bound total retrieval time so a wider frontier hands off a DISCLOSED partial
# corpus (retrieval_wall_hit) rather than looping into 429s. S2 deepener calls stay gated by the existing
# _S2_INTERVAL rate limiter. The only real cost is OpenRouter/S2 API spend + wall-clock, which the
# operator DNA (FAST + BEAT-BOTH, cost-not-a-limit) explicitly treats as non-limiting for these boxes.
