## The decision: flip default from "off" to "enforce"

This is the policy decision that closes the loop on the 2026-05-09 audit-revealed provenance-correctness gap. The architectural fix exists (I-bug-092) but is dormant by default. Without flipping, every production run still has the gap.

## Empirical evidence supporting the flip

**I-bug-094's live test ran against real OpenRouter Gemma 4 31B, 4/4 PASSED in 20.5s** before merge:

| Audit case | Real Gemma 4 31B verdict | Gate behavior |
|---|---|---|
| M2: β-cells/lipid metabolism inserted into URNCST adipocyte-metabolism span | NEUTRAL or CONTRADICTED | DROP (correct) |
| C2: GLP-1 RA class → semaglutide-specific upgrade | NEUTRAL or CONTRADICTED | DROP (correct) |
| C1: 69-80% reach <=6.5% claim with constituent numbers in span | NEUTRAL or CONTRADICTED | DROP (correct) |
| Positive control: conservative paraphrase | ENTAILED | KEEP (correct) |

This is empirical end-to-end confirmation: the prompt + Gemma 4 31B + wiring all together correctly identify the 3 audit-derived fabrication patterns AND accept legit paraphrases. The 4 most important audit cases work without any prompt tuning.

## Risks of flipping default to enforce

1. **Latency**: ~1 OpenRouter call per kept sentence. ~50 sentences/report × 1.5s/call = +75s per generation.
2. **Cost**: ~$0.025/report at Gemma 4 31B prices.
3. **False-positive paraphrase rejection**: the judge could drop legitimate paraphrases that happen to use synonyms (Codex flagged this in I-bug-092 brief verdict as `false_positive_risk`). Live test's positive control passed but that's a single sample.
4. **Judge fail-open during outages**: if OpenRouter has a transient outage, the gate fails open (returns ENTAILED+judge_error tag). Per I-bug-096 telemetry, an operator can see judge_error rate; per I-bug-097 unknown-mode safety net, a typo cannot disable. But sustained OpenRouter degradation = sustained gate degradation.
5. **Regression risk on non-audit patterns**: the 4 audit cases all worked, but there's a long tail of other potential mis-judgments we haven't sampled.

## Risks of NOT flipping default

1. **Production reports continue to ship M2/C2/C1-style fabrications.** That's the whole reason I-bug-092 exists.
2. **Gap-fix is structurally complete but operationally inert.** The cage exists; the lock is open.
3. **Carney delivery context**: the audit found 1 fabricated mechanistic claim, 1 specificity inflation, and 1 unverifiable claim in a single report. Shipping reports with that kind of fabrication to Carney is the failure mode the entire entailment fix exists to prevent.

## What I'm proposing in this PR

**Single-line config change**: `_entailment_mode()` defaults to `"enforce"` instead of `"off"`. Update tests + docstring accordingly. Preserve the operator escape hatch — operators can still set `PG_STRICT_VERIFY_ENTAILMENT=off` to disable explicitly.

LOC: ~5 src/ + ~10 test adjustments. Changes:
- `_entailment_mode()`: default `"off"` → default `"enforce"`
- Module docstring: update mode description
- Update existing tests where `monkeypatch.delenv("PG_STRICT_VERIFY_ENTAILMENT", raising=False)` was relying on default-off behavior — these tests need to set explicit "off" OR be updated to expect enforce-default behavior

1. **Verdict** APPROVE / REQUEST_CHANGES on flipping the default NOW.
2. **Is the I-bug-094 4-case live evidence sufficient**, or do you want additional empirical signal (e.g. an I-bug-093 warn-mode run on a fresh tirzepatide generation with broader sample) BEFORE flipping?

4. **False-positive risk**: should we add a `PG_STRICT_VERIFY_ENTAILMENT_FAIL_OPEN_ON_PARAPHRASE=1` operator escape hatch for known-good content? My read: NO — the existing `=off` operator override is the escape hatch, and adding a paraphrase-specific bypass is feature creep.
5. **Migration path** — drop the test files that expected default-off (since CI was off-mode by default) or update them to set `=off` explicitly?
