## Query-generation method — CONFIRMED it is FS-Researcher, NOT STORM

Verified from real code. Production query generation is **FS-Researcher** (arXiv 2602.01566), one of the 2026 recency picks — `src/polaris_graph/retrieval/fs_researcher_query_gen.py:1-16,43-55`, pinned ON in the production slate via `PG_QGEN_FS_RESEARCHER=1` (`run_gate_b.py:1523`). The sweep dispatches it at `run_honest_sweep_r3.py:10289-10307`. Method: deconstruct the question into a todo-queue of sub-topics; derive ONE search query per todo; fetch and fold in; run a fixed 6-item self-review checklist whose deficits seed the next round's todos; repeat to 35 queries / 6 rounds. IterResearch is the fallback (`PG_QGEN_ITERRESEARCH`, default OFF — lost the validated re-bakeoff). Note: `graph.py`/`graph_v2.py`/`graph_v3.py` and `agents/synthesizer.py` are pipeline B (the LangGraph UI server), NOT the production sweep — the sweep is the standalone `run_honest_sweep_r3.py` CLI, so the question-gen node lives in the FS-Researcher module above, not a graph node. **STORM is a bake-off LOSER and is pinned OFF** (`PG_STORM_ENABLED_IN_BENCHMARK=0`, `run_gate_b.py:578,1902`); the research-planner facet path is also OFF (`PG_USE_RESEARCH_PLANNER=0`, `:586`). So the earlier stage-2 note naming STORM as a live source was wrong for production — corrected here.

## REQUIREMENT-AWARENESS deep-dive (the operator's dynamic-vs-hardcoded principle)

**(a) SCOPE FILTERS — requirement-AWARE (dynamic). Verdict: AWARE.** A real intake layer parses the user's NL prompt for hard scope constraints and flows them through retrieval + selection dynamically, not as a fixed gate:
- Extraction: `retrieval/intake_constraint_extractor.py:1-24` (dateparser + regex primary, injected-LLM fallback) pulls a date window (month-precision: "before June 2023" → end "2023-06"), language, and source-type. Called in the scope gate `nodes/scope_gate.py:1010-1037` under `PG_EXTRACT_USER_CONSTRAINTS=1` (`run_gate_b.py:5831`); parallel scope-facet extraction (source-type/jurisdiction/named include-exclude) at `:1045-1059` under `PG_EXTRACT_SCOPE_CONSTRAINTS=1` (`run_gate_b.py:619`). FILL-not-override: a template/operator constraint always wins; the extractor only recovers a constraint that was being dropped.
- Retrieval adjustment: the parsed date window drives an additive from/to_publication_date OpenAlex lane (`PG_OPENALEX_DATE_FILTER=1`, `run_gate_b.py:1741-1745`) and an in-window landmark-study expander that seeds the pre-print not a later re-publication (`PG_LANDMARK_EXPANDER=1`, `:1746-1750`).
- Filtering adjustment: `retrieval/constraint_enforcement.py:1-24` builds a per-URL scope-weight demote map + out-of-window tail-partition + HARD grounding mask, folded into the SAME selection sort key, under `PG_SCOPE_CONSTRAINT_ENFORCE=1` (`run_gate_b.py:618`; wired `run_honest_sweep_r3.py:14011-14063`). §-1.3-consistent: WEIGHT/DISCLOSE, not silent drop. Caveat: journal-only is extracted but DORMANT by operator veto (extractor docstring); and the OFF-TOPIC judge does NOT read these scope constraints — its "topic" is still only the main question's subject+aspect (stage 4), so scope-awareness lives in the date/facet WEIGHT seam, not in the topic gate.

**(b) DELIVERABLE REQUIREMENTS — requirement-BLIND (hardcoded). Verdict: BLIND.** No component parses user-specified tone, structure, deliverable type, reference/citation style, depth, or audience. The outline is chosen from a fixed domain template set (`config/scope_templates/{clinical,policy,tech,workforce,...}.yaml`) plus the evidence-emergent facet outline — none reads a deliverable spec. Section titles come from a hardcoded clinical 8-title allow-list (`multi_section_generator.py:784-793`) or generic 6-title list (`:800-807`) or evidence-derived facet titles (`:1454`). The per-sentence compose prompt is a fixed rigor prompt; reference rendering is a fixed `[N]` bibliography (`synthesis/report_assembler.py`); length is whatever survives strict_verify plus the analyst-synthesis layer — never a user target. A `grep` across the generator for tone/audience/citation_style/word_count found only incidental word hits, no requirement plumbing. The V30 Report Contract (`nodes/report_contract.py`) is a per-slug REQUIRED-ENTITY/slot spec authored by the operator in config, NOT parsed from the user's prompt.

**(c) THE KEY QUESTION — is there a component that parses the user's requirements and CONFIGURES the pipeline?** PARTIAL. For SCOPE there is: `intake_constraint_extractor` → protocol `user_constraints`/`scope_constraints` → `constraint_enforcement` reconfigures retrieval lanes + selection weights. For the DELIVERABLE (tone/outline shape/format/references) there is NONE — POLARIS runs one fixed compose/outline template regardless of what the user asks for stylistically. So the smart adaptive orchestrator the operator wants exists on the retrieval/scope axis but is absent on the deliverable/presentation axis.

## Ranked biggest gaps (all operator suspicions confirmed, ranked by report-quality impact)

1. **Holistic cross-section review is MISSING in production.** Sections are composed in parallel and concatenated; only faithfulness redaction, exact-verbatim de-recycle, and numeric fact-dedup touch the whole report. Tone, prose-level cross-section contradiction, and depth are never reviewed or adjusted. The reflector already exists (`cross_section_reflector.py`) but is wired only into pipeline B. **Fix: wire a whole-report review+revise pass post-assembly; strict_verify re-runs on every revised sentence.** Fixable, moderate effort (the module + the verify seam both exist).
2. **Off-topic judge is main-prompt-only, not sub-query-aware.** A sub-query-relevant source with a different subject entity than the main question gets OFF_SUBJECT → demoted AND deleted. **Fix: include the row's `query_origin` sub-query in the judge prompt; ON if relevant to EITHER; delete only when off both.** Fixable, small (prompt + one plumbed field).
3. **Outline is one-shot and title-starved.** One LLM call over a 150-row title+tier menu; no full-content view, no iteration after drafts. **Fix: feed basket digests (consolidated claims + corroboration) to the planner; add an outline-refine round.** Fixable, moderate.
4. **Fetch gives up after 2 failed attempts with no low-concurrency retry sweep**, so concurrency-overload timeouts become permanent losses. **Fix: end-of-retrieval retry pass at parallel-8 over the failed set (machinery exists in `resume_refetch` + the 921 wrapper pattern).** Fixable, small.
5. **Deliverable requirements are ignored (requirement-BLIND compose).** POLARIS parses the user's SCOPE (date/language/source-type) and bends retrieval to it, but never parses tone/structure/audience/reference-style/length — outline and compose run a fixed template. **Fix: extend `intake_constraint_extractor` to parse a deliverable spec and thread it into the outline planner + section-prose prompt + reference renderer.** Fixable, moderate (the intake pattern already exists; extend it to the compose side).
6. (Lesser) **Baskets are global, not grouped under sections** — consolidation and section assignment both exist but never meet; the section composer sees rows, not its baskets. Fixable, moderate.
