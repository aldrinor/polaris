## Context

Captured as Codex P2 advisory in `.codex/I-bug-092/codex_diff_audit.txt` iter 1:

> "Persistent judge/config failures fail open even in enforce mode, so a bad model slug, unsupported response_format, or API behavior change could make the sixth check inert while only emitting logs. Recommend telemetry/counters plus an env-gated live OpenRouter test."

This Issue covers the **counters half** of that advisory. The env-gated live OpenRouter test is a separate Issue (I-bug-094).

The risk is real: in `enforce` mode the judge fail-open path returns `("ENTAILED", "judge_error: <ExceptionClass>")`. If OpenRouter starts returning HTTP 503, or the model response format changes, every sentence falls through as ENTAILED. The gate is silently inert — it logs a WARNING per call but no operator-visible alert summarizes "your gate has been failing 100% of the time for the last 4 hours."

## Proposed change

Add module-level counter dict in `strict_verify.py`:

```python
_JUDGE_TELEMETRY: dict[str, int] = {
    "calls": 0,
    "entailed": 0,
    "neutral": 0,
    "contradicted": 0,
    "judge_error": 0,
}
```

Increment in `_EntailmentJudge.judge()` after the verdict is decided:

```python
def judge(self, sentence, span):
    _JUDGE_TELEMETRY["calls"] += 1
    try:
        # ... existing logic ...
        _JUDGE_TELEMETRY[verdict.lower()] += 1
        return verdict, reason
    except Exception as exc:
        _JUDGE_TELEMETRY["judge_error"] += 1
        logger.warning(...)
        return "ENTAILED", f"judge_error: ..."
```

Expose a read function for operators / tests:

```python
def get_judge_telemetry() -> dict[str, int]:
    """Return a snapshot of the entailment-judge telemetry counters.

    Counters are process-lifetime and not reset on env changes — read
    once before a job to compute deltas if needed. Operators concerned
    that the gate has gone silently inert can poll this from a health
    endpoint or scripts/observability tooling.
    """
    return dict(_JUDGE_TELEMETRY)


def reset_judge_telemetry() -> None:
    """Test helper. Operators should NOT call this in production."""
    for key in _JUDGE_TELEMETRY:
        _JUDGE_TELEMETRY[key] = 0
```

## Test surface

- `test_telemetry_starts_at_zero` — fresh process → all counters 0
- `test_calls_increment_on_each_judge_invocation` — invoke fake judge 5 times, calls=5
- `test_entailed_verdict_increments_entailed` — fake returns ENTAILED → entailed=1
- `test_neutral_verdict_increments_neutral` — NEUTRAL → neutral=1
- `test_contradicted_verdict_increments_contradicted` — CONTRADICTED → contradicted=1
- `test_judge_error_increments_on_exception` — fake raises → judge_error=1, returns ENTAILED+tag
- `test_get_judge_telemetry_returns_snapshot` — return value is a dict copy not a live reference
- `test_reset_judge_telemetry_zeroes_all` — reset puts all back to 0

But — the existing `_EntailmentJudge` is the one that needs counter increment, not the FakeJudge. So tests need to instantiate `_EntailmentJudge` somehow + monkeypatch the http call OR add the counter call sites in a way that the FakeJudge can also tick them.

Cleaner design: counters live in the strict_verify module, incremented in `verify_sentence()` AFTER the judge call returns (regardless of judge implementation). The judge itself returns the verdict; strict_verify increments based on the returned verdict. This way tests using FakeJudge automatically tick counters too.

Adjusted increment site: in `verify_sentence()` after `verdict, reason = _get_judge().judge(...)`, increment `_JUDGE_TELEMETRY[verdict.lower()] += 1`. For `judge_error`, since the fake returns "ENTAILED" with reason starting "judge_error:", we tick `judge_error` instead of `entailed` when reason matches.

Actually cleaner: have the judge return a tuple `(verdict, reason, error_flag: bool)` OR have strict_verify check `reason.startswith("judge_error:")` and route to judge_error counter. Slightly hacky but localized.

Cleanest: add counter call sites in strict_verify._verify_via_judge() which dispatches to the judge and counts. ~10 LOC helper.

## What I want from you

1. **Verdict** APPROVE / REQUEST_CHANGES.
2. **Counter increment site**: judge-side vs strict_verify-side? My read: strict_verify-side because (a) tests using FakeJudge automatically tick the counters, (b) the gate semantics live in strict_verify so the counters belong there too, (c) judge can be swapped (different model, future async judge, etc.) without losing telemetry.
3. **`reset_judge_telemetry` exposure**: should this be public or test-only-private? Public lets an operator deliberately reset between jobs. Private (`_reset_judge_telemetry`) keeps the contract narrow. I lean public — the cost is one more callable; the upside is operator flexibility.
4. **Counter granularity**: separate counters per verdict OR a single Counter-like multi-key dict? I propose explicit dict keys (cleaner read).
5. **Should off-mode also tick a counter** (e.g. `skipped_off_mode`)? My read: NO — off mode should pay zero cost, and tick is a tiny cost. Operators who want "is the gate enabled" can read the env var directly.
