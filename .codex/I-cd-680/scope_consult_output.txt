Decision: **B**.

For Phase 1, scope #680 to **minimal real-run resolution for followup + compare**:

- Resolve `run_id -> artifact_dir` through the same `run_store` path already proven by `GET /runs/{run_id}/bundle.tar.gz`.
- Load the existing slice-chain data that Inspector already consumes.
- Preserve golden fixture support where needed.
- Do **not** synthesize `span_start` / `span_end`.
- Do **not** introduce the full typed `EvidenceContract` JSON in Phase 1.

This is the higher-quality choice because the actual blocker for #542/#543 is not missing char-offset provenance; it is that `followup.py` and `compare.py` are fixture-locked via `_GOLDEN_RUN_INDEX`. Building full EvidenceContract now would expand scope, increase regression risk, and create LAW II risk if offsets are inferred rather than recorded by Pipeline A.

I would frame the Phase 1 acceptance criteria as:

1. Real completed runs no longer 404 in follow-up and compare.
2. Both APIs resolve real `run_id` using `run_store`.
3. They consume existing artifact/slice-chain data only.
4. Golden fixture behavior remains intact.
5. Tests cover real-run resolution and missing-artifact failure modes.
6. No fabricated char offsets are emitted.

Yes: the full capability should become a **new post-demo Phase 2 issue** or be explicitly split out from #680. That issue should cover the real EvidenceContract design: Pipeline A records offsets at source, verifier threads them through, and APIs emit typed evidence spans only when grounded offsets exist.
