## Codex iter-1 findings — resolutions

**P1 — `asyncio.wait_for` does not guarantee a hard abandon.** Correct, and
accepted. `wait_for` cancels the backend and then **awaits** the coroutine
finishing its cancellation/cleanup; `_try_crawl4ai`'s `_safe_close_crawler`
(`__aexit__`) cleanup is unguarded, so a Playwright cleanup hang keeps the
wrapper and `gather` stuck past the timeout. Resolved: the wrapper no longer
uses `wait_for`. It uses **`asyncio.wait({task}, timeout=T)`**, which returns
after `T` **without cancelling or awaiting** the task. On timeout the wrapper
issues a **fire-and-forget `task.cancel()` and does NOT await the task** — so
a hung `__aexit__` cannot block the wrapper. This is a genuine hard
wall-clock return. The regression test now models a coroutine that **hangs
during cancellation cleanup**, not cooperative `asyncio.sleep`.

**P2 — "every fetch path" claim vs concurrent-only acceptance.** Resolved by
narrowing the wording: this PR bounds the **concurrent backend fan-out** in
`fetch_with_bypass` — that is where the verified 30-min hang occurred. The
issue's "every fetch path" is scoped to that fan-out (crawl4ai / jina /
firecrawl / trafilatura). The downstream direct-HTTP fallback already uses
`requests`/`httpx` with their own request timeouts; an overall
`fetch_with_bypass` wall-clock bound is deliberately out of scope (would be
a second, broader change). The acceptance criteria below say "concurrent
backend fan-out" explicitly.

## Root cause (verified against HEAD — `src/tools/access_bypass.py`)

`AccessBypass.fetch_with_bypass` (line 377) builds a list of per-backend
coroutines — `self._try_crawl4ai(url)`, `self._try_jina_reader(url)`,
`self._try_firecrawl(url)`, optionally `self._try_trafilatura(url)` — and
runs them with `await asyncio.gather(*concurrent_tasks,
return_exceptions=True)`. `gather` waits for **every** task; there is **no
per-backend wall-clock bound** at this site. A backend wedged in a
Playwright op (`_try_crawl4ai`'s `__aenter__` browser startup or
`_safe_close_crawler` cleanup are unguarded; its inner `crawler.arun` is
`wait_for`-bounded but, per the P1 above, `wait_for` cannot hard-abandon a
hung cleanup) freezes the whole `gather` forever.

## The fix — a hard-abandon bounded-backend wrapper at the gather site

In `src/tools/access_bypass.py`:

1. **`_backend_fetch_timeout() -> float`** — module helper reading env
   `PG_BACKEND_FETCH_TIMEOUT` (default `60.0` s; the per-backend wall-clock
   ceiling). Config-driven (LAW VI).

2. **A module-level `set` `_DETACHED_BACKEND_TASKS`** — holds strong refs to
   abandoned tasks so they are not GC-warned ("Task was destroyed but it is
   pending"); each task gets `add_done_callback(_DETACHED_BACKEND_TASKS.discard)`.

3. **`_bounded_backend(label, coro, url) -> AccessResult`** — the hard bound:
   ```python
   async def _bounded_backend(label, coro, url):
       timeout = _backend_fetch_timeout()
       task = asyncio.ensure_future(coro)
       done, _pending = await asyncio.wait({task}, timeout=timeout)
       if task not in done:
           # HARD abandon: cancel fire-and-forget; do NOT await the task —
           # its cleanup (e.g. Playwright __aexit__) may itself hang.
           task.cancel()
           _DETACHED_BACKEND_TASKS.add(task)
           task.add_done_callback(_DETACHED_BACKEND_TASKS.discard)
           logger.warning("[ACCESS] backend %s exceeded %.0fs wall-clock for "
                          "%s — abandoned (task detached)", label, timeout, url[:60])
           return AccessResult(url=url, content="", access_method=label,
                               legal_alternative=None, success=False,
                               metadata={"error": f"backend_timeout_{timeout:.0f}s"})
       exc = task.exception()
       if exc is not None:
           return AccessResult(url=url, content="", access_method=label,
                               legal_alternative=None, success=False,
                               metadata={"error": f"{type(exc).__name__}: {exc}"})
       return task.result()
   ```
   `asyncio.wait` returns after `timeout` regardless of the task's state;
   the abandoned task is cancelled **without an await**, so a hung
   `__aexit__` cleanup cannot block the wrapper or `gather`.

4. **At the `concurrent_tasks` build site** (lines ~467-481): wrap each
   append — `concurrent_tasks.append(_bounded_backend("crawl4ai",
   self._try_crawl4ai(url), url))`, likewise `jina`, `firecrawl`,
   `trafilatura`.

No post-`gather` change needed: `_bounded_backend` always returns an
`AccessResult`, and the existing result loop scores `AccessResult`
candidates and returns the highest-quality winner; if every backend fails,
`fetch_with_bypass` falls through to its existing direct-HTTP path.

## Explicitly OUT of scope (do NOT require for APPROVE)

- Adding `wait_for`/timeout guards *inside* `_try_crawl4ai` (`__aenter__`,
  `_safe_close_crawler`). The hard-abandon wrapper makes the whole backend
  bounded; guarding inner awaits would not change the wall-clock the bug
  needs and expands the diff.
- An overall `fetch_with_bypass` wall-clock bound (the direct-HTTP fallback
  has its own request timeouts) — see P2 resolution.
- Subprocess-isolating the browser; the broader retrieval-robustness scope
  (G9 Serper canary, G10 browser/network venue suite) — separate issues.

## Regression test — `tests/` (the access_bypass test module)

Proves the **hard** bound — including the cancellation-cleanup hang class —
without Playwright, without a live URL:

- **`test_bounded_backend_hard_abandons_a_cancellation-hanging_backend`**:
  drive `_bounded_backend` with a coroutine that, on `CancelledError`,
  enters a *further* long sleep (modelling a `__aexit__` cleanup that hangs
  during cancellation — the exact class `wait_for` cannot escape). With
  `PG_BACKEND_FETCH_TIMEOUT` low (e.g. `1`), assert `_bounded_backend`
  returns a failure `AccessResult` (`success is False`,
  `metadata['error']` starts `backend_timeout`) within `timeout + epsilon`
  wall-clock — i.e. it does NOT wait for the cancellation cleanup.
- **`test_fetch_with_bypass_survives_one_hung_backend`**: monkeypatch
  `AccessBypass._try_crawl4ai` to such a hang-on-cancel coroutine and
  `_try_jina_reader` to a fast successful `AccessResult`; with
  `PG_BACKEND_FETCH_TIMEOUT` low, assert `fetch_with_bypass` returns the
  jina result within a bounded wall-clock.
- Tests set `PG_BACKEND_FETCH_TIMEOUT` via `monkeypatch.setenv` and run in
  seconds; the hang-on-cancel coroutine's secondary sleep is bounded
  (~10s) so event-loop teardown stays clean.
